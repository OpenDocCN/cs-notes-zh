# 081：并行与同步入门 🚀

![](img/b910c41799f9c8ec0e27da09ba364eee_0.png)

在本节课中，我们将学习并行计算的基本概念，特别是同步问题及其解决方案。我们将探讨为什么现代计算机转向多核架构，并深入理解在并行编程中如何确保数据的一致性和正确性。

## 摩尔定律与并行化趋势 📈

上一节我们介绍了课程主题，本节中我们来看看推动并行计算发展的背景因素。

摩尔定律指出，芯片上的晶体管数量呈指数级增长。然而，单核处理器的顺序性能在达到某个点后已趋于平缓。如今，新的处理器（如酷睿 i7）的性能提升已经放缓。实际上，厂商发布的基准测试分数（如 SPECint 和 SPEC FP）现在也常使用多核并行计算来获得更高的分数。

时钟频率的增长同样遇到了瓶颈。我们未能实现 10 GHz 的处理器。这主要是由于遇到了功耗挑战，同时设计极高频率的处理器也异常困难。

因此，我们开始利用日益增多的晶体管来实现多个核心。我们从双核起步，发展到四核，并持续增加。英特尔目前销售 8 核产品，并即将推出 10 核产品。AMD 有 16 核产品，尽管它并非真正的单芯片 16 核设计。市场上还存在更多核心的众核处理器。

在多核革命之前，人们就已经构建了多处理器系统。他们可以将多个芯片组合成小型或大型系统。在本课程剩余部分，我们将讨论并行计算系统，包括单芯片多核系统和多芯片系统。两者之间存在一些细微差别，但也有很多共同点。

## 对称多处理器与共享内存 💾

在今天的讲座中，我们将重点讨论对称多处理器和内存系统，它们用于共享数据。但共享数据还有其他方式，我们将在后续关于消息传递的讲座中简要提及。

对称多处理器是最简单的推理模型。所有处理器位于上方，内存位于下方。我们首先假设系统中没有缓存，并且每个处理器到内存的距离相等，共享一个大的内存空间。多个处理器可以运行一个程序中的多个线程，或多个不同的程序。

并发挑战不仅源于拥有多个处理器。其他设备，如各种 I/O 控制器（网卡、磁盘控制器、显卡）也需要读写内存。因此，即使在单处理器系统中，内存系统内也有多个代理同时希望读写内存。

## 同步问题与经典模型 🔒

现在我们来谈谈同步。哲学家就餐问题是同步挑战的一个例子。

我们将讨论两个主要的同步思想（尽管还有更多，如广播模型）。同步是以受限制的方式协调通信或任意通信的方法。

以下是两种基本模型：

*   **生产者-消费者模型**：如图右侧所示，生产者产生一些值，消费者消费这些值。这是最基本的形式（一个生产者，一个消费者）。也可以扩展为一个生产者多个消费者、多个生产者多个消费者或多个生产者一个消费者，其核心思想类似。
*   **互斥模型**：确保不超过指定数量的处理器同时访问某个共享资源。这个资源可以是磁盘、显卡，也可以是内存中的一个位置。之所以称为互斥，是因为访问在某一时刻是排他的。最基本的形式是只允许一个处理器访问。更通用的形式是信号量，它允许不超过 N 个处理器同时访问。例如，一个只有两个出队队列的网卡，就可以允许最多两个处理器同时使用。

## 单处理器系统中的同步挑战 ⚙️

即使在单处理器系统中，也可能存在同步挑战。这是如何发生的呢？

首先，单处理器系统中有时存在多个并发实体。例如，磁盘驱动器通过 DMA 向内存写入数据的同时，一个进程可能正试图读取那个内存块。这就需要保证在处理器读取数据块之前，磁盘块已被完全读取。

其次，单处理器系统通常支持多道程序。操作系统会在不同程序之间进行时间片轮转调度，在抢占式环境下，不同进程的指令会交错执行。一个进程正在运行，定时器中断发生，该进程被挂起，切换到另一个进程开始执行其代码。过一段时间再切换回来。因此，即使在单处理器系统上，也可能遇到同步挑战。

## 生产者-消费者代码示例与分析 🧑‍💻

让我们看一个生产者-消费者的代码示例，因为它很有指导意义。

首先抽象地看：我们有一个生产者和一个消费者，他们使用一个大的对称内存块进行通信。两者之间有一个队列，以及头指针和尾指针。这是一个先进先出队列，指针会移动并可能回绕。

生产者拥有自己的寄存器（如 Rtail），消费者拥有自己的寄存器（如 Rhead, Rtail, Rval）。它们各自拥有寄存器的副本，这可能导致不同步的问题。

以下是基本的代码序列：

**生产者**希望将项目 X 放入队列：
1.  将尾指针加载到其寄存器中（`load tail -> Rtail`）。
2.  将值 X 存储到尾指针指向的位置（`store X -> [Rtail]`）。
3.  在其寄存器空间中将尾指针加一（`add 1 -> Rtail`）。
4.  将更新后的尾指针存回内存（`store Rtail -> tail`）。

**消费者**的操作：
1.  将头指针加载到其寄存器中（`load head -> Rhead`）。
2.  将尾指针加载到其寄存器中（`load tail -> Rtail`）。
3.  比较头指针和尾指针是否相等（`cmp Rhead, Rtail`）。如果相等，意味着队列为空，则跳转回步骤 2 重新加载尾指针并循环等待，直到它们不相等。
4.  从头指针指向的位置加载数据到 Rval 寄存器（`load [Rhead] -> Rval`）。
5.  更新头指针到下一个位置（`add 1 -> Rhead`）。
6.  将更新后的头指针存回内存（`store Rhead -> head`）。
7.  对读取到的值进行处理（`do something with Rval`）。

这个程序有些天真。我们讨论过乱序处理器和乱序内存系统。这个代码假设指令按顺序执行，并且消费者和生产者之间的操作在内存系统中不会交错。但真的没有问题吗？

## 内存顺序与同步隐患 ⚠️

问题在于，我们只保证了**同一处理器上对同一地址的存储后加载操作**会按顺序发生。我们讨论过的乱序超标量处理器和乱序内存系统，**没有定义不同处理器之间、甚至同一处理器内部不同内存操作之间的顺序**。这就是我们今天要讨论的核心：如何理解这些指令可能被重排，以及如何防止由此导致的混乱。

让我们关注生产者代码中的两个存储操作（`store X` 和 `store Rtail`），以及消费者代码中的两个加载操作（`load head` 和 `load tail`）。程序员错误地假设了它们之间存在某种因果关系。生产者假设步骤 1（加载尾指针）发生在步骤 4（存储更新后的尾指针）之前，但我们的内存系统（尤其是乱序内存系统）并不保证这一点。同样，消费者的两次加载也可能完全乱序执行。

在单处理器系统中，这个假设或许成立。但在多处理器系统中，这是一个糟糕的假设。

让我们看一些有问题的执行序列：
*   **序列 A**：生产者先执行步骤 2 和 3（更新尾指针值并加一），然后消费者执行步骤 4（加载头指针处的值），最后生产者执行步骤 1（存储值 X）。结果消费者读到了垃圾数据。
*   **序列 B**：消费者先执行步骤 4（加载头指针处的值），然后生产者执行步骤 1（存储值 X），接着生产者执行步骤 2 和 3。这同样是完全乱序的，没有良好的语义。

因此，这段看起来应该有正确语义的代码，实际上可能没有任何有用的语义。我们需要定义这些语义究竟是什么，这正是后续要解决的问题。
