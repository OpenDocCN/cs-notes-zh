# 013：控制冒险与跳转处理

![](img/c3e4185c4e08cce8823a7dad0a99136f_0.png)

在本节课中，我们将要学习控制冒险。控制冒险是流水线处理器中因分支和跳转指令改变程序执行流而引发的性能问题。我们将探讨其产生原因，并介绍一种基础的缓解技术——推测执行。

## 控制冒险概述

上一节我们介绍了数据冒险，本节中我们来看看控制冒险。控制冒险发生在处理器需要决定下一条指令的地址，但该信息尚未在流水线中计算出来时。这会导致流水线停顿，降低性能。

## 计算下一条程序计数器

要理解控制冒险，首先需要知道处理器如何计算下一条指令的地址。对于不同类型的指令，所需信息是不同的。

以下是计算下一条程序计数器（PC）所需的信息：

*   **跳转指令**：需要操作码（确认是跳转）、指令内的偏移量以及当前PC值。公式为：`新PC = 当前PC + 偏移量`。
*   **跳转寄存器指令**：需要操作码（确认是跳转寄存器）以及从寄存器文件中读取的目标地址值。例如在MIPS中：`新PC = 寄存器值`。
*   **条件分支指令**：需要操作码、当前PC、用于判断条件的寄存器值以及指令内的偏移量。在MIPS中，公式为：`新PC = (条件满足) ? (当前PC + 偏移量) : (当前PC + 4)`。
*   **其他指令（如算术逻辑指令）**：需要操作码和当前PC，并通常执行`PC = PC + 4`以顺序执行下一条指令。

问题在于，操作码在**译码阶段**才被识别，寄存器值在**译码/执行阶段**才可用，而条件判断可能要到**执行阶段**才能完成。这意味着，在取指阶段，我们无法立即知道下一条指令的地址。

## 基础控制冒险分析

让我们分析一个最简单的场景：顺序执行两条没有数据依赖的指令。

```
指令1: ADD R1, R2, R3
指令2: SUB R4, R5, R6
```

在五级流水线（取指IF、译码ID、执行EX、访存MEM、写回WB）中，情况如下：

*   指令1进入流水线。
*   当指令1处于IF阶段时，处理器需要为下一条指令（指令2）计算PC。但由于指令1的操作码（ADD）尚未被译码，处理器无法立即确定指令1不是分支或跳转，因此无法安全地获取指令2。

这导致了一个结构性问题：**取指阶段必须停顿一个周期，等待译码阶段确认当前指令的类型**。这种停顿被称为“译码气泡”。

如果每条指令都因此停顿，那么每条指令都需要两个周期才能完成取指，导致**CPI（每指令周期数）变为2**，性能下降一半。这显然是不可接受的。

## 缓解技术：推测执行

为了解决这个问题，现代处理器采用了推测技术。最基础的推测策略是：**总是假设当前指令不是分支或跳转，下一条指令的地址就是PC+4**。

这意味着，在取指阶段，处理器会不加停顿地预取下一条顺序地址的指令。如果后续阶段发现推测错误（例如当前指令确实是跳转），则需要采取纠正措施。

### 推测错误的处理：指令抹杀

当推测错误发生时（例如遇到跳转指令），已经预取并进入流水线前端的错误指令必须被清除。这通过“抹杀”或“转化为空操作”来实现。

以下是处理跳转指令推测错误的硬件机制简述：

1.  在译码阶段识别出跳转指令。
2.  生成一个控制信号，将错误预取指令（在IF/ID寄存器中）替换为一个**空操作**。
3.  同时，使用一个专用的加法器计算跳转目标地址：`跳转目标PC = 当前PC + 偏移量`。
4.  在下一个周期，将正确的目标指令地址送入PC，开始取指。

### 流水线图示

考虑以下指令序列：
```
100: JUMP 304
104: ADD ... (这条指令不应执行)
304: SUB ... (跳转目标)
```

推测执行下的流水线行为如下（`-`代表空操作周期或气泡）：

| 周期 | IF | ID | EX | MEM | WB |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 1 | JUMP 304 | - | - | - | - |
| 2 | ADD (104) | JUMP | - | - | - |
| 3 | SUB (304) | **NOP** (原ADD被抹杀) | JUMP | - | - |
| 4 | ... | SUB | NOP | JUMP | - |
| 5 | ... | ... | SUB | NOP | JUMP |

可以看到，地址104的ADD指令在译码阶段被替换为NOP，而正确的目标指令（304处的SUB）被顺利取指执行。虽然引入了一个气泡，但相比每条指令都停顿，效率更高。

### 一个复杂情况：停顿与跳转同时发生

一个值得思考的问题是：如果流水线因某种原因（如结构冲突）需要在译码阶段停顿，而恰好此时译码阶段有一条跳转指令，会发生什么？

在这种情况下，硬件设计需要仔细处理。一种可能的策略是优先处理跳转：即使流水线其他部分停顿，也允许完成跳转地址的计算和指令抹杀操作，然后在后续周期中插入必要的停顿气泡。具体实现取决于处理器的控制逻辑设计。

## 总结

![](img/c3e4185c4e08cce8823a7dad0a99136f_2.png)

![](img/c3e4185c4e08cce8823a7dad0a99136f_3.png)

本节课中我们一起学习了控制冒险。控制冒险源于处理器无法在取指时确定下一条指令地址。基础的“总是预测为PC+4”推测策略能有效减少停顿，但预测错误时需要抹杀已预取的指令。这是理解更高级分支预测技术的基础。在后续课程中，我们将深入探讨如何通过更精确的预测来进一步降低控制冒险带来的性能损失。
