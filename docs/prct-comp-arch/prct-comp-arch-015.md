# 015：控制冒险与其他

![](img/f3cfb68081729182475e6a1da2bc68c7_0.png)

在本节课中，我们将继续探讨控制冒险，并了解为何处理器无法在每个时钟周期都派发指令。我们将讨论转发机制的局限性、加载指令的延迟、条件分支的影响，并引入异常和中断这两种特殊的控制流改变事件。

## 转发与停顿的权衡

上一节我们介绍了通过转发解决数据冒险，本节中我们来看看为何有时仍需停顿。实现完全转发（全旁路）的成本可能很高。如果试图从流水线的每一个阶段都建立旁路路径，硬件开销会非常大。

因此，在某些情况下，我们可能仍然需要停顿。一个很好的例子是现代处理器，如酷睿 i7。它们实际上并没有在所有不同的功能单元和所有不同的流水线阶段之间都建立旁路，因为它们每个周期可以执行大约6条指令，且流水线深度很大。如果为每个新的源操作数都建立上百条旁路路径，成本将难以承受。

以下是典型的处理方式：
*   识别并实现最常用的旁路路径。
*   对于一些不常用的路径，则选择不构建硬件支持。

这种做法有助于缩短时钟周期时间，但会损害 CPI（每条指令的周期数）。

## 加载指令的延迟

加载指令通常具有两个周期的延迟。在我们讨论“加载-使用”冒险时提到过，加载指令之后的指令不一定能立即使用加载结果。在我们的五级MIPS流水线中，加载结果要到访存阶段才计算出来。因此，即使存在从访存阶段末尾的旁路，处于执行阶段的指令也无法获得该结果。

一个有趣的事实是，最初的MIPS I架构实际上定义了**加载延迟槽**，这与我们讨论过的分支延迟槽非常相似。MIPS I的加载延迟槽是软件可见的，编译器必须调度一条不依赖于该加载结果的指令到这个槽中，以解决流水线冒险。

然而，这个特性最终从指令集架构中移除了，并重新引入了流水线互锁（停顿）机制。原因在于，随着流水线长度和微架构的变化，这种软件可见的延迟槽变得难以维护。它缺乏微架构独立性。例如，如果流水线从五级变为四级，可能就不再需要那个延迟槽了。这也体现了“MIPS”（无互锁流水线级微处理器）这个名字最初的意图——他们不希望在这种“加载-使用”场景下有互锁。后来在MIPS II中，互锁被重新引入。

## 条件分支的影响

导致CPI大于1的另一个重要原因是**条件分支**，它会引入气泡（流水线空泡）。这就是我们到目前为止讨论的所有控制冒险。如果没有某种形式的延迟槽，就必须冲刷（杀死）错误路径上的指令。

这里需要指出关于CPI计算的一个重要注意事项：我们应该从**有效CPI**的角度来思考，而不是简单地计算执行了多少条指令。如果你在程序中添加了无操作指令，而这些指令没有做任何有用的事情，它们就不应该计入你的有效CPI计算。机器可能会将这些算作流水线上执行的有效指令，但在计算CPI时，应始终考虑那些对程序最终目标有贡献的指令周期。

## 其他控制流改变：异常与中断

在本课程中，我们需要讨论的其他控制冒险是那些可以改变程序控制流的事件。它们主要可以分为以下两类：

**异常**是指令执行了无效或违反机器预期用途的操作。例如：
*   **除以零**：在大多数计算机架构中，这是未定义的，会触发“除零错误”异常。
*   **内存故障**：试图访问不被允许的内存。
*   **上溢/下溢异常**：在某些架构中，当数值精度超出范围（例如浮点数变得过大或过小）时触发。

**中断**是外部发生的事件。例如：
*   定时器计时到期。
*   I/O设备试图唤醒处理器或向处理器发送信号。

这些事件之所以重要，并被视为控制冒险，是因为它们是意外进入指令流的事件，会改变后续执行的指令，实质上构成了一种控制流改变。

我们将在本课程后续部分更详细地讨论异常和中断。在此复习中，只需理解异常和中断是不同类型的控制流冒险即可。

**本节课总结**：我们一起学习了控制冒险的剩余部分。我们探讨了完全转发的高成本与停顿的权衡，回顾了加载指令的延迟及其历史解决方案（加载延迟槽），明确了条件分支对CPI的影响，并引入了异常和中断作为特殊的、不可预测的控制流改变事件，它们同样会导致流水线冒险。理解这些因素对于分析处理器性能至关重要。
