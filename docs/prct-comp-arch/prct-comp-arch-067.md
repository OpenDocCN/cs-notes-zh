# 067：TLB处理机制详解

![](img/d5df3e78f72d9c23a1677dacc61adc86_0.png)

在本节课中，我们将要学习**转换后备缓冲器**（TLB）在发生命中与未命中时的不同处理流程。我们将重点探讨硬件与软件两种处理TLB未命中的主流方法，并分析各自的优缺点。

## TLB命中与未命中的基本流程

上一节我们介绍了TLB的基本概念，本节中我们来看看TLB在具体访问时的处理逻辑。

![](img/d5df3e78f72d9c23a1677dacc61adc86_0.png)

如果访问**命中**TLB，处理器可以直接获得所需的物理地址，过程高效。

如果访问**未命中**TLB，处理器则需要通过查询页表来获取正确的地址映射。处理未命中的方法主要有两种。

## 硬件页表遍历（如x86架构）

一种主流方法是采用**硬件页表遍历器**。这种方法被x86等架构所使用。

其工作原理是：当发生TLB未命中时，处理器中的一个专用硬件状态机（即页表遍历器）会被触发。该硬件会**暂停**当前处理器流水线。

随后，这个硬件遍历器会在物理内存中，自动逐级查询多级页表结构。其过程可以抽象为多次索引操作：

```
物理地址 = 页表基址寄存器
For 每一级页表:
    物理地址 = 内存读取(物理地址 + 虚拟地址对应索引)
End For
```

最终，硬件遍历器会找到目标页表项，将其中的地址映射关系**填充**回TLB。这个过程类似于缓存未命中时的处理，但完全由硬件完成。

## 软件页表遍历（如MIPS、Alpha架构）

另一种常见方法是由**软件**（通常是操作系统）来处理TLB未命中。MIPS和Alpha架构采用了这种方法。

这种方法类似于由软件来处理缓存未命中。其可行性基于一个关键事实：**TLB未命中的发生频率相对较低**。

由于TLB未命中并不频繁，因此即使采用速度相对较慢的软件处理方式，对整体性能的影响也在可接受范围内。

采用软件处理还有一个显著优势：**页表布局可以更加灵活**。硬件页表遍历器要求页表结构必须严格符合硬件预设的格式。而软件处理则允许操作系统设计者采用不同的、甚至更优化的页表结构，只需相应的TLB未命中处理程序能理解该结构即可。

## 混合与特殊案例（如PowerPC）

值得注意的是，有些架构采用了混合或折中的方案。例如，PowerPC架构的处理方式就有些特别。

一些PowerPC的实现完全采用硬件页表遍历。而另一些实现则采用硬件为主、软件辅助的方式处理复杂情况。

反过来，以软件处理为主的架构也可能加入少量硬件来提升效率。例如，MIPS架构提供了一个特殊的硬件寄存器，用于存放**第一级页表的基址**。

操作系统软件可以选择是否使用这个寄存器。这相当于为软件遍历过程提供了一点“硬件加速”，但遍历页表的核心工作仍然由软件完成。

![](img/d5df3e78f72d9c23a1677dacc61adc86_2.png)

## 总结

本节课中我们一起学习了TLB未命中的两种核心处理机制：
1.  **硬件页表遍历**：由专用硬件自动完成，速度快，但要求页表格式固定。
2.  **软件页表遍历**：由操作系统软件处理，灵活性高，允许自定义页表结构，因未命中不频繁而对性能影响可控。

![](img/d5df3e78f72d9c23a1677dacc61adc86_2.png)

此外，我们还了解到像PowerPC这样的架构可能采用混合方案，而MIPS等架构则可能用少量硬件辅助软件处理。理解这些机制有助于我们把握不同计算机体系结构在内存管理单元设计上的权衡与特点。
