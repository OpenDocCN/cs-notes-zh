# 078：粗粒度多线程 🧵

![](img/86bd06e3df2797de71dc72aaeeaedc81_0.png)

在本节课中，我们将要学习**粗粒度多线程**的核心概念。这是一种通过执行来自不同线程的指令来隐藏内存访问延迟、提高处理器利用率的技术。我们将从基本思想开始，逐步探讨其硬件实现、潜在挑战以及历史上的实际应用案例。

## 核心思想与基本方法

上一节我们介绍了多线程的基本动机。本节中我们来看看其核心思想。

如果能够保证指令之间没有依赖关系，就可以考虑并行执行这些指令。而保证独立性的一个绝佳方法，就是执行完全不同的程序或线程。

最朴素的方法是：**交错执行来自不同线程的指令**以覆盖延迟。例如，线程1发起一个加载操作。当这个加载操作在内存系统中解析时，处理器可以执行其他线程的工作。当数据从内存返回时，线程1的下一条指令已经准备就绪。这种在处理器流水线中插入其他工作的技术，就称为**多线程**。

这里的一个关键洞察是：需要有足够多的其他工作可做。这些额外的线程会给处理器设计增加一些复杂性，并可能引发局部性挑战。例如，如果你试图在缓存或转换后备缓冲器等结构中利用时间局部性，而在中间插入其他线程的数据或操作，就可能会破坏这种局部性。但乐观地看，如果能找到其他工作，就可以在等待加载返回时将其塞入空闲的流水线槽中。

## 基础实现与硬件支持

了解了基本思想后，本节我们来看看其最基础的实现方式以及所需的硬件支持。

最基础版本的多线程采用某种非常基本的交错执行。我们按顺序运行线程1、线程2、线程3、线程4，然后循环。调度时并不关注指令的延迟或依赖性。如果轮到一个线程槽时无事可做（例如加载操作缓存未命中，需要1000个周期才能返回），那么该周期就不调度任何指令，形成一个空泡周期。

多线程的一个好处是可以利用它来**消除旁路逻辑**。因为如果知道在几个周期内不会执行来自同一线程的指令，就不再需要从ALU到自身的快速旁路。

以下是早期多线程机器的一个例子：假设有一条指令写入寄存器R1，紧接着另一条指令读取R1。在交错执行下，读取操作会得到什么值？早期的多线程机器尝试**分区寄存器文件**来避免这个问题，即通过不同的方式编译程序。但这已不常见。后来演变为为每个线程提供完整的寄存器集副本。

如下图所示，硬件需要复制四份通用寄存器文件和程序计数器。流水线前端有一个增量器来选择线程ID。在简单情况下，它持续递增，按1、2、3、4的顺序循环选择线程。这同时也作为索引，决定访问哪个通用寄存器文件。大多数架构实际上会将它们合并成一个更大的寄存器文件，这些地址位在架构上是不可见的。

我们的简单多线程流水线目前改动很小：复制寄存器文件，复制程序计数器。这可以帮助我们恢复ALU的利用率。

对软件而言，这看起来就像是多个处理器。例如，现代酷睿i7处理器有硬件线程，在操作系统的进程管理中，会显示为两倍于物理核心数量的虚拟处理器。

## 成本、挑战与权衡

上一节我们看到了硬件支持的基本形式。本节中我们来分析实现多线程的成本和面临的挑战。

简单的成本包括复制程序计数器和通用寄存器。更困难但也必须复制的是系统状态，以实现CPU间的完全隔离。例如页表、基于页表的寄存器、异常处理程序地址等系统状态。像x86这样的架构有大量系统状态，复制起来可能很困难。MIPS相对较少，但依然存在。必须为每个线程复制所有这些状态，这是有成本的，但好处是能够隐藏内存延迟，重用ALU，提高效率。

更大的挑战在于可能引发的**时间局部性和空间局部性冲突**。如果有四个线程都在执行，它们会争夺缓存中的空间和容量。例如，如果有16个硬件线程和一个8路组相联缓存，而它们总是想访问缓存的索引0（例如栈起始地址），那么只有8路可用。当时间片轮转16个线程再回到第一个线程时，由于缓存只有8路，第一个线程需要的数据肯定已经被替换出去，导致没有时间局部性，产生冲突缺失。

消极的一面是线程可能相互冲突。但如果两个线程协同工作，实际上可以在缓存中产生**积极的干扰或建设性干扰**。如果不同线程处理大致相同的数据，这实际上非常好，因为它们会为彼此预取数据。但需要小心权衡：线程何时争夺共享资源，何时在引入共享资源上协作并利用处理器的局部性结构。

缓存是一个例子，另一个例子是转换后备缓冲区条目。一种解决方案是实际**分区缓存或TLB**，以保证线程间不冲突。但缺点是，如果缓存大小固定，分区实际上会减少每个线程可用的缓存大小。因此，通常人们不会尝试静态分区，但如果希望线程间完全无干扰，可以考虑这样做。

## 线程调度策略演进

了解了挑战之后，本节我们来看看几种不同的线程调度方法，它们从简单到复杂不断演进。

最简单的是我们目前讨论的：对N个硬件线程进行**固定交错**。基本上每个线程执行一条指令，然后切换到下一个线程。这种方法可以追溯到20世纪60年代，非常简单。其优点是可以在处理器中移除一些互锁和旁路逻辑。

接下来可以考虑的是：尝试在调度时间片中为不同线程分配不同的位置。例如，我们知道黄色线程是最高优先级线程，需要最多的时槽。我们可以进行不均匀的分配，让黄色线程每隔一个时槽执行一次，橙色和蓝色线程获得较小的比例。这在一定程度上仍然是固定交错或固定顺序，但可以通过软件（或操作系统）根据线程优先级进行控制，因此比完全固定的交错设计在硬件上更灵活。这需要一些硬件改动，不能仅仅是一个递增线程ID的计数器，而需要一个选择器来决定执行哪个线程，但仍然相对简单。

更复杂一些的策略是让**硬件**决定执行哪个线程。甚至可以做到：当检测到一个线程执行长延迟操作时，立即切换到另一个线程。这样就有目的地用其他线程的工作来回填。类似HEP处理器的设计就是早期例子。

## 粗粒度多线程定义与历史案例

上一节介绍了不同的调度策略。本节我们来明确粗粒度多线程的定义，并回顾一些重要的历史案例。

我们称之为**粗粒度多线程**，是因为在任何一个周期内，只有一个线程在运行。你可能会问，如何在一个周期内运行多个线程？如果使用超标量或多发射处理器，可以考虑让不同的流水线同时执行来自不同线程的指令，这称为**同时多线程**，我们稍后会讨论。

但在我们的粗粒度方法中，总结来说，可以在缓存未命中时交换线程，并利用处理器中的空泡周期做其他工作。

这里有一个历史案例：**HEP处理器**。由Burton Smith（现就职于微软研究院）在20世纪80年代设计。这台处理器非常有趣，因为它有大量线程和少量处理器。每个处理器有120个线程，时钟频率在当时相对适中。其设计目标是处理内存延迟。该机器具有非常高带宽的内存系统。实际上，它允许每条指令都进行加载或存储，而性能不会下降，因为每个处理器有120个线程，且内存延迟小于120个周期。如果每条指令都加载，并且有足够带宽满足所有加载请求，就可以从每个不同线程执行加载操作，每个线程彼此独立，从而将内存访问延迟流水线化。

这一思想被延续下来。Burton Smith创立了Tera公司（后更名为Tera Computer Systems），该公司后来收购了Cray。Tera有类似的设计理念：更多处理器，是HEP处理器的进一步演进。每处理器128个活动线程，大量处理器（256个），因此有大量活动程序。这就要求在程序中找到足够的线程级并行性。这种架构**没有缓存**，因为如果内存系统有足够带宽，可以每个周期都进行加载，并用其他线程覆盖延迟。但需要注意，这可能不利于功耗，并且完全没有利用数据引用的局部性。此外，必须提供大量线程。这种架构主要应用于那些数据引用本身就没有时间或空间局部性的场景，例如大数据集的数据挖掘。这种思想在今天仍然存在于Cray XMT（极致多线程架构）中。

另一个历史花絮是**MIT Alewife处理器**。它使用SPARC处理器，SPARC具有寄存器窗口的概念。在函数调用和返回时，可以滑动窗口来改变访问的寄存器文件部分。MIT Alewife扩展了这一思想，通过特殊指令快速切换整个寄存器文件的视图，从而非常有效地实现了四线程多线程，并引入了“在远程内存访问时切换线程”的概念。

## 现代应用与总结

最后，我们来看看粗粒度多线程在现代处理器中的应用，并对本节课内容进行总结。

多线程，特别是粗粒度多线程，在今天仍然存在。一个很好的例子是Oracle（原Sun，更早是Afara）的Niagara处理器。Afara是设计该处理器的公司，后被Sun收购，Sun又被Oracle收购。Sun Niagara处理器专为吞吐计算设计，可以运行大量独立线程，是一种多线程处理器。Niagara 1有8个核心，每核心4线程。后来不断升级，Niagara 3（即Sun T3）有16个核心，每核心8线程，提供了大量的并行性。

**本节课中我们一起学习了**：
1.  **粗粒度多线程的核心思想**：通过交错执行独立线程的指令来隐藏内存延迟，提高硬件利用率。
2.  **基础硬件实现**：需要为每个线程复制关键状态，如程序计数器和寄存器文件，并添加线程选择逻辑。
3.  **潜在的成本与挑战**：包括系统状态复制的复杂性，以及多线程可能引发的缓存冲突和局部性破坏问题。
4.  **线程调度策略**：从简单的固定交错，到基于优先级的调度，再到硬件自适应的调度。
5.  **历史与现代案例**：从HEP、Tera、MIT Alewife等早期研究，到现代Oracle Niagara处理器的商业应用，粗粒度多线程一直是提高处理器吞吐量的有效技术之一。
