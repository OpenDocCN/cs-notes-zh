# 108：目录一致性协议的可扩展性 🚀

在本节课中，我们将深入探讨目录一致性协议的可扩展性，涵盖多逻辑通信通道、内存排序点、前向进度保证以及目录结构的演进。我们将分析如何避免死锁和活锁，并了解现代大规模并行系统中的目录协议实现。

## 多逻辑通信通道与死锁避免 🔄

上一节我们介绍了目录协议的基本原理，本节中我们来看看如何管理复杂的通信流程以避免死锁。

多个逻辑通信通道是必需的。我们之前讨论的所有不同消息类型，加上图中未绘制的约10种类型，不能同时使用同一个网络。如果只有一个互连网络或一个逻辑网络，可能会出现响应消息被排在请求消息之后的情况，从而产生更多流量。

在这些协议中，通信分为多个阶段。消息从A点发送到B点，然后从B点到C点，再从C点返回B点，最后从B点返回A点。如果将这些通信叠加到一个互连网络上，并采用队头处理（即优先处理队列头部的消息），很容易出现新响应消息“插队”的情况，该响应可能依赖于系统中其他地方正在进行的其他事务，而你正在等待它。例如：
*   一个新请求到达，排在一个响应之后。
*   一个新请求到达，排在一个响应之前。
突然之间，就可能发生死锁。

解决此问题的一种方法是，分析所有不同的消息类型，并确定哪些类型可以同时共存于网络上。一个很好的例子是，在一个相对基本的缓存一致性协议中，可能有12种不同的消息类型。可以尝试将其缩减为3或4个不同的流量类别，这些类别的流量混合时永远不会引入死锁。如果想确保绝对安全，可以建立12个独立的网络或12个不同的逻辑网络，但这构建成本高昂。因此，人们尝试找出不同流量的等价类，然后只建立相应数量的网络。但必须将这些流隔离在不同的逻辑或物理通道上，以确保不会发生死锁。

## 内存排序点与仲裁 ⚖️

现在我们已经将内存分布开来，因此需要开始关注内存排序点。对于给定的内存地址，需要确定内存操作的发生顺序。就像在基于总线的协议中会有一个仲裁器来决定谁赢得事务或总线访问权一样，在基于目录的系统中，通常，给定地址的目录是最终的仲裁者。

如果两个消息从两个不同的核心同时到达，都希望获得对特定地址的写访问权，那么先到达目录的那个通常会获胜。有时这可能不公平，因此一些系统会尝试采用某种机制，防止离目录近的核心总是赢得对争用缓存行的访问。可能会引入某种退避协议。但正是目录保证了原子性：它确保在任一时刻，特定缓存行只能从一种状态转换到另一种状态。目录就是排序点。可以说，先到达主目录的消息获胜。对该行的后续请求将失败。

那么，失败的请求会怎样？目录很可能会发送一个否定确认或一个“重试”消息。它会说：“我现在无法处理这个请求，该行当前正在转换中，其他人已经赢得了转换权，请稍后重试该内存事务。”

现在，回到缓存控制器这边，情况变得相当复杂，因为它可能会收到一个重试或否定确认。同时，它可能还会收到针对同一缓存行的其他消息。在这种情况下，它需要优先处理来自目录的消息，而不是主处理器的流水线请求，因为目录是该内存位置的最终排序点。

## 前向进度保证与活锁避免 🛡️

最后，必须考虑前向进度保证。这意味着很容易构建一个这样的内存系统：数据被拉入缓存，但在能够对该地址实际执行加载或存储操作之前，它就被逐出了缓存。突然之间，缓存行可能在两个缓存之间来回“弹跳”，形成活锁场景。

在构建这些基于目录的一致性系统时（基于总线的协议中也可能发生，但通常可能性较小），必须要有某种前向进度保证。可能性较小的原因是在基于目录的系统中，通信时间通常更长，因此“脆弱窗口期”也更长。

前向进度保证意味着，一旦数据进入你的缓存，你至少需要对其执行一次内存操作，以保证在将内存交还给目录控制器之前已经完成了一次操作。例如，如果你执行加载操作，实际上是将数据以共享状态带入缓存，执行一次实际的加载，然后才允许放弃它。因此，在完成一次内存事务之前，你不会回复目录控制器发来的无效化请求。同样，如果你以修改状态带入数据，在交还数据给内存控制器之前，需要先执行一次存储操作。这对于确保系统有前向进度、避免活锁至关重要。

## 目录结构的演进与可扩展性 📈

好的，接下来我们将探讨更具前瞻性的内容。我们一直在讨论所谓的**全映射目录**，即为每个处理器核心设置一个位。如果系统中有1000个核心，那将是一个非常庞大的位图。而1000个核心同时读取所有数据或某一特定缓存行的情况并不常见。因此这可能造成浪费，并且全映射目录的大小以O(N)增长，这并不理想。

因此，人们研究了不同的协议，这里只是简要提及。这很大程度上是该领域的研究和未来方向。

以下是几种主要的目录结构优化方案：

*   **有限指针目录**：其思想是使用基于有限指针的目录。不是使用共享者列表的位掩码，而是使用基于2进制编码的数字，最多到某个上限。这就是为什么它被称为有限目录或有限指针目录，因为无法将系统中的所有节点都列在这个列表中，条目数不够。但你可以命名它们，因为你有实际数字的二进制编码。如果共享者数量超过了整个列表的容量（例如，你有5个条目，突然出现了第6个共享者，需要移除一个），通常会有一个溢出位，表示共享者数量超过N。当数量超过N且处于共享状态，突然需要转换为修改状态时，系统将直接向系统中的所有其他缓存发送广播消息或无效化消息。通常，这是一个很好的权衡，因为极广泛共享的缓存行并不常见，这是在存储空间与发送更多消息之间的有趣权衡。

*   **无限制目录**：这里有一个有趣的协议叫做“无限制”目录。思路类似，是一个带溢出位的有限目录。但如果发生溢出，你开始在软件中、在主内存的一个结构体中维护共享者列表。这要求系统具有非常快速的中断/陷阱机制，以便当这种情况发生时（因为你正在处理缓存行），中断主处理器，由主处理器提供共享者列表的其余部分（例如，当共享者列表溢出时）。在这两者之间还有很多研究，并且这个领域目前仍有活跃的未来研究。

除了简单的目录一致性，我们还有一些非常酷的片上一致性协议。这也是当前被大量研究的原因，因为人们过去构建了大规模并行的基于目录的机器，它们在某些应用中表现很好。但现在我们开始将越来越多的核心集成到单个芯片上，因此开始看到片上一致性，并研究如何利用快速的片上通信与目录结合，来构建更有趣的一致性协议。

还有所谓的**COMA系统**（全缓存内存架构），其中数据从不驻留在主内存中，目录四处移动。这有点超出本课程范围，如果想了解，可以阅读关于KSR1的资料。此外，如何扩展共享者列表也是一个活跃的研究方向。

## 现代系统实例与总结 🖥️

最后，简要谈谈这些技术的最新版本。我们有**SGI UV 1000**，它是SGI Origin和Origin 2000机器的后代。它拥有大量核心（2560个），全部保持缓存一致性。它使用基于目录的一致性协议，是非均匀内存访问架构。所有核心通过一个多机箱的2D Torus网络连接在一起。普林斯顿大学的H PCCRC中心就有一台这样的机器。

一个**片上示例**是**Tilera Tile 64 Pro**，它有64个核心，每个核心本身就是一个目录主节点，并运行基于目录的协议。我之前谈到将通信流量划分为不同的流。在这个芯片上，有三个不同的内存网络。因此，我们必须设计出三个自身不会死锁的流量类别。还有四个内存控制器直接连接到互连系统。由于这种设计，通信延迟各不相同。这里的核心与内存控制器通信非常快，而那里的核心则需要更长时间。但这个核心可能离那个内存控制器很近，所以存在非均匀性。
