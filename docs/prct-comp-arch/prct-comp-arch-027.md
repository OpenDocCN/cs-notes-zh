## 课程概述

上节课我们介绍了顺序处理器和记分牌机制，并讨论了前端顺序、发射顺序但写回和提交乱序的处理器。本节课，我们将探讨性能更高的设计，包括前端顺序、后端乱序的处理器，以及前端顺序、所有后端阶段（包括提交）都乱序的处理器。最后，我们将介绍性能最高的设计：顺序取指、乱序发射、乱序执行、乱序写回，但顺序提交的处理器。这种设计可以在保证精确异常的同时实现高性能。

## 从顺序前端与乱序后端开始

上一节我们介绍了前端和发射都顺序的处理器。本节中，我们来看看一种更常见的折中方案：**前端顺序，但后端（写回和提交）乱序**的处理器。这种设计需要引入新的硬件结构来管理指令的乱序完成。

为了实现这种设计，我们需要两个关键组件：
1.  **重排序缓冲区**：用于在指令乱序执行完毕后，按程序顺序重新排序，以便顺序提交。
2.  **存储缓冲区**：用于暂存乱序的存储操作，确保它们按顺序更新内存。

以下是这种处理器需要解决的核心问题：乱序提交可能导致**精确异常**难以实现。如果一条指令的结果在提交（即永久更新架构状态）之后，才发现它前面的指令发生了异常，那么处理器状态就已经被错误地修改了。

## 理解乱序提交的挑战

为什么乱序提交看起来有些奇怪呢？关键在于异常处理的精确性。

假设提交点位于流水线末端。在乱序提交的处理器中，完全有可能出现这种情况：一条指令的结果已经写入了**架构寄存器文件**，但随后才发现这条指令之前（按程序顺序）的某条指令触发了异常。这意味着处理器已经提交了不按程序顺序的、可能无效的结果。

我们来看一个简化的例子：
1.  指令A（可能引发异常的指令，如除零）尚未执行完毕。
2.  指令B（与A无关的指令）已执行完毕并将其结果提交到寄存器文件。
3.  此时，指令A的异常才被检测到。
这时，架构状态（寄存器B的值）已经被一个在异常指令之后才应该生效的结果污染了，破坏了程序的正确状态。本节课后续会展示具体的例子。

## 高性能设计：顺序取指与提交，乱序执行

最后，我们来探讨性能潜力最高的设计：**顺序取指、乱序发射、乱序执行、乱序写回，但顺序提交**的处理器。

这种设计可以在流水线末端设置一个提交点，从而支持**精确异常**。为了实现乱序发射，我们需要引入一个新的组件：**发射队列**。

发射队列是指令在准备好被发射到执行单元之前“生活”的地方。它允许处理器查看一窗口的指令，并动态地选择那些操作数已就绪的指令优先发射，从而打破程序顺序的限制，充分利用执行单元。

## 回顾激励示例

在整个课程中，我们将反复使用同一个激励示例来理解不同设计的优劣。考虑以下两条独立的指令序列：

```
序列1: LD F2, 0(R1)      ; 加载
        MULTD F4, F2, F0  ; 乘法
        ADDD F6, F4, F2   ; 加法

序列2: LD F8, 0(R2)      ; 加载
        MULTD F10, F8, F0 ; 乘法
        ADDD F12, F10, F8 ; 加法
```

这两组指令序列彼此没有依赖关系。如果你的机器有足够的并行资源，理论上应该可以并行执行这些操作。不同的处理器设计实现这种并行的能力各不相同。

## 本节课总结

![](img/02d78553e689b6281e5f9d8e1b769395_4.png)

本节课我们一起深入探讨了乱序处理器的多种设计形态。我们从前端顺序、后端乱序的设计入手，引入了**重排序缓冲区**和**存储缓冲区**的概念。我们分析了乱序提交对**精确异常**带来的挑战。最后，我们介绍了目前高性能处理器中常见的设计模式：顺序取指与提交，但中间阶段（发射、执行、写回）全乱序，这需要依赖**发射队列**来实现灵活的指令调度。理解这些设计之间的权衡，是掌握现代处理器如何挖掘指令级并行的关键。
