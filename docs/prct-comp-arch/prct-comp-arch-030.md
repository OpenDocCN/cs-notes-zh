# 030：乱序执行处理器详解

![](img/ab561dd7c4abfca9651af9c5e6534874_0.png)

在本节课中，我们将深入探讨一种重要的处理器设计：**顺序取指、乱序发射、乱序写回、乱序提交**的处理器。我们将了解其核心组件——**发射队列**的工作原理，并通过对比分析其性能特点。

## 概述

上一节我们讨论了顺序发射处理器的一些局限性。本节中，我们来看看如何通过引入**乱序发射**机制来提升性能。核心思想是：指令仍然按顺序被取指和解码，但允许它们在操作数就绪时，**不按顺序**地进入执行单元。

## 发射队列：乱序发射的核心

为了实现乱序发射，我们需要引入一个新的数据结构：**发射队列**。它本质上是一个缓冲区，指令按顺序进入，但可以被乱序地取出并发射到执行单元。

以下是发射队列内部条目的关键字段（以MIPS指令为例）：

*   **操作码**：指令本身。
*   **立即数字段**：指令中的立即数。
*   **源操作数1/2**：
    *   **寄存器标识符**：需要读取的寄存器编号（如 `R5`）。
    *   **V位**：指示该指令**是否使用**此源操作数。例如，某些指令可能只使用一个源。
    *   **P位**：指示该源操作数的值**是否尚未就绪**。如果为1，表示流水线中后续有一条指令会写入该寄存器。
*   **目的操作数**：
    *   **寄存器标识符**：指令将要写入的寄存器编号。
    *   **V位**：指示该指令**是否写入**目的寄存器。

## 发射队列如何工作

发射队列的逻辑是动态的。每个周期，硬件会扫描队列中的所有指令，检查其**就绪状态**。一条指令就绪的条件是：
1.  所有它**需要使用**的源操作数（V位=1）的P位都为0（即值已就绪）。
2.  目标执行单元没有**结构冲突**（例如ALU正忙）。

一旦指令就绪且资源可用，它就可以被**乱序地**从发射队列中取出，进入“发射”阶段，随后进行寄存器读取并开始执行。

当一条指令在流水线末尾**提交**时，它会从发射队列中被移除。同时，处理器会扫描队列中所有其他条目，如果发现有指令的源操作数寄存器标识符与刚提交指令的目的寄存器标识符匹配，则将该源操作数的**P位清零**，表示该值现在已就绪。这可能会唤醒一条或多条等待中的指令。

## 集中式与分布式发射队列

在逻辑上，一个大的**集中式发射队列**似乎很理想，可以全局查看所有指令。但其实现可能较复杂，且所有指令类型（如整数、浮点）混在一起，通信开销可能较大。

另一种设计是**分布式发射队列**。例如，可以为整数ALU、浮点单元、乘法器等不同的功能单元分别设置独立的发射队列。解码后的指令被“引导”到相应的队列中。这种设计在早期多芯片处理器中很常见，在现代高度集成的芯片中也有其优势，例如能更好地匹配不同功能单元的特性。Tomasulo算法论文就早期探讨了这种分布式设计。

## 流水线性能分析

让我们通过一个流水线时序图的例子，来观察这种处理器的行为。

![](img/ab561dd7c4abfca9651af9c5e6534874_0.png)

图中，`I` 表示指令进入发射队列。关键点在于，即使指令很早就进入了发射队列，它的实际发射也可能被延迟，原因包括：
*   **数据依赖**：等待前一条指令产生源操作数。
*   **结构冲突**：所需功能单元正被占用。
*   **单周期单发射限制**：每个周期只能发射一条指令，即使有多条指令就绪，也必须排队。

一个有趣的发现是，即使我们预先将所有指令加载到发射队列中，最终性能可能**并不会比指令逐条流入队列的情况更好**。因为性能瓶颈转移到了**发射带宽**（单周期只能发射一条）和**执行资源数量**（只有一个ALU）上。

这自然引出了下一步的优化方向：**超标量处理器**。即，通过复制硬件资源（如多个发射端口、多个ALU），实现每个周期发射并执行多条指令，从而进一步提升性能。

## 关于记分牌的补充说明

之前我们讨论的记分牌主要跟踪**写后读**危害。在更复杂的乱序流水线中，我们还需要考虑**写后写**危害。此时，记分牌可能不仅需要记录寄存器状态，还需要记录是**哪个功能单元**将要写入该寄存器，并在流水线中传递这个信息，以确保写入顺序的正确性。

## 总结

本节课中我们一起学习了：
1.  **乱序发射处理器**的基本模型：顺序取指，乱序发射/执行/写回/提交。
2.  实现乱序发射的核心组件——**发射队列**的结构与工作原理，特别是其通过 **V位** 和 **P位** 来跟踪数据依赖和就绪状态的机制。
3.  发射队列的两种组织形式：**集中式**与**分布式**。
4.  通过流水线分析发现，仅实现乱序发射而不增加发射带宽和执行资源，性能提升可能受限，这为引入**超标量设计**提供了动机。
5.  提到了在复杂流水线中，记分牌需要扩展以处理**写后写**危害。
