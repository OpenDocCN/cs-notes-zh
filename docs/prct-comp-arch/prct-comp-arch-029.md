# 029：乱序执行处理器（I2O1）详解 🚀

![](img/1a3d37887a39d6f8890054432aae2347_0.png)

在本节课中，我们将学习一种更复杂的处理器设计：顺序前端、乱序发射、乱序写回、顺序提交（I2O1）。这种设计能够解决精确异常处理等关键问题。我们将详细探讨其内部结构，特别是重排序缓冲区和未来存储缓冲区。

## 概述

上一节我们介绍了顺序处理器。本节中，我们来看看一种混合设计：I2O1处理器。它的前端（取指、译码）和提交阶段是顺序的，但发射和写回阶段允许乱序执行。这种设计的核心优势在于能够实现**精确异常**，因为指令的最终提交（更新架构状态）是严格按照程序顺序进行的。

## 新增的关键结构

以下是I2O1处理器中引入的核心组件：

*   **物理寄存器文件**：也称为未来文件。它保存处理器**推测执行**的中间结果。这些值尚未提交，如果发生异常或分支预测错误，可以被丢弃。
*   **架构寄存器文件**：代表处理器的**已提交状态**。只有在指令提交时，其结果才会从物理寄存器文件写入这里。
*   **重排序缓冲区**：一个关键的数据结构，用于确保乱序执行的指令能够**按顺序提交**。指令完成（写回）后进入ROB，但只有成为最旧的已完成指令时才能提交。
*   **未来存储缓冲区**：用于暂存已执行但尚未提交到主存的存储操作（地址和数据）。加载指令需要优先检查此缓冲区，以确保读到最新的数据。

## 数据读写路径分析

现在，我们来看看数据在这些结构中的流动。

*   **架构寄存器文件**：在正常执行中**不被读取**。它仅在发生异常或分支预测错误需要恢复架构状态时被使用。
*   **物理寄存器文件**：用于流水线中所有中间值的读写。记分板现在跟踪的是物理寄存器，而非架构寄存器。
*   **重排序缓冲区**：
    *   **写入**：在指令发射阶段分配一个ROB条目；在指令完成阶段更新其状态。
    *   **读取/清理**：在指令提交阶段，根据ROB中的信息，将物理寄存器文件的值写入架构寄存器文件，并清理该ROB条目。
*   **未来存储缓冲区**：在存储指令执行结束时写入。当存储最终提交到内存系统时被清理。加载指令可能需要从此缓冲区读取数据（旁路）。

## 深入理解重排序缓冲区

重排序缓冲区是保证顺序提交的核心。我们可以将其视为一个循环队列，有头指针和尾指针。

以下是ROB条目中需要存储的最小信息集合：

*   **状态**：表示指令状态，例如 `P`（等待中）、`F`（已完成）。
*   **推测位**：标记该指令是否位于一个尚未解决的分支之后。如果分支预测错误，可以快速找到并刷新所有带此标记的指令。
*   **存储位**：标记该指令是否为存储操作，以便在提交时处理未来存储缓冲区。
*   **目标有效位**：指示该指令是否写寄存器。
*   **物理寄存器编号**：指向物理寄存器文件中存储该指令结果的位置。ROB本身不存数据值，只存指针，以节省空间。

**提交规则**：只有**最旧的**指令状态变为“已完成”时，才能将其提交，并从ROB中移除。即使后面的指令先完成，也必须等待。

## 异常与提交点

乱序写回、顺序提交的设计如何实现精确异常呢？

1.  当一条指令（如乘法）执行并发生异常时，它将其结果写入**物理寄存器文件**（推测状态）。
2.  该指令在ROB中被标记，但**不会**被提交到架构寄存器文件。
3.  由于提交是顺序的，所有在该异常指令**之后**的、仍在流水线中的指令都无法提交。
4.  当处理器处理异常时，它可以将**架构寄存器文件**的完整状态复制到**物理寄存器文件**，从而一次性回滚所有推测状态。有些设计通过重命名指针来实现，效率更高。

## 分支预测错误处理

对于分支预测错误，ROB中的推测位提供了便利。

1.  分支指令本身被提交后，其方向确定。
2.  如果分支预测错误，处理器需要清除在错误路径上取出的所有指令。
3.  通过检查ROB中所有标记为“推测”的条目，处理器可以快速识别并**无效化**这些条目，同时清空流水线。
4.  性能考虑：可以立即无效化（需要多端口ROB支持），也可以让错误路径指令继续流到提交阶段再清理（性能较低，占用资源）。

## 处理存储缺失停顿

顺序提交可能因存储操作缓存缺失而引发性能问题。如果一个存储指令需要等待数据从下级缓存调入，它会阻塞提交阶段，延迟后面所有指令的提交。

**解决方案**：将“提交到处理器状态”与“写入内存系统”**解耦**。
*   存储指令在通过提交阶段后，即被视为**已提交**（可以从ROB中移除）。
*   其数据被放入未来存储缓冲区，并允许其**异步地**完成向内存的写入。
*   后续的加载/存储指令需要通过旁路逻辑检查这个缓冲区。
*   这样，提交阶段就不会被慢速的存储操作阻塞，提高了整体吞吐量。
