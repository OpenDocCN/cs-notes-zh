在本节课中，我们将学习并发编程中的核心概念：锁与信号量。我们将从一个生产者-消费者模型的例子出发，探讨当多个消费者同时存在时可能出现的问题，并引入锁和信号量作为解决方案。

---

## 回顾生产者-消费者模型

上一节我们介绍了顺序一致性模型。现在，让我们回到生产者-消费者的例子。

我们有一个生产者生产数据，一个消费者消费数据。在顺序一致性模型下，之前展示的代码序列可以正常工作，因为该模型禁止了可能导致问题的读写指令重排。

## 多消费者引发的问题

但是，当我们引入两个消费者时，情况发生了变化。

以下是我们的原始代码片段。其中一个关键操作是检查头指针 `head`，以判断队列中是否有数据可用。

```c
if (head != tail) {
    // 读取数据
}
```

如果有两个线程或进程同时执行这段代码，它们可能会同时通过这个检查。接下来可能发生的情况是，它们会尝试读取队列中的同一个值。

假设消费者1和消费者2的指令以交替的方式执行。最终结果是，两个消费者从队列中读出了相同的值。这显然不是我们期望的行为。

## 引入临界区与互斥

我们希望保证代码中的这个特定块（即检查并读取数据的部分）在执行时，没有其他线程或进程正在执行相同的代码块。

因此，我们需要引入锁和信号量的概念。

基本思想是，使用一个“守卫”——它可以是一段硬件机制或一个内存位置——来保护一段代码（称为临界区）的执行。

## 锁的类型：互斥锁与信号量

这种保护机制主要有两种形式：

1.  **互斥锁**：确保同一时间只有一个进程或处理器能够执行临界区代码。这实现了互斥访问。
    *   **核心概念**：`Mutex` (Mutual Exclusion)

2.  **信号量**：这是一个更通用的概念。它允许最多 `N` 个进程同时进入临界区。
    *   **核心概念**：`Semaphore`，其状态由一个整数值 `N` 控制。

为什么需要信号量呢？举个例子，假设你的网卡有两个出站队列，你有 `P` 个处理器。你希望允许两个人同时使用这些资源（但不关心具体是哪两个人），但绝不能超过两个。这时，仅允许单线程访问的互斥锁就不够用了，我们需要更通用的信号量机制。
