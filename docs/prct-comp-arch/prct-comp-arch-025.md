# 025：中断与异常 🖥️

![](img/8c0fff1717b2a3b2474b0f0b5ab93af7_0.png)

在本节课中，我们将要学习计算机体系结构中的中断与异常机制。理解这些概念对于掌握处理器如何处理意外事件、实现操作系统功能以及构建高性能的乱序执行处理器至关重要。

## 概述

中断与异常是处理器响应内部或外部事件的机制，它们会临时打断正常的指令执行流程，转而执行特定的处理程序。本节将探讨其类型、处理流程以及在流水线中的实现细节。

## 中断与异常的定义

中断是处理器响应外部或内部事件的一种机制。它可能是与指令执行同步发生的，也可能是由外部事件异步触发的。中断会暂时将控制流重定向到特定的处理程序，处理完毕后通常会返回原指令流继续执行。

以下是中断处理的基本流程示意图：
1.  程序正常执行指令 I-1。
2.  在执行指令 I 时，发生中断，指令 I 并未实际提交。
3.  处理器跳转到中断处理程序执行。
4.  处理程序执行完毕后，可能返回并重新执行指令 I，也可能不返回（例如，在发生致命错误时）。

## 中断与异常的类型

中断主要分为两大类：异步中断和同步中断。

### 异步中断

异步中断，也称为外部中断，其发生时间不与特定指令的执行严格同步，因此难以确定将其关联到流水线中的哪条指令。以下是异步中断的常见例子：
*   **设备中断**：例如，网络卡接收到数据包。
*   **定时器中断**：例如，X86处理器上的定时器周期性触发。
*   **硬件故障**：例如，ECC内存错误。

处理多个同时发生的异步中断时，通常需要一个**优先级编码器**来决定处理顺序。有些系统允许操作系统通过可编程中断控制器来动态调整中断优先级。

![](img/8c0fff1717b2a3b2474b0f0b5ab93af7_2.png)

### 同步中断

![](img/8c0fff1717b2a3b2474b0f0b5ab93af7_4.png)

同步中断，通常称为异常或陷阱，其发生与正在执行的指令直接相关。以下是同步中断的常见例子：
*   **非法指令异常**：尝试执行不在ISA手册定义范围内的指令。
*   **特权指令异常**：用户程序尝试执行只有操作系统才能执行的指令。
*   **算术溢出/下溢异常**：运算结果超出数据类型的表示范围。
*   **浮点异常**：例如，产生非规格化数。
*   **除零异常**：尝试进行除以零的操作。
*   **非对齐内存访问异常**：在某些架构上，访问未按字长对齐的内存地址会触发此异常。
*   **缺页异常**：尝试访问未正确映射的物理内存。
*   **系统调用**：在X86上，通过`int`指令主动触发中断以实现系统调用。

## 中断处理机制

当发生中断时，硬件需要执行一系列操作来保存当前状态并跳转到处理程序。

以下是中断发生时硬件需要执行的关键步骤：
1.  **停止当前程序**：在某个点暂停当前指令流的执行。
2.  **保存程序计数器（PC）**：为了将来能返回到被中断的指令，需要将当前的PC值保存起来。保存的位置可能是一个专用的异常PC寄存器（如MIPS中的EPC），也可能被压入内存栈中（如X86）。
3.  **保存处理器状态**：中断处理程序开始执行时，原有的寄存器值仍然是有效的，不能被丢弃。不同架构有不同的保存机制：
    *   **MIPS**：预留两个专用寄存器（如`$k0`, `$k1`）供中断处理程序使用，用于将其他寄存器保存到内存。
    *   **X86**：硬件机制（如`pusha`指令）可以将多个寄存器自动压栈，现代操作系统则倾向于只保存必要的寄存器。
4.  **屏蔽其他中断**：为了防止在中断处理程序中再次被中断（即中断嵌套），通常需要暂时屏蔽（关闭）其他中断。处理完成后，再通过返回指令（如`iret`）重新启用中断。更复杂的系统可能允许在将状态安全保存到内存后，在处理程序中重新启用中断以实现嵌套。

## 流水线中的中断处理

在流水线处理器中，中断或异常可能在流水线的不同阶段被检测到。

以下是流水线各阶段可能产生的异常示例：
*   **取指阶段**：PC地址异常（如访问非法的内存区域）。
*   **译码阶段**：非法操作码异常。
*   **执行阶段**：算术溢出/下溢、浮点异常。
*   **访存阶段**：数据地址异常（如缺页、非对齐访问）。

### 处理原则

当流水线中多个阶段同时产生异常时，需要遵循以下原则：
1.  **优先级**：异常处理的优先级应从流水线前端（取指）到后端（写回）依次降低。最老的指令（流水线中最靠前的指令）产生的异常具有最高优先级。
2.  **提交点**：处理器需要确定一个**提交点**，在此点之后，指令对架构状态的更改变得不可撤销。通常，提交点位于流水线末端附近（如访存或写回阶段之后），以确保所有可能的异常都已发生并被检测到。
3.  **处理流程**：检测到异常后，相关信息会随指令在流水线中传递。直到提交点，处理器确认该异常需要处理，然后执行以下操作：
    *   将**异常PC**（可能是触发异常的指令地址，也可能是其下一条指令的地址，取决于架构定义）保存起来。
    *   向流水线前端发送**清除信号**，冲刷掉该异常指令之后的所有后续指令。
    *   开始从中断向量表指定的地址取指，执行中断处理程序。

### 精确异常与不精确异常

*   **精确异常**：处理器能够精确地停止在导致异常的指令处，并确保该指令之前的所有指令都已完成，之后的指令都未执行。这是现代通用处理器所需支持的特性。
*   **不精确异常**：处理器无法精确确定异常与指令的对应关系，可能允许异常指令之后的指令对状态产生部分影响。这通常出现在一些简单的嵌入式处理器中，不利于运行复杂的操作系统。

## 异常处理与推测执行

现代高性能处理器采用推测执行，包括控制推测（分支预测）和异常推测。

以下是异常推测的处理方式：
1.  **乐观推测**：处理器默认指令不会发生异常，继续预取和执行后续指令。
2.  **延迟处理**：即使在执行阶段检测到异常（如算术溢出），也不会立即处理，而是将异常标记随指令传递。
3.  **提交时确认**：在提交点，如果确认异常发生，则启动恢复机制：
    *   在简单的有序流水线中，只需**冲刷流水线**，从异常处理程序重新开始。
    *   在乱序执行流水线中，情况更复杂。因为指令可能乱序执行和完成，需要更复杂的机制（如**重排序缓冲区**和**影子寄存器**）来确保能够精确地回滚到异常发生时的状态，这将是后续课程讨论的重点。
4.  **旁路与推测**：在异常指令结果产生后到提交点之前，其数据可能已经通过旁路网络传递给后续的推测执行指令。这是允许的，但如果最终异常被确认，这些后续的推测指令及其产生的影响都将被清除。

本节课中我们一起学习了中断与异常的核心概念。我们了解了异步中断与同步异常的区别及其常见类型，深入探讨了中断发生时硬件保存状态、跳转处理程序的关键步骤。我们重点分析了在流水线处理器中处理多阶段异常的原则，包括优先级、提交点的概念，以及实现精确异常的必要性。最后，我们触及了异常处理与推测执行（特别是乱序执行）之间的交互，认识到高性能处理器需要复杂的机制来保证异常处理的精确性。理解这些机制是掌握现代处理器设计和操作系统原理的基础。
