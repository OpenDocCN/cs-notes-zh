## 概述


取指和发射逻辑在这种情况下需要交换指令的位置。第一条加法指令进入A单元，第一条加载指令进入B单元，没有问题。

接下来是第二条加载指令，它被取指后位于指令寄存器0，这意味着它希望进入A流水线。但我们需要交换这两条指令。如图所示，我们实际上让这条加法指令进入了A流水线，这与之前的安排相反。

在这个例子中，仍然存在停顿。最后，这里会出现一个结构冒险。这个结构冒险引入了停顿。我们同时取指了这两条加载指令，但一次只能执行一条加载操作。因此，我们需要在译码阶段停顿其中一条加载指令，并将其执行稍微推后。这导致其流水线图与没有冲突或结构冒险的例子不同。

## 双发射数据相关
现在，我们来看一个更复杂的例子：双发射数据相关。当存在数据相关时会发生什么？

不幸的是，当存在数据相关时，即使没有旁路，也会出现问题。这里的前两条指令没有数据相关。但第三条指令向寄存器5写入，紧接着下一条指令从寄存器5读取。这是一个写后读相关。

由于在这个流水线中还没有实现旁路，我们实际上必须停顿第二条指令，等待第一条指令完成，尽管它们本可以同时执行。但这里存在真实的数据相关，因此我们需要在第二条指令中引入停顿周期。

如果实现了完全旁路，我们可能仍然需要停顿。但现在，我们不必等待这个值到达流水线末端才在ALU中使用它。我们可以通过旁路，在A0阶段之后就将加法结果传递回来。

这里看到的是相同的指令序列，但现在值从A0阶段旁路到了译码阶段，我们可以更快地继续执行。因此，旁路在这里对我们帮助很大，并且它与超标量特性是交叉作用的。

指令顺序很重要。这里我们交换了最后两条指令，只是将它们翻转了一下。我们将一个写后读相关，变成了一个读后写相关。正因为如此，执行实际上提前了一个周期，没有产生停顿。仅仅通过改变指令的顺序，就会改变数据依赖关系，从而改变执行顺序和长度。

## 取指逻辑与对齐问题总结
我想简要总结一下取指逻辑和对齐问题。让我们看一些包含跳转指令的代码。

```
地址        指令
...         ...
...         jump to 0x100
...         op1
...         jump to 0x204
...         op2
0x100       op3
0x104       op4
...         jump to 0x30C
0x204       op5
0x208       op6
0x30C       op7
0x310       op8
...
```

这是我们的缓存。假设缓存块大小为4条指令长。我们来看看执行这段代码需要多少个周期。

首先，假设没有对齐约束。在周期0，我们执行前两条指令，从指令缓存中取出它们，它们整齐地排列在一起，没有问题。接下来的两条指令A和C彼此相邻，很好。然后我们跳转到地址0x100，执行那里的两条指令，它们也在开头并对齐，没有问题。

现在开始遇到一些奇怪的情况。我们跳转到了一个缓存行的中间。在这个例子中，我们跳转到了地址0x204。我们的块大小是4条指令，但我们跳转到的不是该块的第一条指令。在一个功能完善的取指单元中，假设可以在任何对齐方式下执行，那么生活就很轻松。我们可以在周期3同时取出并执行这两条指令。

如果跳转到缓存块的末尾（例如地址0x30C），并尝试同时取出两条指令，那么一条指令在这个缓存行，另一条在下一个缓存行。我们是否需要同时从缓存中取出两个东西？是的，如果需要同时执行这条指令和那条指令，就需要。假设目前的发射逻辑允许我们这样做，比如指令缓存是双端口的。最后，指令Op5在这里执行完毕，只是顺序执行，没有跳转。

实际上很难实现的事情包括跨缓存行取指，以及根据取指单元逻辑，可能还需要在缓存行内任意位置取指。正如所说，我们可能需要在缓存上增加额外的端口。

这段代码执行时，实际上没有引入任何停顿，只是顺序执行，每个周期执行两条指令。

## 带有对齐约束的情况
现在，让我们看看带有对齐约束的情况。这是我们最初的例子，看看在这里可能尝试执行什么。我们跳转时，只使用了行中间的两条指令。

假设每个周期只能取半个块或类似的大小，因为我们的缓存宽度就是如此。在一些体系结构中，如果存在这样的对齐问题，并且不允许跨行取指，你可能不得不取出一些永远不会用到的额外数据。这样就会浪费带宽。

同时，执行周期也会改变。让我们看看相同的代码序列，在存在对齐约束时执行会发生什么。

回到这里。我们执行指令A和B，流水线良好。当执行到地址0x8时，需要交换指令，因为跳转指令需要进入A流水线，但除此之外一切正常。

现在，我们跳转到一个缓存行的中间，情况开始变得更有趣。我们基本上最终会浪费周期。这将需要7个周期，而之前只需要5个周期，因为我们有效地引入了死周期，取出了我们根本不使用的指令。

这里的三个“X”表示我们取出的指令。例如，地址0x200处的指令，我们取出了它，但没有使用。我们取出了这两条指令，但都没有使用。

因此，拥有一个复杂的取指单元，或者一个不能完全旁路、或对齐不友好的取指单元，可能会给我们的性能带来严重问题。

## 总结
本节课中，我们一起学习了超标量处理器中的取指逻辑与指令对齐。我们分析了双发射如何提升理想情况下的指令吞吐率（CPI可达0.5），也探讨了数据相关和结构冒险如何引入停顿。通过对比有无对齐约束的取指过程，我们认识到指令对齐和缓存设计对处理器性能至关重要，不当的设计会导致带宽浪费和周期增加。下一节课，我们将继续讨论相关话题。
