# 088：无原子操作的互斥实现 🧠

![](img/acb052197e2ed8af6ba4dc2a17526344_0.png)

在本节课中，我们将探讨如何在不依赖原子操作的情况下实现互斥。这是一个复杂但基础的话题，对于理解并发编程的本质至关重要。我们将从简单的尝试开始，逐步分析其问题，最终介绍经典的Dekker算法和面包店算法的基本思想。

## 概述

互斥是确保多个线程安全访问共享资源的关键机制。通常，我们依赖如`test-and-set`这样的原子操作。但如果没有这些特殊指令，仅使用普通的加载和存储操作，我们能否实现互斥？本节课程将深入探讨这个问题，重点分析仅有两个线程参与的简单场景。

## 第一次尝试：简单的标志检查

首先，我们来看一个最直观的实现方法。每个线程使用一个独立的标志变量来表示自己是否想进入临界区。

```c
// 线程 1
c1 = 1;
while (c2 == 1) {} // 等待
// 临界区
c1 = 0;

// 线程 2
c2 = 1;
while (c1 == 1) {} // 等待
// 临界区
c2 = 0;
```

**核心逻辑**：线程在进入临界区前，先设置自己的标志，然后循环检查另一个线程的标志。只有当对方标志为0时，自己才能进入。

然而，这种方法存在一个致命缺陷。如果两个线程几乎同时设置自己的标志为1，然后同时去检查对方的标志，它们会发现对方也想进入临界区（标志为1），于是双方都会陷入无限等待，导致**死锁**。

## 第二次尝试：失败后释放标志

为了解决死锁问题，我们进行改进：如果线程检查发现冲突（即对方标志也为1），则主动释放自己的标志，稍后重试。

```c
// 线程 1
while (1) {
    c1 = 1;
    if (c2 == 0) break; // 成功获取
    c1 = 0; // 释放标志并重试
}
// 临界区
c1 = 0;

// 线程 2
while (1) {
    c2 = 1;
    if (c1 == 0) break; // 成功获取
    c2 = 0; // 释放标志并重试
}
// 临界区
c2 = 0;
```

这个方案避免了死锁，但引入了新的问题：**活锁**。如果两个线程完美地同步执行，它们可能会持续重复“设置标志 -> 发现冲突 -> 释放标志”的循环，谁也无法进入临界区。此外，该方案也无法保证公平性，一个线程可能会长时间“饥饿”，始终无法获得锁。

## Dekker算法：引入共享的“轮次”变量

为了解决活锁和饥饿问题，我们需要一个更精巧的协议。这就是**Dekker算法**。它在之前的基础上引入了一个共享变量`turn`，作为打破平局的“轮次”指示器。

以下是Dekker算法的核心代码框架：

```c
// 共享变量
int c1 = 0, c2 = 0, turn = 1;

// 线程 1
c1 = 1;
while (c2 == 1) {
    if (turn != 1) {
        c1 = 0;
        while (turn != 1) {} // 等待轮次
        c1 = 1;
    }
}
// 临界区
turn = 2;
c1 = 0;

// 线程 2
c2 = 1;
while (c1 == 1) {
    if (turn != 2) {
        c2 = 0;
        while (turn != 2) {} // 等待轮次
        c2 = 1;
    }
}
// 临界区
turn = 1;
c2 = 0;
```

**算法原理**：
1.  线程首先声明自己希望进入临界区（设置`c1`或`c2`为1）。
2.  如果检查发现对方也在等待（`c2 == 1`或`c1 == 1`），则查看`turn`变量。
3.  如果`turn`不属于自己，则线程会礼貌地暂时降低自己的标志（`c1=0`或`c2=0`），并等待`turn`变为自己的轮次。
4.  一旦轮到自己，它再重新声明意愿（`c1=1`或`c2=1`），并再次检查。
5.  成功进入临界区后，在退出时，线程将`turn`设置为另一个线程的编号，从而将进入机会“传递”给对方。

`turn`这个共享变量起到了两个关键作用：一是作为冲突时的**裁决者**，强制其中一个线程等待；二是作为**公平性保证**，确保两个线程能够交替进入临界区，防止饥饿。

## 扩展到更多线程：面包店算法思想

Dekker算法完美解决了两个线程的互斥问题。但对于**N个线程**的情况，实现会变得非常复杂。Dekker本人也证明了这一点。一个更易于理解和推广的算法是**面包店算法**。

其灵感来源于现实中的面包店或熟食店：顾客进店时先取一个号码，然后等待叫号。号码最小的顾客有权获得服务。

**算法的基本思想**：
1.  线程在“取号机”上获取一个号码。这个“取号”操作需要以某种方式生成一个递增的、唯一的号码。
2.  在进入临界区前，线程需要检查所有其他线程。它必须等待，直到满足以下条件：
    *   自己没有拿到号码。
    *   或者，自己是所有已取号线程中号码最小的那个。
3.  这就像在等待屏幕上的叫号器叫到自己的号码。

在计算机中实现时，挑战在于如何在没有中央叫号器（原子操作）的情况下，让线程安全地“取号”并判断自己是否拥有最小号码。面包店算法通过一组共享变量和特定的检查顺序巧妙地解决了这个问题。它属于更广泛的**取票锁**类别，能有效防止多个线程一拥而上造成混乱。

## 总结

本节课我们一起学习了在不依赖硬件原子操作的情况下实现互斥的挑战与方法。
*   我们首先看到了简单的标志检查会导致**死锁**。
*   改进后的“失败释放”方案避免了死锁，但可能引起**活锁**和**饥饿**。
*   **Dekker算法**通过引入一个共享的`turn`变量作为裁决者和公平性保证，成功地为两个线程实现了正确且公平的互斥。
*   对于多线程场景，**面包店算法**（取票锁）提供了一种优雅的思路，通过模拟现实中的排队机制来实现互斥。
