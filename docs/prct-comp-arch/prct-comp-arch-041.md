# 041：谓词执行简介 🧠

![](img/d771423952a234c590d220d71ab31644_0.png)

在本节课中，我们将探讨超长指令字架构面临的一个关键问题——难以预测的分支，并学习一种名为“谓词执行”的解决方案。我们将从基本概念入手，逐步理解如何通过将控制流转换为数据流来提升指令级并行性。

---

## 概述

超长指令字架构在处理难以预测的分支时，其指令级并行性会受到限制。本节将介绍“谓词执行”技术，它通过添加特殊的硬件指令来消除分支，从而避免分支预测错误带来的性能损失。我们将首先学习“部分谓词执行”，然后探讨其扩展形式“完全谓词执行”。

---

## 部分谓词执行

上一节我们讨论了VLIW架构的挑战，本节中我们来看看如何通过“谓词执行”来解决分支预测难题。其核心思想是移除分支指令，将控制流依赖转化为数据依赖。

为此，硬件需要添加两条特殊的指令。这两条指令的功能类似于C语言中的三元条件运算符（`? :`）。

**核心指令公式如下：**
*   `move_if_zero Rd, Rs`： 如果源操作数为0，则将Rs的值移动到Rd。
*   `move_if_not_zero Rd, Rs`： 如果源操作数不为0，则将Rs的值移动到Rd。

C语言中的三元运算符 `a = c ? d : e;` 可以理解为：如果条件c为真，则a赋值为d，否则赋值为e。上述两条指令实现了类似但更基础的功能。

这种设计的巧妙之处在于，它允许我们将控制流转换为数据指令。考虑以下使用分支的代码片段：

```assembly
slt  t0, a0, a1    # 计算条件 (a0 < a1)，结果存入t0
beqz t0, ELSE      # 如果t0为0（即条件为假），跳转到ELSE标签
add  t1, a2, a3    # if 块内的指令
j    END           # 跳过else块
ELSE:
sub  t1, a2, a3    # else 块内的指令
END:
```

这段代码包含两个控制流操作：一个条件分支`beqz`和一个无条件跳转`j`。使用谓词指令后，我们可以将这段`if-then-else`逻辑压缩到一条指令流中，从而完全消除分支。

这对于超长指令字处理器至关重要，因为分支预测错误会导致大量无效指令，编译器无法在预测错误路径上调度指令。而使用谓词后，编译器可以尝试调度非依赖操作，前提是它能在编译时生成并行代码序列。

以下是关于部分谓词执行的两个关键问题及其解答：

**1. 如果if或else块内包含多条指令怎么办？**
上述例子中每个块只有一条指令，情况比较简单。对于更复杂的情况，部分谓词执行的典型做法是：**交错执行if和else两个分支的所有指令**，然后通过条件移动指令在最后选择正确的结果。例如，在x86架构中这类指令常被称为`CMOV`，在MIPS中则称为`MOVZ`。这种做法并非世界末日，但意味着你可能需要执行一些原本不需要执行的额外指令。

**2. 如果if和else块的长度严重不平衡怎么办？**
这也不是无法解决的问题。如果一边有1000条指令，另一边只有2条，那么采用谓词执行可能会导致在常见情况下（执行2条指令的那一侧）额外执行1000条无用指令，这显然效率低下。此时，更明智的选择可能是保留一个实际的分支指令，而不是尝试进行谓词化。

## 完全谓词执行

部分谓词执行通过添加简单的条件移动指令来工作。现在，让我们看看它的扩展形式——完全谓词执行。

![](img/d771423952a234c590d220d71ab31644_2.png)

完全谓词执行的理念更进一步。它不再仅仅添加几条特殊指令，而是**允许指令序列中的几乎每一条指令（除了分支等少数指令外）都能根据一个谓词寄存器的值来决定是否执行（即被“无效化”）**。

它的工作原理如下：首先，架构会有一组额外的谓词寄存器，用于存储条件判断的结果。然后，在每条指令前，可以附加一个括号，里面指定一个谓词寄存器。只有当该谓词寄存器的值为“真”时，这条指令才会实际执行。

与部分谓词执行相比，完全谓词执行可以实现更复杂的操作。你不需要在最后进行数据移动或簿记操作，而是可以直接只执行所需分支路径上的指令，避免了执行额外工作。

Scott Mahlke在1995年的ISCA会议上指出，如果配合足够智能的编译器（例如他当时在UIUC的IMPACT编译器），使用完全谓词执行可以消除程序中大约50%的分支，特别是那些短小的分支。这项技术后来出现在HP的PlayDoh架构及其编译器中，该编译器正是基于UIUC的IMPACT项目。由此可见，这项技术能带来显著的性能收益。
