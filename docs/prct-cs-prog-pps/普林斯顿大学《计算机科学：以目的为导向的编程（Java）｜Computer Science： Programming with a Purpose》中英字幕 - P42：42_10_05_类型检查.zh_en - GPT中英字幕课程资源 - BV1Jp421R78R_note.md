## 静态类型检查与动态类型检查

上一节我们介绍了面向对象编程的核心思想。本节中，我们来看看确保程序正确性的关键机制——类型检查。

所有变量都必须有声明类型，系统在编译时检查类型错误。除非系统确认你指定的操作与所涉及的变量类型兼容，否则甚至无法运行你的程序。

另一种方法称为动态或运行时类型检查，这是 Python 等语言采用的方式。Java 也进行一些运行时检查，但让我们重点讨论这两种方式的对比。

其核心思想是：在动态类型系统中，是**值**而非**变量**拥有定义好的类型。因此，系统可以在即将执行操作时，即时检查类型是否正确。

以下是两种检查方式的对比：

*   **静态（编译时）类型检查**：在程序运行前（编译阶段）检查类型兼容性。
*   **动态（运行时）类型检查**：在程序执行过程中检查类型兼容性。

这两种是寻找类型错误的截然不同的方法。问题是，哪种最好？当然，你永远不能将一个字符串乘以一个双精度值，这是一个错误。问题在于，你应该在编译时还是运行时发现这个错误？

## 类型检查的争论

关于静态类型和动态类型哪个更好，存在着持续的激烈争论。

一些人认为静态类型不值得麻烦，为所有变量指定类型并检查它们太痛苦了。另一方面，如果我们知道类型没问题，得到的编译代码可能效率更高。经过仔细类型检查的代码可能更可靠。

但另一方面，也许有很多高级特性，比如泛型，我们想编写能处理多种不同类型数据的程序，使用静态类型可能太困难。还会出现许多其他问题，人们仍然在双方激烈争论。

---

## 类型检查 vs. 自动化测试

一个观点是，类型检查是寻找错误的一种方式，但我们还可以做另一件事：利用计算机，通过编写程序来生成大量输入，并根据我们生成的输入来检查程序，从而实现**自动化程序测试**。

早在 60 年代，我们之前提到过的 Dijkstra 就曾这样评价程序测试：“**程序测试可以非常有效地显示错误的存在，但对于显示其不存在却完全无能为力。**”

Dijkstra 是试图通过数学推理来确保程序正确性的先驱。静态类型检查在这方面可以发挥重要作用。

相比之下，如今有很多人持不同观点。以下是一个典型的例子：“既然静态类型检查无法覆盖所有可能性，你将需要自动化测试。一旦你有了自动化测试，静态类型检查就是多余的。”

当我看到这个引用时，我把它展示给了我的同事 Dave Walker，他是这个领域的世界级专家。以下是他写的回应：

> “亲爱的随机 Python 博主，你为什么不把静态类型检查看作是一种补充形式的、完全自动化的测试，来增强你的其他测试技术呢？实际上，我不知道有任何其他测试基础设施能像类型检查器那样自动化、快速和响应迅速。顺便说一句，类型检查是一种特殊类型的测试，它能完美地扩展到任意大小的软件，因为它仅基于模块的接口来检查两个模块的组合是否正确，而无需重新检查它们的实现。传统测试无法以同样的方式扩展。另外，你是否知道类型检查能够保证某些类别的错误**绝对不存在**？如果你希望你的系统是安全的，这一点尤其重要。Python 做不到这一点。”

其核心思想是，我们教授的这种模块化编程方式，结合静态类型检查来构建更大的程序，真正实现了这种方法：我们利用计算机的力量来帮助我们推理程序应该做什么，尽可能详细地指定我们认为它们应该做什么。通过组合模块来获得关于越来越大的程序的“证明”，这在现代系统中是一个极其强大的理念。

我个人认为，对于自动化测试来说，输入的可能性**太多太多**了，以至于它根本无法完全有效，所以我支持 Dave 的观点。

## 历史视角：类型检查的重要性

我将用一个早期编程语言的老故事来结束。

在机器语言初期，对类型的支持不多，这实际上是 C 语言的创新之一。Charles Simonyi 是 1970 年代施乐帕克研究中心的一名程序员，他有一个想法：通过将类型编码在变量名本身中，使程序更容易理解。我们在现代语言中甚至不这样做，但一些古老的语言会这样做。例如在 Fortran 语言中，如果变量名以 I、J 或 K 开头，它就是整数，否则就是浮点值。

但 Charles 认为，他需要编写处理字符串和其他类型数据的程序，所以他决定将类型放在变量名的前几个字符中。当时计算机很小，有各种限制，比如变量名只能有 8 个字符的限制。所以他会省略元音字母并进行截断。因此，Charles 代码中一个典型的变量名会是 `ARR U8 FBN`。

这代表一个**无符号 8 位整数数组**，变量名是 Fibonacci 的缩写。对 Charles 来说，知道变量的类型比知道变量名的含义线索重要得多。他和与他合作的编程团队基于这个系统编写了庞大的程序。当你思考一下，查看一行代码时，仅仅通过阅读代码就能知道变量的类型，这在调试时是非常有用的。你可以在阅读代码时进行类型检查。当然，它存在变量名短且无元音的问题，但 Charles 充分利用了这一点。他实际上使用这样的代码构建了 Microsoft Word 的第一个版本。

我讲这个故事是为了说明一个道理：**类型检查在大型软件系统中一直都非常重要**。这不是可以忽视或将会消失的东西。
