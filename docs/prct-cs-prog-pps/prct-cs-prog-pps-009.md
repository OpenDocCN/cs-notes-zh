# 009：调试技巧 🐛

![](img/79a59c0331baeec7f5da643eb04be797_0.png)

在本节课中，我们将学习程序开发中一个至关重要的环节：调试。我们将通过一个具体的例子，了解如何发现并修复程序中的错误，使程序能够正确、高效地运行。

---

## 概述

调试是程序开发的核心部分。程序员大部分时间都花在寻找和修复错误上。即使对于专家，这也是常态。我们将通过一个分解大整数的程序示例，系统地学习调试过程，涵盖从语法错误到性能问题的各个层面。

## 什么是Bug？

**Bug** 是程序中的一个错误。编写程序时犯错是正常的，每个人都会遇到。调试就是利用“编辑-编译-运行”的循环过程，从程序中消除这些错误的过程。

正如早期程序员莫里斯·威尔克斯所言：“我们一开始编程就惊讶地发现，让程序正确运行并不像我们想象的那么容易。我清楚地记得那一刻，我意识到我未来生活的大部分时间都将用于寻找自己程序中的错误。” 这就是程序员的日常。

## 调试为何具有挑战性？

当程序中包含条件语句和循环时，可能的执行路径和控制流数量会急剧增加。没有循环的程序只有一条执行序列。但如果有 `n` 个条件语句，就可能产生 `2^n` 种执行序列。一旦加入循环，可能发生的情况就变得无限多。

你的程序需要为所有可能的情况工作，而你必须找出问题所在。像我们之前做的“赌徒破产”模拟这样的程序，情况通常更为复杂，包含嵌套的条件语句和循环，使得调试工作极具挑战性。

好消息是，条件语句和循环为我们提供了结构，帮助我们理解程序。我们可以将它们作为单元进行推理，这有助于理解程序的行为。

## 一个调试示例：整数分解

我们将通过一个分解大整数的程序来演示调试过程。例如，整数 `2 * 2 * 7 * 13 * 13 * 397`。给定任意整数，我们想找出它的所有质因数。如果整数是质数，则只输出它本身。

这个问题在密码学中有重要应用。许多世界电子商务所依赖的加密系统，其安全性正是基于分解大整数的难度。

### 初始程序与思路

以下是解决该问题的一个简单程序思路：

我们考虑每个小于待分解数 `n` 的整数 `i`。只要 `i` 能整除 `n`，我们就打印出 `i`（因为它是 `n` 的一个因子），然后将 `n` 替换为 `n / i`。`i` 本身可能仍然是 `n / i` 的因子，因此这是一个 `while` 循环。

这引出了以下代码（注意：这段代码包含错误，我们将调试它）：

```java
long n = Long.parseLong(args[0]);
for (int i = 0; i < n; i++) {
    while (n % i == 0) {
        System.out.print(i + " ");
        n = n / i;
    }
}
System.out.println();
```

### 调试过程

现在，让我们开始调试这个程序。

#### 第一步：消除语法错误

程序要正常运行，首先必须是一个合法的Java程序。编译器会帮助我们找出语法错误。

以下是常见的语法错误及其修复：

*   **错误1：缺少分号。** 编译器提示在 `long` 后需要一个分号。检查后发现，在声明 `long n` 后确实忘记了分号。修复它。
*   **错误2：变量未声明。** 编译器提示在 `for` 循环中找不到符号 `i`。这是因为 `i` 没有声明类型。将其声明为 `int`。

修复这些错误后，程序可以成功编译。这是第一步：消除所有语法错误。编译器是我们的朋友，它能帮助我们找到这些错误。

#### 第二步：消除运行时错误

程序编译通过后，下一步是运行它，看它是否按预期工作。程序是合法的，但它做了什么？我们需要运行来发现。

以下是运行时错误的发现与修复：

*   **错误3：数组越界。** 运行程序时出现 `ArrayIndexOutOfBoundsException`。这意味着我们忘记在命令行提供要分解的数字。修复：确保提供输入参数。
*   **错误4：除以零。** 提供输入（如 `98`）后，程序在第8行抛出除以零的错误。问题在于 `n % i` 的计算，而 `i` 从 `0` 开始。`0` 不可能是任何数的因子，我们应该从 `2` 开始。修复：将 `for` 循环的起始值改为 `2`。
*   **错误5：无限循环。** 修复后，程序为输入 `98` 陷入无限循环，不断打印 `2`。检查发现，`while` 循环缺少大括号 `{}`，导致逻辑错误。修复：添加正确的大括号。

现在，程序对于输入 `98` 可以运行并输出 `2 7 7`。这是一个巨大的进步，但输出格式不完美（缺少换行），并且程序对其他输入可能仍有问题。

#### 第三步：测试与逻辑错误

我们需要测试更多案例，确保程序在各种输入下都能正确工作。

测试发现以下问题：

*   输入 `5`（质数），程序没有输出。
*   输入 `6`，程序输出 `2`，但没有输出另一个因子 `3`。

为了找出问题，一个有效的方法是添加打印语句来跟踪变量值。对于初学者，当你想知道一个变量的值时，就插入一条打印该变量的语句。

在循环中添加打印 `i` 和 `n` 的语句后，观察执行过程。我们发现，在除尽所有因子后，如果剩下的 `n` 大于 `1`（即它是一个质数），我们需要将其打印出来。

修复方法：在循环结束后，检查 `n` 是否大于 `1`，如果是，则打印它，然后换行。

**重要提示**：修复代码后，**必须重新编译**程序，否则运行的仍是旧版本。这是程序员常犯的错误。

#### 第四步：性能优化

现在程序逻辑正确了，但我们需要问：它的效率足够高吗？能解决大规模问题吗？

测试大数分解时，程序变得非常慢。这是一个**性能错误**。性能问题和其他类型的错误一样严重，需要修复。

分析程序发现，我们不需要尝试所有小于 `n` 的 `i`。如果 `i * i > n`，那么我们已经测试了所有可能的较小因子，不可能再有更大的因子了（因为如果存在一个大于 `sqrt(n)` 的因子，我们早就找到了与之配对的较小因子）。

因此，我们可以将循环条件从 `i < n` 优化为 `i <= n / i`。这个小小的改动带来了巨大的性能提升。

优化后的代码片段：
```java
for (int i = 2; i <= n / i; i++) {
    while (n % i == 0) {
        System.out.print(i + " ");
        n = n / i;
    }
}
if (n > 1) System.out.print(n);
System.out.println();
```

经过此优化，程序可以在合理时间内分解18位甚至更大的整数。不过，互联网商务仍然是安全的，因为它依赖于分解200位这样巨大的整数，目前还没有非常快速的通用算法。

## 调试过程总结

综上所述，程序开发实际上是一个四步循环过程，而不仅仅是三步：

1.  **编辑**：编写或修改代码。
2.  **编译**：查找并修复**语法错误**。编译器会帮助你。
3.  **运行**：查找并修复**语义错误**或**运行时错误**。通过测试和打印语句来定位。
4.  **测试与优化**：在真实数据上测试程序。检查是否存在**性能问题**，并寻找优化方法。

只有当你对这四步的结果都感到满意时，才能认为程序已调试完成，可以提交评分或交付使用。
