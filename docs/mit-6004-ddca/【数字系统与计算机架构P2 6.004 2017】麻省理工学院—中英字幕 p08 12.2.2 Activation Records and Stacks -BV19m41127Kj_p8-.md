The problem we need to solve is where to store the values needed by the procedure， its arguments。

So we'll need a block of storage for each active procedure call。

 What we'll call the activation record。 As we saw in the factorial example。

What we need is a way to dynamically allocate storage for an activation record when the procedure is called。

Which can then be reclaimed when their procedure returns。

Let's see how activation records come and go， as execution proceeds。

The first activation record is for the call fact of 3。

 It created the beginning of the procedure and holds among other things。

Early compiler writers recognize that activation records are allocated and delocd in last in first out order。

So they invented the stack， a data structure that implements a push operation to add a record to the top of the stack and to pop operation to remove the top element。

New activation records are pushed onto the stack during procedure calls and popped from the stack when the procedure call returns。

Note that stack operations affect the top， in other words， most recent record on the stack。

C procedures only need to access the top activation record on the stack， Other programming languages。

 for example， Java support accesses to other activation records。

 the stack supports both modes of operations。One final technical note。

 some programming languages support closures， for example， JavaScript， or continuations， for example。

 Python's yield statement。Where the activation records need to be preserved even after the procedure returns。

 in these cases， the simple last in first out behavior of the stack is no longer sufficient and will need another scheme for allocating and delocating activation records。

But that's a topic for another course。

Here's how we'll implement the stack on the beta。We'll dedicate one of the beta Regs R29 to be the stack pointer that will be used to manage stack operations。

When we push a word onto the stack， we'll increment the stack pointer。

 So the stack grows to successively higher addresses as words are pushed onto the stack。

Will adopt the convention that S P points to。 In other words。

 its value is the address of the first unused stack location。

 the location that will be filled by the next push。

 So locations with add addresses lower than the value in S P correspond to words that have been previously allocated Word can be pushed to or popped from the stack at any point in execution。

 but will impose the rule that code sequences that push words onto the stack must pop those words at the end of execution。

So when a code sequence finishes execution， S will have the same value as it had before the sequence started。

This is called the stack discipline and ensures that intervening uses of the stack don't affect later stack references。

We'll allocate a large region of memory to hold the stack located so the stack can grow without overriding other program storage。

Most systems require that you specify a maximum stack size when running a program and will signal an execution error if the program attempts to push too many items onto the stack。

For our beta stack implementation， we'll use existing instructions to implement stack operations。

 so for us， the stack is strictly a set of software conventions。

Other ISAs provide instructions specifically for stack operations。

There are many other sensible stack conventions， so you'll need to read up on the conventions adopted by the particular ISA or programming language youll be using。
