Is there any chance we can not only detect a single bit error。

 but also correct the error to recover the original data。 Sure， here's how。



![](img/e798b2e8bf32dd33c0093ca113686aea_2.png)

By increasing the hamming distance between valid code words to3。

 we guarantee that the sets of code words produced by single bit errors don't overlap。



![](img/e798b2e8bf32dd33c0093ca113686aea_4.png)

![](img/e798b2e8bf32dd33c0093ca113686aea_5.png)

The set of code words produced by corrupting 0，0，0， that's 10，0。

010 or 001 has no code words in common with the set of code words produced by corrupting 111。

 that's 110，101 and 011。

![](img/e798b2e8bf32dd33c0093ca113686aea_7.png)

![](img/e798b2e8bf32dd33c0093ca113686aea_8.png)

Assuming that at most one error occurred， we can deduce the original code word from whatever code word we receive。

 For example， if we receive 0，0，1， we deduce that the original code word was 0，0，0。

 and there has been a single bit error。

![](img/e798b2e8bf32dd33c0093ca113686aea_10.png)

![](img/e798b2e8bf32dd33c0093ca113686aea_11.png)

Again， we can generalize this insight if we want to correct up to E errors。

 the minimum hamming distance between valid code words must be at least2 E+1。



![](img/e798b2e8bf32dd33c0093ca113686aea_13.png)

![](img/e798b2e8bf32dd33c0093ca113686aea_14.png)

For example， to correct single bit errors， we need valid code words with a minimum hamming distance of three。
