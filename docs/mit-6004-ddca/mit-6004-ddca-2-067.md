# 067：6.004 信号量实战示例 🚦

![](img/c8f2ffd57812531a9101f24d242bc6d8_1.png)

在本节课中，我们将学习如何使用信号量来确保程序中不同的先后顺序约束得到满足。

![](img/c8f2ffd57812531a9101f24d242bc6d8_3.png)

在深入探讨如何使用信号量来强制执行先后顺序要求之前，我们先回顾一下我们可用的工具。

![](img/c8f2ffd57812531a9101f24d242bc6d8_5.png)

## 信号量基础回顾

你可以将信号量视为一种数量有限的共享资源。

![](img/c8f2ffd57812531a9101f24d242bc6d8_7.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_8.png)

如果我们将一个信号量 `S` 初始化为 `0`，那么它表示当前资源 `S` 不可用。

![](img/c8f2ffd57812531a9101f24d242bc6d8_10.png)

如果 `S` 等于 `1`，则意味着恰好有一个 `S` 资源可供使用。如果 `S` 等于 `2`，则表示有两个 `S` 资源可用，依此类推。

![](img/c8f2ffd57812531a9101f24d242bc6d8_11.png)

为了使用共享资源，一个进程必须首先获取该资源。这通过在需要该资源的代码前添加一个 `wait(S)` 调用来实现。

![](img/c8f2ffd57812531a9101f24d242bc6d8_13.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_14.png)

只要 `S` 的值等于 `0`，等待此资源的代码就会被阻塞，意味着它无法通过 `wait(S)` 命令。

要通过 `wait(S)` 调用，信号量 `S` 的值必须大于 `0`，表示资源可用。获取资源是通过将信号量的值减 `1` 来实现的。

![](img/c8f2ffd57812531a9101f24d242bc6d8_16.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_18.png)

与 `wait(S)` 命令类似，我们还有一个 `signal(S)` 命令。

![](img/c8f2ffd57812531a9101f24d242bc6d8_20.png)

对信号量 `S` 执行 `signal` 操作表示一个额外的 `S` 资源变得可用。`signal(S)` 命令将 `S` 的值增加 `1`。

![](img/c8f2ffd57812531a9101f24d242bc6d8_22.png)

这样做的结果是，一个正在等待 `S` 的进程现在将能够获取它并继续执行下一行代码。

![](img/c8f2ffd57812531a9101f24d242bc6d8_24.png)

## 应用场景：两个并发进程

![](img/c8f2ffd57812531a9101f24d242bc6d8_26.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_27.png)

现在让我们考虑两个并发运行的进程 `P1` 和 `P2`。

`P1` 有两段代码，其中 `A` 段在 `B` 段之前。类似地，`P2` 也有两段代码，`C` 段在 `D` 段之前。

![](img/c8f2ffd57812531a9101f24d242bc6d8_29.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_30.png)

在每个进程内部，执行是顺序进行的，因此我们保证 `A` 总是在 `B` 之前执行，`C` 总是在 `D` 之前执行。我们还假设没有循环，每个进程只运行一次。

![](img/c8f2ffd57812531a9101f24d242bc6d8_32.png)

我们想考虑如何利用不同的信号量来确保代码中任何必要的先后顺序约束。

![](img/c8f2ffd57812531a9101f24d242bc6d8_34.png)

### 场景一：确保 B 在 C 之前完成

![](img/c8f2ffd57812531a9101f24d242bc6d8_36.png)

假设我们需要满足的约束是：`B` 段代码必须在 `C` 段代码开始执行之前完成。

我们可以使用信号量 `S` 来实现这一点，首先在共享内存中将信号量初始化为 `0`。

![](img/c8f2ffd57812531a9101f24d242bc6d8_38.png)

接下来，为了确保 `C` 段代码不会过早开始运行，我们在 `C` 段代码前添加一个 `wait(S)` 调用。只要 `S` 等于 `0`，进程 `P2` 中的代码就无法运行。

![](img/c8f2ffd57812531a9101f24d242bc6d8_40.png)

另一方面，`P1` 则不受此约束。因此 `A` 段代码可以立即开始运行。由于 `B` 段在 `A` 段之后，它将在 `A` 段之后执行。

一旦 `B` 段完成，进程 `P1` 需要发出信号通知我们的信号量，以表明现在进程 `P2` 可以开始执行了。`signal(S)` 调用将把 `S` 设置为 `1`，这将允许 `P2` 最终越过 `wait(S)` 命令。

![](img/c8f2ffd57812531a9101f24d242bc6d8_42.png)

以下是实现此约束的代码结构：

![](img/c8f2ffd57812531a9101f24d242bc6d8_44.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_45.png)

```c
// 共享信号量
semaphore S = 0;

![](img/c8f2ffd57812531a9101f24d242bc6d8_47.png)

// 进程 P1
A();
B();
signal(S);

// 进程 P2
wait(S);
C();
D();
```

![](img/c8f2ffd57812531a9101f24d242bc6d8_49.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_50.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_51.png)

### 场景二：确保进程互斥（不重叠）

接下来，让我们考虑一个稍微复杂一点的约束：`D` 在 `A` 之前，或者 `B` 在 `C` 之前。

换句话说，`P1` 和 `P2` 不能重叠。一个必须先运行，然后是另一个，但任何一个都可以是首先运行的。

为了实现这一点，我们希望将信号量 `S` 用作一个互斥信号量。互斥信号量或互斥锁确保一次只能有一个完整的代码块运行而不会被中断。

![](img/c8f2ffd57812531a9101f24d242bc6d8_53.png)

这可以通过将信号量 `S` 初始化为 `1`，并在两个进程的开头都放置一个 `wait(S)` 语句来实现。由于 `S` 初始化为 `1`，两个进程中只有一个能够获取 `S` 信号量。无论哪个进程碰巧先获取到它，它就会先运行。

![](img/c8f2ffd57812531a9101f24d242bc6d8_54.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_55.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_56.png)

我们需要添加的最后一段代码是，在两个进程代码的末尾，我们必须执行 `signal(S)`。如果我们不执行 `signal(S)`，那么只有碰巧先获取到 `S` 信号量的进程能够运行，而另一个进程则会卡在等待 `S` 信号量的状态。

![](img/c8f2ffd57812531a9101f24d242bc6d8_57.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_58.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_59.png)

如果在进程结束时我们执行 `signal(S)`，那么 `S` 会递增回 `1`，从而允许下一个进程执行。

![](img/c8f2ffd57812531a9101f24d242bc6d8_61.png)

请注意，因为这段代码没有循环，所以不用担心第一个进程再次获取 `S` 信号量。

![](img/c8f2ffd57812531a9101f24d242bc6d8_63.png)

以下是实现互斥的代码结构：

![](img/c8f2ffd57812531a9101f24d242bc6d8_65.png)

```c
// 共享信号量（互斥锁）
semaphore S = 1;

![](img/c8f2ffd57812531a9101f24d242bc6d8_67.png)

// 进程 P1
wait(S);
A();
B();
signal(S);

![](img/c8f2ffd57812531a9101f24d242bc6d8_68.png)

// 进程 P2
wait(S);
C();
D();
signal(S);
```

![](img/c8f2ffd57812531a9101f24d242bc6d8_70.png)

### 场景三：确保两个进程的第一段代码先于第二段代码运行

![](img/c8f2ffd57812531a9101f24d242bc6d8_72.png)

最后，让我们考虑最后一组约束。

![](img/c8f2ffd57812531a9101f24d242bc6d8_74.png)

在这种情况下，我们希望确保进程 `P1` 和 `P2` 的第一段代码都在进程 `P1` 和 `P2` 的第二段代码之前运行。换句话说，`A` 必须在 `B` 和 `D` 之前，`C` 也必须在 `B` 和 `D` 之前。

![](img/c8f2ffd57812531a9101f24d242bc6d8_75.png)

`A` 必须在 `B` 之前，以及 `C` 必须在 `D` 之前的约束是默认满足的，因为代码总是按顺序执行。这意味着我们的约束简化为 `A` 在 `D` 之前，以及 `C` 在 `B` 之前。

![](img/c8f2ffd57812531a9101f24d242bc6d8_77.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_78.png)

为了实现这一点，我们需要使用两个信号量，比如 `S` 和 `T`，并将它们初始化为 `0`。

![](img/c8f2ffd57812531a9101f24d242bc6d8_80.png)

在一个进程的第一段代码完成后，它应该向另一个进程发出信号，表明如果另一个进程已经完成了其第一段代码，那么它可以开始其第二段代码。

![](img/c8f2ffd57812531a9101f24d242bc6d8_81.png)

为了确保另一个进程已经完成了其第一段代码，我们将 `signal` 调用放在每个进程的两段代码之间。

![](img/c8f2ffd57812531a9101f24d242bc6d8_83.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_84.png)

除了向另一个进程发出可以继续的信号外，每个进程还需要等待另一个进程正在发出信号的信号量。

![](img/c8f2ffd57812531a9101f24d242bc6d8_86.png)

这种 `signal` 和 `wait` 的组合确保了代码的 `A` 和 `C` 段将在 `B` 和 `D` 段之前运行。

![](img/c8f2ffd57812531a9101f24d242bc6d8_88.png)

由于信号量初始化为 `0`，`wait(S)` 在 `P1` 调用 `signal(S)` 之前不会完成，而那时它已经完成了 `A` 段。

![](img/c8f2ffd57812531a9101f24d242bc6d8_90.png)

类似地，`wait(T)` 在 `P2` 调用 `signal(T)` 之前不会完成，而那时它已经完成了 `C` 段。

![](img/c8f2ffd57812531a9101f24d242bc6d8_92.png)

因此，一旦进程能够通过它们的 `wait` 命令，我们就保证两个进程的第一段代码都已经运行了。

我们也没有通过要求 `A` 在 `C` 之前运行或 `C` 在 `A` 之前运行等来强加任何额外的约束。

![](img/c8f2ffd57812531a9101f24d242bc6d8_94.png)

当然，我们可以交换使用 `S` 和 `T` 信号量，最终得到完全相同的行为。但请注意，我们不能交换 `signal` 和 `wait` 命令的顺序。如果我们试图在 `signal` 之前调用 `wait`，那么两个进程都会陷入死锁，等待一个永远不会被发出信号的信号量。

![](img/c8f2ffd57812531a9101f24d242bc6d8_95.png)

这突出了一个事实：在使用信号量时，你必须始终非常小心，不仅要考虑满足所需的要求，还要确保没有可能陷入死锁情况，即一个或多个进程永远无法运行完成。

![](img/c8f2ffd57812531a9101f24d242bc6d8_98.png)

本节课中，我们一起学习了如何使用信号量来协调并发进程的执行顺序。我们回顾了信号量的基本操作 `wait` 和 `signal`，并通过三个具体示例演示了如何应用它们来实现不同的先后顺序约束：确保一段代码在另一段之前执行、实现进程间的互斥访问，以及确保多个进程的第一阶段都在第二阶段之前完成。最重要的是，我们认识到在使用信号量时必须谨慎设计，以避免死锁的发生。
