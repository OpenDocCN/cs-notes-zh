Before diving into the use of semaphos to enforce our precedence requirements。

You can think of a semaphore as a shared resource that is limited in quantity。



![](img/c8f2ffd57812531a9101f24d242bc6d8_3.png)

If we have a semaphore S that is initialized to 0， then it represents the fact that currently resource S is not available。



![](img/c8f2ffd57812531a9101f24d242bc6d8_5.png)

If S equals1， then that means that exactly one S resource is available for use。 If S equals 2。

 then there are two S resources available and so on。



![](img/c8f2ffd57812531a9101f24d242bc6d8_7.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_8.png)

In order to make use of the shared resource， a process must first grab that resource this is achieved by adding a weight S call before the code that requires the resource。



![](img/c8f2ffd57812531a9101f24d242bc6d8_10.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_11.png)

As long as the value of S equals 0， the code that is waiting for this resource is stalled。

 meaning that it can't get past the way S command。

![](img/c8f2ffd57812531a9101f24d242bc6d8_13.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_14.png)

To get past the weight S call， the value of semaphore S must be greater than 0。

 indicating that the resource is available。 Grabing the resource is achieved by decrementing the value of the semaphore by one。



![](img/c8f2ffd57812531a9101f24d242bc6d8_16.png)

Annalogous to the weight S command， we have a signal S command。



![](img/c8f2ffd57812531a9101f24d242bc6d8_18.png)

A signal of semaphore S indicates that one additional S resource has become available。

 The signal S command increments the value of S by one。



![](img/c8f2ffd57812531a9101f24d242bc6d8_20.png)

The result of this is that a process that is waiting on S will now be able to grab it and proceed with the next line of coat。



![](img/c8f2ffd57812531a9101f24d242bc6d8_22.png)

Now let's consider two processes， P1 and P2 that run concurrently。



![](img/c8f2ffd57812531a9101f24d242bc6d8_24.png)

P1 has two sections of code， where section A is followed by Section B。 Similarlyly。

 P2 has two sections which are C followed by D。

![](img/c8f2ffd57812531a9101f24d242bc6d8_26.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_27.png)

Within each process， execution proceeds sequentially。

 so we are guaranteed that A will always proceed。 B， and C will always proceed D。

 Let's also assume that there is no looping and that each process runs exactly once。😊。



![](img/c8f2ffd57812531a9101f24d242bc6d8_29.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_30.png)

We want to consider how we can make use of different semaphoes to ensure any necessary precedent constraints in the code。



![](img/c8f2ffd57812531a9101f24d242bc6d8_32.png)

Suppose that the constraint that we need to satisfy is that Section B code completes before the Section C code begins execution。



![](img/c8f2ffd57812531a9101f24d242bc6d8_34.png)

We can achieve this using semaphore S by first initializing the semaphore to0 in shared memory。😊。



![](img/c8f2ffd57812531a9101f24d242bc6d8_36.png)

Next， in order to ensure that Section C code does not begin running too early。

 we add a weight S call before the Section C code。As long as S equals0。

 the code and process P2 will not get to run。

![](img/c8f2ffd57812531a9101f24d242bc6d8_38.png)

Be one， on the other hand， will not be constrained in this way。

So Section A code can begin running right away。Since Section B follows Section A。

 it will be executed after Section A。

![](img/c8f2ffd57812531a9101f24d242bc6d8_40.png)

Once section B completes process P1 needs to signal our semaphore to indicate that it is now O for process P2 to begin its execution。

 The signal S call will set S equal to 1， which will allow P2 to finally move beyond the weight S command。



![](img/c8f2ffd57812531a9101f24d242bc6d8_42.png)

Next， let's consider a slightly more complicated constraint where Section D precede Section A or Section B precede Section C。



![](img/c8f2ffd57812531a9101f24d242bc6d8_44.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_45.png)

In other words， P1 and P2 cannot overlap。 One has to run followed by the other。

 but either of them can be the one to run first。

![](img/c8f2ffd57812531a9101f24d242bc6d8_47.png)

To achieve this， we want to use our S semapho as a mutual exclusion semaphore。

 A mutual exclusion semaphore or muttex ensures that only one complete block of code can run at a time without getting interrupted。



![](img/c8f2ffd57812531a9101f24d242bc6d8_49.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_50.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_51.png)

This can be achieved by initializing our semaphore S to1 and having a weight S statement at the top of both processes。

 Since S is initialized to1， only one of the two processes will be able to grab the S semaphore。

 Whichever process happens to grab it first， is the one that will run first。

 There is one last piece of code we need to add to complete our requirements。

 which is that at the end of both processes code， we must signal S。 If we do not signal S。

 Then only the process that happen to grab the S semaphore first will get to run while the other is stuck waiting for the S semaphore。

😊。

![](img/c8f2ffd57812531a9101f24d242bc6d8_53.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_54.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_59.png)

If at the end of the process， we signal S， then S getss incremented back to one thuss allowing the next process to execute。



![](img/c8f2ffd57812531a9101f24d242bc6d8_61.png)

Note that because this code does not loop， there is no concern about the first process grabbing the S semaphore again。



![](img/c8f2ffd57812531a9101f24d242bc6d8_63.png)

Finally， let's consider one last set of constraints。



![](img/c8f2ffd57812531a9101f24d242bc6d8_65.png)

In this case， we want to ensure that the first section of both processes P1 and P2 run before the second section of processes P1 and P2。

 In other words， A must proceed B and D and C must proceed B and D。



![](img/c8f2ffd57812531a9101f24d242bc6d8_67.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_68.png)

The constraint that A must proceed B， and C must proceed D is satisfied by default because the code is always executed in order。

This means that our constraint reduces to A preceding D and C preceding B。



![](img/c8f2ffd57812531a9101f24d242bc6d8_70.png)

To achieve this， we need to use two semaphoes， say S and T。And initialize them to 0。



![](img/c8f2ffd57812531a9101f24d242bc6d8_72.png)

After the first section of a process completes， it should signal to the other process that it may begin its second section of code provided that it has already completed its first section of code。



![](img/c8f2ffd57812531a9101f24d242bc6d8_74.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_75.png)

To ensure that it has already completed its first section of code。

 we place the signal calls between the two sections of code in each process。



![](img/c8f2ffd57812531a9101f24d242bc6d8_77.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_78.png)

In addition to signaling the other process that it may proceed。

 each of the processes needs to wait for the semapho that the other process is signaling。



![](img/c8f2ffd57812531a9101f24d242bc6d8_80.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_81.png)

This combination of signal and weight ensures that sections A and C of the code will run before sections B and D。



![](img/c8f2ffd57812531a9101f24d242bc6d8_83.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_84.png)

Since the semaphoes are initialized to0， the weight S will not complete until P1 calls signal S。

 at which point it has already completed Section A。



![](img/c8f2ffd57812531a9101f24d242bc6d8_86.png)

Similarly， the way T will not complete until P2 calls signal t。

 at which point it has already completed Section C。



![](img/c8f2ffd57812531a9101f24d242bc6d8_88.png)

So once the processes can get past their weight commands。

 we are guaranteed that both first sections of code have already run。



![](img/c8f2ffd57812531a9101f24d242bc6d8_90.png)

We have also not forced any additional constraints by requiring A to run before C or C to run before A and so on。



![](img/c8f2ffd57812531a9101f24d242bc6d8_92.png)

Of course， we could have swapped our use of the S&T semaphoes and ended up with exactly the same behavior。

Note， however， that we could not swap the signal and weight commands around。

 If we tried to call weight before signal， then both processes would get deadlocked。

 waiting for a semaphore that never gets signaled。

![](img/c8f2ffd57812531a9101f24d242bc6d8_94.png)

![](img/c8f2ffd57812531a9101f24d242bc6d8_95.png)

This highlights the fact that when using semaphores。

 you must always be very careful to not only worry about satisfying the desired requirements。

 but also ensuring that there is no possibility of ending up in a deadlock situation where one or more processes can never run to completion。
