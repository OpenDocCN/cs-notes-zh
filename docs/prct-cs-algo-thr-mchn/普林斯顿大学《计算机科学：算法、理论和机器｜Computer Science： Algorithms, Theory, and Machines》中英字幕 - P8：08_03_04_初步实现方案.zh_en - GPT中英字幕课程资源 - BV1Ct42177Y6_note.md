# 计算机科学：算法、理论和机器：08：初步实现方案 🧱

在本节课中，我们将要学习栈数据结构的一个非常简单的初步实现方案。这个方案虽然基础，但它清晰地揭示了实现我们之前定义的API时所面临的挑战，特别是关于内存使用效率的问题。

## 概述

我们将从一个极其简单的“稻草人”实现方案开始。这个方案在计算机发展早期经常被使用，因为它足够简单。通过分析它，我们可以理解实现栈API的核心挑战，并为后续更高效的实现打下基础。

## 实现方案详述

上一节我们介绍了栈的抽象概念和API，本节中我们来看看一个具体的、简单的实现。

我们首先对规范进行一些简化。首先，我们只为一种特定类型的元素（例如字符串）实现栈。其次，我们要求客户端在构造函数中提供栈的容量。这样做的理由是，我们可以直接使用一个字符串数组来存储栈中的值。

让我们看看修改后的API。在构造函数中，我们需要指定栈的容量。`push`方法接收一个字符串，`pop`方法返回一个字符串。这与我们最初定义的通用API略有不同，但能让我们立刻开始审视实现代码。核心思想是，使用数组作为底层数据结构，方法的实现就会变得非常简单。

### 数据结构设计

一旦决定使用数组作为底层数据结构，方法的实现就变得极其简单。

我们将按顺序在数组中存放栈中的元素。变量 `n` 将作为数组中下一个空闲位置的索引，同时也代表栈的当前大小。这有点像一种“倒置”的栈表示法，栈顶元素位于数组的靠后位置。

我们的数据类型包含两个实例变量：一个数组和一个表示大小的整数 `n`。

```java
// 实例变量
private String[] a; // 存储元素的数组
private int n;      // 栈的大小（也是下一个空闲位置的索引）
```

构造函数则根据客户端提供的容量创建数组。

```java
// 构造函数
public FixedCapacityStackOfStrings(int capacity) {
    a = new String[capacity];
    n = 0;
}
```

### 方法实现

有了这个数据结构，现在来实现栈的操作。这些方法都非常简单，基本上是一行代码。

以下是各个方法的实现：

*   **`isEmpty`**：检查栈是否为空，只需判断 `n` 是否为0。
    ```java
    public boolean isEmpty() {
        return n == 0;
    }
    ```
*   **`push`**：将新元素放入数组的 `n` 位置，然后递增 `n`。
    ```java
    public void push(String item) {
        a[n++] = item;
    }
    ```
*   **`pop`**：先递减 `n`，然后返回 `n` 位置（现在是栈顶）的元素。
    ```java
    public String pop() {
        return a[--n];
    }
    ```
*   **`size`**：返回当前栈的大小，即 `n`。
    ```java
    public int size() {
        return n;
    }
    ```

所有操作都是常数时间的一行代码。即使在最简单的机器语言中，这也只需要几条指令。最早需要栈来计算算术表达式和处理函数调用的计算机系统，使用的就是类似这样的实现。

### 测试客户端

这是一个测试客户端，其行为类似于课程开始时展示的例子。它会从标准输入读取字符串，如果读到的不是减号，就将其压入栈；如果是减号，则弹出栈顶元素并打印。

```java
public static void main(String[] args) {
    FixedCapacityStackOfStrings stack = new FixedCapacityStackOfStrings(100);
    while (!StdIn.isEmpty()) {
        String item = StdIn.readString();
        if (!item.equals("-"))
            stack.push(item);
        else if (!stack.isEmpty())
            StdOut.print(stack.pop() + " ");
    }
}
```

如果使用这个客户端运行我们最初的例子（“to be or not to - be - - that - - - is”），将会得到预期的结果“to be not that or be”。

在继续深入之前，思考一个有趣的问题是值得的。由于栈是后进先出的，你能否通过安排压入和弹出的顺序，让元素以排序后的顺序弹出？答案是否定的。例如，序列“5”，“6”，“1”就无法通过任何压入/弹出序列让5在6之后、1之前弹出。即使是如此简单的数据结构，也能引发出迷人的数学问题。相比之下，对于队列（先进先出）来说，这个问题就很无趣，因为元素总是按进入的顺序离开。

## 方案评估与局限性

现在，让我们快速评估一下这个“稻草人”实现。

这个实现虽然简单，但它并不是一个真正的、符合我们最初API规范的栈。它是一个固定容量的集合，只有在数据量不超过容量时才表现得像栈。这带来了很多限制：

1.  **它没有实现栈的API**：因为它要求客户端在构造函数中提供容量，而API规范中不应有此参数。
2.  **它不满足性能要求**：虽然所有操作都是常数时间，但它**不满足“内存使用应与集合中元素数量成线性关系”** 的要求。为了应对最坏情况（栈满），我们必须预先分配一个可能很大的数组，但在大多数时候，栈的实际大小远小于容量，这就造成了大量的空间浪费。在一个典型的计算应用中，可能存在大量这样的栈，所有浪费的空间累积起来将非常可观。

下图展示了使用一个容量为20的数组运行示例时的内存状态。你可以清楚地看到，当栈中元素很少时，大部分数组空间是闲置的。

![](img/e18ed87b090eba9a4d0a76251866b162_1.png)

因此，`FixedCapacityStackOfStrings` 不是一个合格的栈实现。它不满足性能规范，特别是空间效率的要求。为了满足这些要求，我们需要一种新的、能够动态调整大小的数据结构。这正是我们接下来要讨论的内容。

## 总结

![](img/e18ed87b090eba9a4d0a76251866b162_1.png)

![](img/e18ed87b090eba9a4d0a76251866b162_2.png)

本节课中我们一起学习了栈的一个初步实现方案。我们使用固定大小的数组作为底层存储，实现了压栈、弹栈等基本操作，所有操作都是常数时间复杂度。然而，这个方案的主要问题在于空间效率低下，需要客户端预先指定容量，且大部分时间内存未被充分利用，违反了栈API关于空间使用的性能要求。这指明了我们下一步的改进方向：需要一个能够动态调整大小的数据结构。
