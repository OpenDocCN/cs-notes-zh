在本节课中，我们将开始讨论数据结构，即在计算机内部组织数据的方式，以使其更高效地完成特定任务。首先，我们将介绍两种非常基础的抽象数据类型：栈和队列。它们看似简单，但实际上处于许多计算过程的核心。与我们的所有工作一样，我们将使用应用程序接口来精确描述这些抽象概念。

## 数据类型与数据结构回顾

在深入之前，我们先简要回顾一下数据类型和数据结构之间的区别。

数据类型是一组值以及对这组值的一组操作。Java内置了一些数据类型，如 `int`、`double` 或 `String`。然而，我们使用的大多数数据类型并非Java内置的，而是我们自己定义的，例如本课程中考虑过的 `Complex`、`Picture` 或 `Char` 等。我们使用API来描述这些数据类型。

数据结构则是另一回事，它涉及计算机内部数据的表示以及数据之间的关系。在本讲座中，你将看到，即使使用一些非常简单的机制，我们也能构建出相当复杂的数据结构。有时数据结构是Java内置的，例如一维或二维数组。但大多数数据结构并非内置，这正是我们今天要开始探讨的内容。今天我们将讨论链表，下次我们将讨论树。

对于每个数据类型，你有一组值（即你的数据），这就面临一个设计挑战：我们应该使用什么数据结构？这主要涉及两个资源考量：第一，需要多少内存；第二，给定该数据结构，数据类型的方法需要多少时间。每次我们定义一个需要处理大量数据的数据类型时，这些考量都将指导我们的设计决策。

## 栈和队列的API

现在，让我们看看栈和队列的API。它们是集合的例子。集合是一种抽象数据类型，其值是一组相同类型的项（我们称之为多重集，因为项可以相等）。实际上，几种基础的集合ADT，特别是栈和队列，其区别仅在于操作规范的细节上。

### 栈 📚

栈的基本操作是向集合添加一个新项（我们称之为添加到顶部），第二个操作是移除并返回最近添加的项。这被称为**后进先出**原则。因此，我们从顶部取走项。这些就是栈的操作。此外，我们还需要测试集合是否为空，以及获取集合的大小。这些是表征下推栈的基本操作。

### 队列 🚶‍♂️🚶‍♀️

队列非常相似，唯一的区别在于移除和返回哪个项的规范。对于队列，我们考虑在末尾添加新项，从开头取走项。这被称为**先进先出**原则。同样，我们需要测试集合是否为空，并返回集合的大小。

因此，栈和队列是非常相似的抽象数据类型，它们仅在“使用LIFO还是FIFO原则来决定移除和返回哪个项”这一细节上有所不同。

这两种数据类型自然出现在无数的应用程序中，我们稍后会研究其中的一些。我们想要强调的一个关键特性是：从程序的角度来看，集合的大小没有预定的限制。栈或队列可以容纳多少项没有限制，这在概念上非常重要，并且允许设计出能够经受时间考验的代码。随着计算机变得越来越大，程序可以处理越来越多的数据，因为规范中没有对集合大小做出规定。我们稍后会回到这一点，这是一个微妙的点，但也是本讲座中我们想要强调的一个非常重要的点。

## 栈和队列的操作示例

让我们看一些栈和队列的操作示例。

### 栈操作示例

我们通常将栈的操作命名为 `push` 和 `pop`。`push` 是向集合添加一个项。`pop` 是移除并返回最近添加的项。

以下是一个操作序列的示例：
```
push 2
push B
push 1
pop -> 返回 1
push B
pop -> 返回 B
push 2
...
```
在这个例子中，新项总是被放在栈的顶部。当我们执行 `pop` 时，总是移除并返回顶部的项。这种操作顺序清晰地展示了后进先出的原则。

### 队列操作示例

对于队列，我们使用 `enqueue` 和 `dequeue` 操作。`enqueue` 是在队列末尾添加项，`dequeue` 是从队列开头移除并返回项。

以下是一个操作序列的示例：
```
enqueue 2
enqueue B
enqueue 1
dequeue -> 返回 2
enqueue B
dequeue -> 返回 B
enqueue 2
...
```
在这个例子中，新项被添加到末尾，而移除操作总是从开头进行。这实现了先进先出的原则。

## 泛型与参数化数据类型

我们希望能够在栈或队列上放置任何类型的数据，这在Java代码中需要更多的努力。Java处理这类旨在适用于任何类型数据的集合数据结构的方式称为**泛型**或**参数化数据类型**。

理解这种代码并不难，如果你查看我们的代码并将其用作模板，你将能够自己编写这类代码。

其思想是，在数据类型的定义中，不使用真实的类型名称，而是使用一个占位符类型名称（放在尖括号内）。然后在客户端代码中，我们用真实的数据类型替换这个占位符。让我们看看这是如何工作的，我们会给出一些例子。现在，我只想提前指出这种代码，以便我们能够指定API。

因此，我们不只是说定义栈是什么，而是说定义“项的栈”是什么。尖括号内的 `Item` 不是一个真实的类型名称，而是一个占位符类型名称。我们想要一个项的栈，并且希望能够用任何类型的数据替换 `Item`。

### 栈的API规范

栈的API规范如下：
```java
public class Stack<Item> {
    // 构造函数：创建一个空的栈
    Stack()
    // 方法：将项压入栈顶
    void push(Item item)
    // 方法：弹出并返回栈顶的项
    Item pop()
    // 方法：检查栈是否为空
    boolean isEmpty()
    // 方法：返回栈中项的数量
    int size()
}
```
构造函数创建一个 `Item` 类型的空栈。`push` 方法接受一个 `Item` 类型的参数，并将其添加到栈中。`pop` 方法不接受参数，只返回最近被推入栈的项，其类型是我们的占位符名称 `Item`。`isEmpty` 返回一个布尔值，表示栈是否为空。`size` 返回一个整数，表示栈中项的数量。

### 队列的API规范

队列的API规范与栈类似，只是方法名称不同：
```java
public class Queue<Item> {
    // 构造函数：创建一个空的队列
    Queue()
    // 方法：将项加入队列末尾
    void enqueue(Item item)
    // 方法：出队并返回队列开头的项
    Item dequeue()
    // 方法：检查队列是否为空
    boolean isEmpty()
    // 方法：返回队列中项的数量
    int size()
}
```
我们将向队列添加项的方法称为 `enqueue`，将移除并返回最近入队项的方法称为 `dequeue`。除此之外，API的定义完全相同，仅在名称上有所不同。同样，我们也有 `isEmpty` 和 `size` 方法。

这些就是我们的API规范。接下来，我们将看一些客户端代码，然后再看实现。

## 性能保证

现在，我们要添加另一个非常重要的点，因为它在许多应用程序中都至关重要，并且我们已经在几个例子中看到过：**性能很重要**。我们想要做的是提供性能保证，这样我们的客户端代码不仅简单、安全、清晰，而且高效。我们不希望在我们的实现中隐藏性能缺陷。在某种程度上，当我们讨论数据结构是什么时，我们希望指定它附带一些性能保证。

这有点超出你在语言库规范中通常看到的内容，但它非常重要，特别是对于像这样的基础数据类型，我们稍后会经常回到这一点。

对于栈和队列，我们希望规范首先要求所有操作都应在**常数时间**内完成。这些是简单的操作，应该只占用几条指令。其次，不应浪费内存。如果集合只有几个项，我们应该只使用少量内存；如果有很多项，我们应该使用大量内存，但应与集合大小成线性关系，即只使用一个常数因子乘以集合中的项数。当然，如果为空，大小为零；但如果有任意数量的项，内存使用量应受常数乘以项数的限制。

我之前提到的另一点是，代码中不应有任何对集合大小的限制。正如我们将看到的，如果你有这样的限制，解决这个问题可能很容易，但设定一个更好的设计目标是：即使在未来集合可能变得更大时，我们的代码仍然可以使用。我们不希望为了适应技术变化而不得不更改我们的代码。

因此，这些是要求。同样，如果客户端代码要具有可扩展性，它就需要能够在摩尔定律面前保持有用。

## 实现标准

![](img/5801290b66040133ff51b74bb4a1751c_2.png)

正如我提到的，在编程语言中，通常认为如果你实现了API（即我们指定要使用的方法），那么你就实现了栈和队列的抽象。但我们对实现提出了更高的标准：我们将说，如果你不满足这些性能规范，那么你就没有实现这些抽象。如果某个操作需要线性时间，或者内存使用量巨大，那么它就不是栈；如果某个操作需要超过常数时间，那么它就不是队列。那些是其他类型的数据结构，但不是栈和队列。

我们的意思是，如果你想使用那个名称，你不仅必须匹配API，还必须满足性能规范，否则你谈论的就是其他类型的数据结构。

## 总结

![](img/5801290b66040133ff51b74bb4a1751c_4.png)

在本节课中，我们一起学习了数据结构的基础，特别是两种核心的抽象数据类型：栈和队列。我们明确了数据类型（值的集合及操作）与数据结构（数据的内部表示及关系）之间的区别。我们详细探讨了栈（后进先出）和队列（先进先出）的API规范，包括使用泛型来支持任意数据类型。我们还强调了性能保证的重要性，要求栈和队列的所有操作必须在常数时间内完成，内存使用与集合大小成线性关系，并且对集合大小没有预设限制。这些严格的规范确保了代码的简洁性、安全性和未来的可扩展性。
