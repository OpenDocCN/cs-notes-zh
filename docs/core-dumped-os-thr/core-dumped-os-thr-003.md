# 003：处理器中的一位如何保护操作系统的完整性 🔒

在本节课中，我们将要学习操作系统如何通过硬件与软件的巧妙配合，来保护自身不被用户程序“篡权”。我们将深入探讨处理器中的一个特殊位——模式位，以及它如何与中断、系统调用等机制协同工作，构建起现代操作系统的安全基石。

## 概述：软件之间的权力博弈 💻

我们首先了解到，操作系统是硬件与用户程序之间的中介。这意味着用户程序不能在没有操作系统干预的情况下直接访问硬件资源。然而，操作系统和用户程序本质上都是软件，都是用编程语言编写的。既然如此，为什么用户程序不能“压倒”操作系统并控制计算机呢？

上一节我们提出了这个核心问题，本节中我们来看看硬件是如何为操作系统提供“特权”支持的。

## 硬件支持：模式位与特权指令 ⚙️

在某些情况下，仅靠软件无法解决问题，需要硬件支持。每个计算机处理器都有一套可以执行的操作集合，称为**指令集**。如果我们想赋予某个软件比另一个软件更多的系统控制权，可以定义一个特殊的指令子集，并非所有进程都能使用。

![](img/81b523838df92d5e07e05f3732927d6c_1.png)

许多架构定义了所谓的**特权指令**。为了强制执行这一点，硬件必须支持不同的操作模式。常见的实现方法是使用一个特殊的1位寄存器，它可以存储1或0。

这个寄存器被称为**模式位**。当它的值设置为1时，处理器运行在**特权模式**下，这意味着它可以执行指令集中的任何指令，包括特权指令。当值设置为0时，处理器运行在**受限模式**下，意味着它不能执行特权指令。

CPU内部使用组合逻辑电路来解码传入的指令。模式位对普通指令几乎没有影响，因为它们可以在任何操作模式下运行。然而，对于特权指令，模式位中保存的值决定了正在运行的进程是否被允许执行它们。用逻辑可以表示为：

```
if (mode_bit == 1) {
    // 允许执行特权指令
    execute_privileged_instruction();
} else {
    // 触发异常或中断
    trigger_interrupt();
}
```

这样，特权指令只有在处理器处于特权模式下才能执行。

## 模式切换的钥匙：中断机制 🔑

![](img/81b523838df92d5e07e05f3732927d6c_3.png)

一个随之而来的问题是：如何或谁能改变模式位的值？这是整个机制中最微妙的部分，因为如果任何用户进程都能简单地切换到特权模式，这个功能就毫无意义。

切换机制通常通过**中断**来实现。让我们暂时放下模式位，先解释什么是中断。

当处理器运行一个程序时，地址寄存器指向要执行的下一条指令的内存地址。中断就像是发送给CPU的一个信号，通知它某个事件已经发生。例如，用户按下按键或移动鼠标，都可以触发中断，让系统对该事件做出反应。

当收到信号时，它会导致处理器暂停当前进程，并跳转到一个特定的内存位置。这个内存位置包含CPU将用来处理中断的可执行代码。

![](img/81b523838df92d5e07e05f3732927d6c_5.png)

这个中断处理例程可以分为四个步骤：
1.  CPU需要代码来保存被中断进程的状态。
2.  执行处理中断的代码，例如调用打印字符串的例程。
3.  中断处理完毕后，CPU执行代码以恢复被中断进程的状态。
4.  最后，CPU执行一条称为**中断返回**的特殊指令，使其能够跳回被中断的进程并恢复其执行。

这就是中断的基本流程：它暂时停止当前进程，并将控制权转移到另一个例程。

![](img/81b523838df92d5e07e05f3732927d6c_7.png)

现在，回到模式位。中断不仅可以用来改变CPU的程序计数器，还可以自动将模式位的值翻转为1。这意味着，驻留在那个内存位置的任何代码都获得了CPU的控制权，并且是在特权模式下获得的。

![](img/81b523838df92d5e07e05f3732927d6c_9.png)

这里有一点必须非常清楚：任何可执行实体（如网络浏览器、文本编辑器等）都可能被加载到这个内存区域。这就是为什么我们必须极其小心：一旦这个实体获得特权模式，它几乎可以在计算机内做任何事情，包括确保没有其他进程获得特权模式。

现代的方法是确保操作系统被加载到那里，这样它就可以使用特权指令来保持对系统的完全控制。支持操作模式的处理器正是为此而设计的。

![](img/81b523838df92d5e07e05f3732927d6c_11.png)

## 内核模式 vs. 用户模式 🛡️

特权模式旨在供操作系统的**内核**使用，因此通常被称为**内核模式**。另一方面，受限模式是为用户程序设计的，防止它们压倒操作系统，因此通常被称为**用户模式**。

![](img/81b523838df92d5e07e05f3732927d6c_13.png)

![](img/81b523838df92d5e07e05f3732927d6c_14.png)

以下是内核模式可以做，但用户模式不能做的一些事情：

![](img/81b523838df92d5e07e05f3732927d6c_16.png)

*   **I/O设备交互**：最明显的例子是与网络卡、显卡、硬盘驱动器以及显示器、键盘等外围设备的交互。直接操作这些设备需要特权指令，意味着CPU只能在内核模式下进行。
*   **内存管理单元控制**：特权指令允许操作一个称为**内存管理单元**的特殊硬件组件。该单元负责限制CPU可以访问的物理内存区域。控制MMU意味着完全控制内存，这就是为什么操作系统可以读取和写入物理内存的任何地方，包括用户程序的地址空间。
*   **重定义中断处理**：如前所述，中断导致CPU跳转到包含处理中断所需例程的特定内存位置。问题是内存并不总是以相同的方式使用，处理中断的代码可能并不总是加载在同一个位置。如果中断导致CPU切换到内核模式并进入用户进程的地址空间，那将是危险的。特权指令允许操作系统定义当中断触发时CPU必须跳转到的确切内存位置。并且，由于内存管理单元只能通过特权指令操作，操作系统使用它来确保用户进程无法访问存储中断处理程序的内存位置。这可以防止任何恶意进程覆盖中断处理程序并试图控制系统。

在内核模式下，我们完全控制CPU的中断流。现在你可能已经注意到，虽然硬件提供了工具，但最终取决于软件（即操作系统）是否正确使用它们来维持对系统的控制。

## 用户程序的出路：系统调用 📞

用户程序在用户模式下仍然被允许使用广泛的指令，可以执行移动和复制数据、执行算术、浮点、位运算、评估条件和运行循环等基本任务。然而，如果程序不能做像读取文件、在屏幕上显示文本或通过网络发送消息这样的基本事情，它们就不会很有用。那么它们是如何做到的呢？

你可能听说过API。为了允许用户程序访问硬件资源，操作系统以库的形式提供一组函数，这些函数被称为**系统调用**，对用户程序至关重要。例如，如果用户程序需要处理文件，它将需要一个系统调用来打开文件，每次读取或写入文件时需要更多的系统调用，工作完成后需要另一个系统调用来关闭文件。

系统调用仍然是函数，这意味着它们必须有最终被编译成机器码的内部实现。但这里事情开始显得有点奇怪。因为这些函数是库的一部分，当我们编写用户程序并使用这些函数时，它们成为我们源代码的一部分。这意味着在编译时它们会成为我们用户程序的一部分，但这似乎说不通，因为我们刚刚确定用户程序在用户模式下运行，因此不能使用操作硬件所需的特权指令。

那么到底发生了什么？如果我们仔细观察，会注意到在这些系统调用函数内部的某个点，有一条触发中断的指令。是的，中断不仅可以由硬件触发，也可以通过软件触发。在这种情况下，`int`指令导致程序触发中断，使程序计数器跳转到操作系统地址空间中的一个例程。记住，当中断发生时，CPU切换到内核模式，允许该例程执行所有操作硬件和完成系统调用所需的特权指令。一旦所需的特权指令被执行，就需要将控制权返回给被中断的进程，并确保模式位被设置回零，以便程序按预期在用户模式下恢复执行。

![](img/81b523838df92d5e07e05f3732927d6c_18.png)

具体实现方式因架构而异。在这个例子中，它由中断返回指令处理，但在其他架构中，可能需要单独的指令来切换操作模式，涉及专用指令或显式覆盖模式位，然后是另一条指令将控制权返回给被中断的进程。

另一种可视化的方式是：函数的一部分驻留在用户程序的地址空间中，在用户模式下运行，而另一部分驻留在操作系统的地址空间中，在内核模式下运行。

## 系统调用的利与弊 ⚖️

总的来说，我们可以说系统调用为用户程序提供了必要的服务。系统调用最重要的优势是它们为开发者提供的**硬件抽象**。许多人不知道直接操作硬件有多么困难和复杂。由于系统调用为我们处理硬件交互，当我们需要访问硬件资源时，作为开发者，我们只需要调用它们。系统调用非常有用，因为它们让我们可以基于代码思考，而不是晶体管。

另一个重要优势是**安全性**，这与硬件抽象密切相关。直接操作硬件容易出错，可能导致漏洞和系统故障。一个从Python或JavaScript等语言转向低级编程的人常问的问题是：我会烧坏我的电脑吗？答案总是不会，因为操作系统为我们管理硬件。即使我们写了一个非常糟糕的C程序，我们也不能以可能损坏计算机的方式直接操作硬件。

![](img/81b523838df92d5e07e05f3732927d6c_20.png)

第三个优势是**可移植性**。如果多个平台遵循某些准则，相同的源代码可以在不同的系统中编译。

![](img/81b523838df92d5e07e05f3732927d6c_22.png)

系统调用也有一些缺点。第一个是**性能开销**，这也与它们提供的硬件抽象有关。系统调用所依赖的中断在性能方面并不廉价，它们需要额外的步骤，如上下文切换，而具有缓存等功能的现代架构可能进一步增加性能成本。

![](img/81b523838df92d5e07e05f3732927d6c_24.png)

![](img/81b523838df92d5e07e05f3732927d6c_26.png)

此外，当程序调用系统调用时，它实际上会中断自己，将控制权传递给操作系统。一旦操作系统获得控制权，不能保证它会立即处理该系统调用。操作系统可能会利用这个机会执行其他任务，如运行CPU调度器或管理系统资源。最终系统调用会被处理，但对于用户程序来说，不能保证这会很快发生。

从某个角度看，系统调用可以被视为陷阱；硬件抽象是诱饵，诱使用户程序将控制权交还给操作系统。

![](img/81b523838df92d5e07e05f3732927d6c_28.png)

另一个缺点是**平台依赖性**，这与我前面提到的可移植性优点相矛盾。虽然许多平台旨在实现兼容性，但存在一些奇怪的变体。

## 从协作式到抢占式：定时器的角色 ⏱️

总结CPU操作模式：机制很简单。用户程序可以使用中断使CPU切换到内核模式，但代价是中断自己并将CPU的控制权交给操作系统。反过来，操作系统可以使用特权指令恢复一个被中断的进程，但强制它继续在用户模式下运行。

![](img/81b523838df92d5e07e05f3732927d6c_30.png)

系统调用的概念非常成功，以至于一些架构通过提供专用指令使其调用更容易，无需手动设置中断。因此，在许多架构中，你可以找到`syscall`指令和`sysret`特权指令，它们也使汇编代码更具可读性。

![](img/81b523838df92d5e07e05f3732927d6c_32.png)

当一个操作系统依赖用户程序的协作来重新获得控制权并执行管理任务时，它被归类为**非抢占式**或**协作式**操作系统。这种类型的通用操作系统已不再使用，因为它们存在一个大问题：想象一个包含无限循环且内部没有任何系统调用的程序。CPU将无限期地卡在执行循环中。如果一个进程（有意或无意地）避免使用系统调用，CPU将永远不会将控制权返回给操作系统，从而有效地饿死所有其他进程。

![](img/81b523838df92d5e07e05f3732927d6c_34.png)

再次，仅靠软件无法解决这个问题；需要硬件支持。解决方案涉及使用一个**定时器**。当定时器到期时，它会触发一个硬件中断。这个定时器内置于CPU中，只能通过特权指令控制。

![](img/81b523838df92d5e07e05f3732927d6c_36.png)

在将CPU分配给用户进程之前，操作系统设置定时器，当进程运行时，定时器开始倒计时。用户进程仍然可以通过系统调用自愿将控制权返回给操作系统。然而，如果一个进程调用系统调用花费的时间太长，定时器确保最终会触发中断，强制进程停止并将控制权交还给操作系统。

![](img/81b523838df92d5e07e05f3732927d6c_38.png)

这种机制允许内核模式完全控制每个进程可以使用多少CPU时间。此外，当CPU支持操作模式时，如果用户程序试图直接操作硬件，硬件必须能够做出反应。如果处理器在用户模式下运行并遇到特权指令，硬件可以被配置为触发中断，立即将控制权传递给操作系统，以便它可以处理用户程序的可疑行为，很可能会终止其执行。

中断是CPU的一个基本功能，因此更准确地说，内核模式授予了对CPU使用方式的完全控制权。当硬件架构支持这些功能并且操作系统利用它们时，我们谈论的是**抢占式**操作系统。

## 内核模式的风险：驱动程序与第三方软件 ⚠️

考虑到所有这些，我们现在应该理解用户程序和操作系统都只是软件。由于用户程序被限制在用户模式下，它们不能直接与硬件交互。相反，它们必须将控制权传递给操作系统，由操作系统代表它们操作硬件。因此，操作系统始终处于中间的想法只是对实际情况的一个非常高级的抽象。

许多人可能想知道，除了操作系统之外，是否有其他软件可以在内核模式下运行。答案是肯定的。例如，当一个新的显卡发布时，操作系统可能不知道如何控制它，因为操作系统是在显卡存在之前开发的。操作系统开发人员不可能为存在的每一个硬件设备编写代码，所以这个任务通常留给硬件制造商。解决方案被称为**驱动程序**，即设计用于控制特定硬件的软件。由于控制硬件需要特权指令，驱动程序必须在内核模式下运行。当用户程序需要硬件时，操作系统使用适当的驱动程序。

![](img/81b523838df92d5e07e05f3732927d6c_40.png)

![](img/81b523838df92d5e07e05f3732927d6c_42.png)

然而，这种技术带来一个严重的缺点：任何在内核模式下运行的代码都有权访问特权指令，这使得操作系统无法将其与其他进程甚至自身完全隔离。因此，所有在内核模式下运行的代码共享一个单一的地址空间。如果驱动程序错误地写入错误的内存地址，它可能会损坏对操作系统至关重要的数据；如果驱动程序崩溃，它将导致整个操作系统崩溃。没有操作系统，用户程序就无法运行，整个系统就会瘫痪。

这就是为什么有错误的驱动程序经常导致Windows蓝屏死机。尽管存在风险，但允许代码在内核模式下运行是必要的，因为驱动程序是必不可少的。同样，操作系统开发人员几乎不可能为每一个硬件设备编写代码。

但第三方代码可以在内核模式下运行的事实，导致其用途超出了设备驱动程序的范围，包括电子游戏反作弊系统和恶意软件检测软件等。让我快速解释一下为什么会发生这种情况：

首先，在内核模式下运行的软件不依赖系统调用，这使得它比在用户模式下运行要快得多。想象一个禁止使用控制器以防止对鼠标和键盘玩家不公平优势的游戏。玩家可能会通过使用第三方程序将控制器映射到键盘输入来作弊。由于游戏在用户模式下运行，它无法判断输入是否真的来自键盘；然而，内核模式下的反作弊软件可以直接监控硬件活动，检测输入是否真的来自控制器。对于恶意软件检测，内核模式允许软件扫描网络卡等设备，监控网络流量，并检查运行进程的地址空间以查找可疑活动，例如监控私人数据并将其发送到不受信任的端点。

现在明白为什么这类软件必须在内核模式下运行了：如果它们在用户模式下运行，就无法完成它们的工作。然而，将所有东西都放在内核模式并不总是一个好主意。以CrowdStrike为例，这是一个安装在全世界数百万台商业设备上的恶意软件检测软件。在9月份，一次更新损坏了一个配置文件，用零填充了它。文件损坏后，软件崩溃，导致操作系统也崩溃。结果是全球性的混乱，数千台设备显示蓝屏死机，造成数十亿美元的商业损失。

另一个例子是Vanguard，Riot Games的反作弊软件，用于《英雄联盟》和《无畏契约》等热门游戏。首次发布时，它导致数百名玩家出现蓝屏死机，但更可怕的部分是它与腾讯的联系，腾讯是一家拥有Riot Games的中国公司。事实证明，不安装Vanguard就无法玩Riot的游戏，这出于反作弊目的可以理解。恼人的是，Vanguard在内核模式下运行，即使你不玩游戏时也是如此，这意味着它可以监控你在电脑上所做的一切。虽然没有证据表明它这样做，但腾讯是一家可能受政府监督的中国公司这一事实，引发了隐私担忧。考虑到Vanguard在全世界数百万台计算机上运行，这变得相当令人不安。

## 总结 📝

本节课中我们一起学习了处理器如何通过一个简单的模式位，配合中断和系统调用机制，构建起操作系统的安全防线。我们了解到：

1.  **模式位**是硬件提供的特权开关，区分了**内核模式**和**用户模式**。
2.  **中断**是切换模式、将控制权交还给操作系统的关键机制。
3.  **系统调用**是用户程序请求操作系统服务的标准接口，它通过触发中断进入内核模式。
4.  硬件**定时器**的引入使操作系统成为**抢占式**的，可以强制收回CPU控制权。
5.  允许**驱动程序**等第三方代码在内核模式下运行带来了强大的功能，但也引入了系统稳定性和安全性的重大风险。
