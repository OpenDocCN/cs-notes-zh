## 进程的定义与组成

![](img/c00802411f0b454bd905b678d79ff89e_11.png)

在上一节中，我们讨论了并发。本节中，我们来看看操作系统中的一个核心概念：进程。

一个进程被非正式地定义为**一个正在执行的程序**。这意味着“程序”和“进程”是两个不同的概念。
*   **程序**是一个被动的实体，例如一个存储在磁盘上的可执行文件。
*   **进程**是程序被加载到内存并开始执行后的活动实体。

当我们运行一个程序时，其可执行文件被加载到内存中。此时，程序就变成了一个进程。

进程的执行可能需要额外的内存来存储用户输入和临时结果，操作系统负责分配这些内存。分配给一个进程的内存被称为该进程的**地址空间**。我们稍后会在视频中回到这个概念。

然而，进程不仅仅是它的地址空间。现代操作系统使用并发，允许多个进程通过交替访问计算机资源来执行。在内部，交替访问CPU是通过将进程放入队列来实现的。我们将在本节结束时理解这是如何工作的。

现在的关键点是，在任何给定时刻，只有一个进程可以使用CPU，而所有其他进程都在等待轮到它们。

## 并发执行带来的挑战

CPU内部有通用寄存器、指令寄存器、地址寄存器（也称为程序计数器）、栈指针和标志位等组件。当一个进程获得CPU访问权时，它会使用这些组件来操作和移动数据。这就是运行程序的本质。

但在多个进程之间交替CPU访问并不像听起来那么简单。如果我们简单地切换进程，获得CPU访问权的进程会发现自己在属于前一个进程的CPU状态下。这会导致两个主要问题：

1.  **安全问题**：当前进程可以访问前一个进程的敏感信息。例如，如果前一个进程正在哈希密码，部分密码可能仍存储在寄存器中，当前进程可以恶意读取这些信息。
2.  **正确性问题**：即使当前进程没有恶意意图，它也需要操作寄存器来执行自己的任务。这样做会改变前一个进程的CPU状态。当前一个进程稍后重新获得CPU访问权时，它被中断时的CPU状态已经丢失，导致执行出错。

为了更清楚地说明，让我们看一个例子。

假设我们想运行两个程序。为了简单起见，我们用汇编代码表示它们。假设两个程序同时启动，操作系统允许每个进程在执行两条指令后切换CPU。

*   **进程A**的指令：
    1.  `LOAD 12, R0` （将值12加载到寄存器0）
    2.  `LOAD 20, R1` （将值20加载到寄存器1）
    3.  `ADD R0, R1` （将R0和R1的值相加）

*   **进程B**的指令：
    1.  `LOAD 100, R0`
    2.  `LOAD 35, R1`
    3.  `ADD R0, R1`

![](img/c00802411f0b454bd905b678d79ff89e_13.png)

**执行流程（无保护）：**
1.  进程A执行指令1和2。此时 `R0=12`, `R1=20`。
2.  操作系统将CPU切换到进程B，**但没有保存进程A的CPU状态**。
3.  进程B执行指令1和2。此时 `R0=100`, `R1=35`。**它覆盖了进程A留在寄存器中的数据**。
4.  操作系统将CPU切换回进程A。假设它恢复了正确的程序计数器地址。
5.  进程A执行指令3 (`ADD R0, R1`)。它期望加12和20，但实际上加的是100和35，得到错误的结果135。
6.  进程A的加法操作还会覆盖`R0`的值，导致后续进程B恢复时也使用错误的数据。

最终，两个进程的结果都出错了。这个问题在成百上千个进程并发执行时会成为噩梦，并且由于进程执行顺序难以预测，错误结果也是非确定性的。

## 解决方案：上下文切换

为了确保安全并避免冲突，在切换CPU访问权时需要额外的步骤。这个解决方案的核心是**上下文切换**。

让我们使用相同的例子，但这次在切换进程时执行正确的步骤：

1.  进程A执行指令1和2。此时 `R0=12`, `R1=20`。
2.  在将CPU分配给进程B**之前**，操作系统运行一个特殊例程来**捕获**进程A的当前CPU状态（寄存器值、程序计数器、标志位等）。这就像拍一张快照，并将其保存到内存中。
3.  然后，操作系统**恢复**进程B之前保存的CPU状态（对于首次运行的进程，寄存器通常初始化为0，程序计数器指向其第一条指令）。
4.  进程B开始执行指令1和2。此时 `R0=100`, `R1=35`。
5.  进程B被中断。操作系统再次捕获并保存进程B的CPU状态。
6.  操作系统检索并恢复进程A之前保存的CPU状态到CPU寄存器中。
7.  进程A恢复执行指令3。此时它发现 `R0=12`, `R1=20`，与它被中断时完全一致，因此能正确计算出32。

通过在每个进程切换时保存和恢复其完整的CPU状态，我们解决了安全和正确性问题。进程无法访问其他进程的数据，并且可以确信自己使用的数据是正确的。

这个捕获一个进程的CPU状态并恢复另一个进程的状态以便其继续执行的动作，就称为**上下文切换**。

## 进程的完整视图与进程控制块（PCB）

![](img/c00802411f0b454bd905b678d79ff89e_25.png)

![](img/c00802411f0b454bd905b678d79ff89e_27.png)

![](img/c00802411f0b454bd905b678d79ff89e_29.png)

现在我们知道，一个进程拥有地址空间、程序计数器、寄存器（统称为进程的CPU状态）。但进程还拥有更多东西。

![](img/c00802411f0b454bd905b678d79ff89e_31.png)

一个进程可能还拥有一个打开文件的列表，以及分配给它的I/O设备。因此，进程不仅仅是代码，它是**一整个与其它进程隔离的执行上下文**。这就是为什么我们之前提到的内核例程被称为“上下文”切换——当我们切换进程时，我们替换的是系统运行的整个上下文。

![](img/c00802411f0b454bd905b678d79ff89e_33.png)

这也从一个高级视角解释了为什么多个进程可以有相同的可执行指令，但在执行时会产生不同的结果。这不仅关乎指令本身，还关乎执行这些指令的**上下文**。用一个词来描述进程最合适：**上下文**。

现在的问题是，如果进程是这样一个包含从低级组件（如寄存器）到高级事物（如文件列表）的完整上下文，我们如何将它们放入队列中进行管理？答案是通过**进程控制块**。

PCB代表**进程控制块**，是操作系统用来跟踪每个进程的特殊数据结构。PCB不是进程本身，而是进程的**表示**。它充当一个仓库，存储启动或恢复进程所需的所有数据，以及一些统计信息。正是这个表示（PCB）被实际放入调度队列中。

一个典型的PCB包含以下信息：
*   **进程ID**：唯一标识符。
*   **进程状态**：如运行、就绪、等待。
*   **CPU状态**：程序计数器、寄存器、标志位等（上下文切换时保存/恢复的数据）。
*   **内存管理信息**：地址空间的界限、页表指针等。
*   **资源信息**：打开的文件列表、分配的I/O设备等。
*   **记账信息**：CPU使用时间、时间限制等。
*   **进程关系**：指向父进程和子进程PCB的指针。

在Linux内核源码中，PCB对应的结构体是 `task_struct`（路径：`/include/linux/sched.h`）。有趣的是，Linux称之为“任务”而非“进程”，因为它用同一个数据结构统一表示线程和进程，将它们都视为调度的基本单位。

## 总结

![](img/c00802411f0b454bd905b678d79ff89e_37.png)

![](img/c00802411f0b454bd905b678d79ff89e_39.png)

本节课中，我们一起学习了：
1.  **进程**是正在执行的程序，是一个包含代码、数据、CPU状态和系统资源的完整**执行上下文**。
2.  简单的进程切换会导致**安全**和**正确性**问题，因为进程会相互干扰对方的CPU状态和数据。
3.  **上下文切换**通过在执行切换前**保存**当前进程的CPU状态，并**恢复**下一个进程的CPU状态，完美解决了上述问题。
4.  操作系统通过**进程控制块**来管理和表示进程。PCB包含了管理和调度一个进程所需的所有信息，并被用于维护进程队列。

![](img/c00802411f0b454bd905b678d79ff89e_41.png)

理解进程和上下文切换是理解操作系统如何实现并发、多任务以及后续CPU调度等复杂机制的基础。请记住，本节所讨论的内容对单处理器系统和多核系统同样有效，唯一的区别是多核系统可以同时处理多个上下文，因为每个核心都有自己的执行流水线。

![](img/c00802411f0b454bd905b678d79ff89e_43.png)
