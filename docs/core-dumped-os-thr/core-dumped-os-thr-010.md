操作系统原理：P10：编程中最诡异的Bug - 竞态条件 🏁

在本节课中，我们将要学习并发编程中一个非常棘手且难以捉摸的问题——竞态条件。我们将探讨它的成因、表现形式以及为何它如此危险。

并发是本频道反复讨论的主题，它与CPU调度一起，是将计算资源分配给多个可执行实体的技术。系统在它们之间快速切换CPU，从而创造出它们都在同时运行的假象。当这些进程独立运行时，一切往往都很顺利。但一旦并发的实体需要交互，事情就会变得棘手。如果处理不当，后果可能是灾难性的。

### 核心概念：非原子操作

要理解竞态条件，首先需要掌握一个关键概念：**大多数操作都不是原子的**。

当我们初学编程时，被告知我们只是在编写指令，为计算机定义一系列步骤以达到目标。但大多数初学者不知道的是，源代码的每一行并不一定对应一条CPU指令。实际上，有些行可能只编译成一条指令，而另一些则需要很多条。

![](img/86c02e7233aa172866edb769ef167f66_3.png)

例如，将两个数字相加就需要多条指令：一条指令将每个操作数从内存取到CPU寄存器，另一条指令告诉CPU将寄存器中的值相加，最后一条指令将结果存回内存，以便后续访问。作为优秀的系统程序员，我们必须牢记，我们在源代码中编写的大多数操作都需要多个CPU步骤才能完成其预期目标。

![](img/86c02e7233aa172866edb769ef167f66_5.png)

![](img/86c02e7233aa172866edb769ef167f66_7.png)

**公式/代码示例：**
```c
// 在C语言中，像 `counter++` 这样的操作通常不是原子的
// 它可能对应多条机器指令：
// 1. 从内存加载 counter 的值到寄存器
// 2. 将寄存器中的值加1
// 3. 将结果存回内存中的 counter
int counter = 0;
counter++; // 非原子操作
```

### 线程与并发回顾

上一节我们介绍了非原子操作的概念，本节中我们来看看并发的基础——线程。

我们之前讨论过，线程通常通过系统调用创建。例如，在C语言中，在符合POSIX标准的系统上，最常见的创建线程方式是使用 `pthread_create` 函数。

**代码示例：**
```c
#include <pthread.h>

void* thread_function(void* arg) {
    // 线程执行的代码
    return NULL;
}

int main() {
    pthread_t thread_id;
    // 创建新线程
    pthread_create(&thread_id, NULL, thread_function, NULL);
    // 主线程继续执行，不等待新线程结束
    // ...
    pthread_join(thread_id, NULL); // 等待线程结束
    return 0;
}
```

这个函数接收一个指向另一个函数的指针。当我们调用 `pthread_create` 时，底层会发生很多事情。内部的系统调用通知操作系统，作为参数传递的函数可以开始作为一个独立的可执行实体运行，与发起调用的线程并发执行。因此，在我们的例子中，系统调用之后，主函数会继续运行，而无需等待新创建的线程完成。

另外请记住，**并发并不等同于并行**。在多核系统中，并发函数可以并行执行，但在单核系统上，CPU核心是共享的，每个线程轮流使用。切换发生得非常快，以至于看起来像是同时在运行。

### 并发的作用与风险

![](img/86c02e7233aa172866edb769ef167f66_13.png)

并发在许多用例中非常有用。例如，现代HTTP服务器严重依赖线程来同时处理多个客户端请求。如果没有并发，主函数必须在接受另一个客户端请求之前，先服务完当前客户的请求，你可以想象这有多低效。有了并发，主线程监听传入的请求，一旦有请求到达，它就生成一个新线程来处理。由于该线程独立运行，主线程可以回去监听新的请求。

![](img/86c02e7233aa172866edb769ef167f66_15.png)

但尽管并发如此有用，使用时也必须谨慎。假设服务器返回请求体中发送的客户端名称。在这种情况下，每个线程只处理自己的客户端，不与其他线程交互。由于没有共享数据，不会出什么大问题。

然而，如果线程需要共享和修改相同的数据呢？想象一下，服务器不是回复客户端名称，而是返回一个所有线程共享的字符串，这个字符串由另一个线程定期从一个句子列表中更新。在这个场景中，线程需要某种方式进行通信。

### 一个竞态条件的实例

几乎每个并发编程的新手在实现这种通信时都会出错。许多人会定义一个全局变量来存储当前字符串，创建一个定期更新它的函数，并生成一个运行此函数的线程。写入线程根据需要简单地修改这个变量，而处理线程在需要响应客户端时从中读取。

这正是我在自己机器上尝试的。我定义了一个共享变量，硬编码了一个句子列表，写入线程循环遍历该列表，更新共享变量。

为了观察行为，我将处理线程发送回客户端的每个响应都记录到一个文本文件中。乍一看，一切正常；写入线程似乎定期更新字符串。

但后来我发现了这个：一个响应返回了一个不在我硬编码列表中的字符串。由于某种原因，最后一个字母不同了，显然出了问题。

仅仅看这一个实例不足以得出结论，它在超过900万次请求中只发生了一次。事实上，通过滚动日志几乎不可能注意到它。我之所以能找到它，是因为我确切地知道要找什么，并编写了一个程序来搜索异常。

![](img/86c02e7233aa172866edb769ef167f66_20.png)

所以我再次运行测试，这次时间更长，并用同一个程序分析结果。在近8500万次请求之后，我发现了更多可疑的输出，这次问题更加明显。为了更清楚，我将为列表中的每个句子分配不同的颜色，这样你就能确切地看到发生了什么。

事实证明，在某些响应中，服务器发送的是不同句子的组合。

那么，这里发生了什么？这是程序行为异常的一种罕见情况，不是因为我们的逻辑有缺陷，而是因为我们忽略了系统底层的工作原理。

### 深入分析：字符串操作的非原子性

为了理解为什么我们的程序行为如此奇怪，我们需要看看字符串是如何工作的。

信不信由你，字符串数据类型值得拥有自己专门的视频，但为了简单起见，我们将专注于C风格的字符串。总之，在C语言中，字符串在内存中表示为一个以特殊空字符结尾的字符序列。

这个空字符很方便，因为它消除了同时存储指针和字符串长度的需要，就像在一些面向对象语言中所做的那样。目前与我们相关的是，**大多数字符串操作都不是原子的**。

例如，当打印一个字符串时，函数会逐个字符遍历，将它们输出到控制台，直到遇到空字符。同样，复制字符串也是一个多步骤的操作。如果字符串足够长，计算机无法简单地在一步中将整个字符数组从一个内存位置复制到另一个位置。相反，它必须循环遍历每个字符，逐个复制，直到到达空终止符。

现在，为了加速这个过程，一些架构会以块为单位复制数据，但即便如此，这些块也有最大大小，并且根据字符串的长度，操作可能仍然需要多个步骤。为了简单起见，我们坚持使用逐个字符的方法。

我选择这个例子的原因是它清楚地展示了底层发生了什么。试想一下，如果一个线程正在从共享字符串读取，而另一个线程正在向它写入，并发可能导致操作系统在写入者复制到一半时中断它，将CPU交给另一个进程。当这种情况发生时，共享字符串就处于无效状态，因为它的当前值是我们从未打算让它成为的值。如果调度器决定在写入线程恢复并完成更新之前，将CPU分配给一个读取此共享数据的线程，那么该线程最终将复制不一致的数据。这正是我们例子中发生的情况。

在修改字符串时，写入线程被中断，处理客户端请求的线程最终在字符串仍不完整时复制了它。这就是为什么所有错误都显示由列表中不同有效句子的片段组成的字符串。

### 竞态条件的定义与成因

**竞态条件**是一种软件错误，当程序的结果取决于多个线程或进程访问和修改共享资源的不可预测顺序时发生。

竞态条件通常是**抢占式操作系统**的副作用；这正是此类错误发生所需的不可预测性所在。对于那些不熟悉的人来说，抢占式操作系统是指会强制性地不时中断正在运行的进程，以便其他进程可以获得CPU访问权限的系统。这是系统在所有活动程序之间公平分配计算资源的方式。

当我们讨论CPU调度时，我们提到了**时间片**，这是一个固定的时间量，定义了进程或线程在被中断之前可以使用CPU多长时间。通过系统调用，进程和线程可以自愿放弃对CPU的控制，以允许其他程序运行。但是，如果一个线程或进程在其时间片到期之前没有完成当前任务，硬件计时器会触发一个中断，抢占该进程，这意味着操作系统会暂停它并将CPU重新分配给其他任务。

因此，如果数据修改所需的时间超过一个时间片，数据将不可避免地进入无效状态，因为至少保证会有一次中断。

然而，这并不意味着较短的操作就是安全的。即使数据修改在一个时间片内完成，竞态条件仍然可能发生。为什么呢？首先，如果操作在一个时间片接近结束时开始，它很可能会被中断。但除了时间问题，CPU还可能被许多与调度器无关的其他事件中断，例如其他进程设置的计时器、用户在UI中按下按钮、传入的网络数据包等。所有这类事件都会触发中断，而中断，无论其来源如何，都可以抢占当前正在运行的线程，即使其时间片尚未到期。不幸的是，我们无法阻止这些事件的发生，至少在不使系统对那些中断本应处理的事件响应性降低的情况下无法做到。

### 尝试解决：控制变量的陷阱

那么，我们能否至少尝试防止它们破坏我们的数据呢？让我们回到我们的例子并尝试一下。

如果我们引入一个控制变量，一个标志，指示共享数据当前是否正在被修改，会怎么样？例如，1表示正在写入，0表示没有写入。有了这个设置，想要从共享字符串读取的线程可以先检查标志，看看写入线程当前是否正在修改它。

现在，虽然这听起来是个好主意，但我们需要谨慎，因为一旦这个控制变量在线程之间共享，它本身就变成了共享数据，因此也容易受到竞态条件的影响。

设置或检查变量的值并不总是原子的。在某些架构上，写入变量可能是原子的，但读取和比较其值几乎从来不是；这通常涉及将变量的值从内存加载到寄存器，将该寄存器的值与某个常量或期望值进行比较，然后根据比较结果采取行动。

想象一下这种情况：一个读取线程检查标志，它将值加载到寄存器中，但就在比较之后，它被中断了。然后写入线程将标志设置为1并开始修改共享字符串，但它在完成之前也被中断了。接着，当读取线程再次被调度时，其状态被恢复，因此它从上次中断的地方继续执行，其寄存器中仍然保存着标志的旧值。由于该线程在写入线程修改标志之前获取了标志的值，现在的比较是基于一个过时的值。结果，该线程错误地认为读取共享字符串是安全的，即使数据处于无效状态。我们的变量并没有真正起到太大作用。

![](img/86c02e7233aa172866edb769ef167f66_26.png)

因此，除非我们真的知道自己在做什么，否则应该避免使用常规变量作为标志来控制对共享数据的访问。我提出这一点是因为有一些经典的基于软件的解决方案，比如这个叫做**彼得森解决方案**的方法，它们依赖于许多微妙的假设，才能在非常特定的情况下正确工作。这些假设包括软件行为、编译器优化甚至硬件架构等细节。

### 写入-写入竞态条件

![](img/86c02e7233aa172866edb769ef167f66_30.png)

![](img/86c02e7233aa172866edb769ef167f66_32.png)

到目前为止，我们看的是一个线程写入共享数据，而其他线程只从中读取的情况。但也有多个线程需要同时写入共享数据的情况。在这些情况下，竞态条件同样危险。

考虑一个接收一个数字并找出0到该数字之间存在多少个质数的程序。使用并行性是为了让过程更快，因此每个线程可以在单独的CPU核心上运行。为此，线程使用一个共享计数器，一个整数，每当线程找到一个质数时，该计数器就会递增。

问题在于，在许多架构中，递增一个值并不是一个原子操作。如果本例中的两个线程几乎同时找到质数，我们期望计数增加到459。但由于它们并行运行，它们可能都获取相同的值，独立地递增它，然后将结果写回，导致计数器只增加1，即使找到了两个质数。这里没有中断参与。竞态条件的发生是因为对共享变量的并行访问。

但这并不意味着没有并行性情况就会更好。这里有一个简单的场景：假设程序已经找到了10个质数。突然，五个线程几乎同时找到了质数。之后的预期计数应该是15。但看看可能会发生什么。

假设第一个线程获取计数并递增它，但在将结果写回内存之前被中断了。然后其他四个线程在没有中断的情况下运行，每个都成功地递增了计数，因此计数器达到14。接着，第一个线程恢复，正好从它中断的地方继续，就在将递增后的值存储回内存的指令之前。但由于该值是在其他四个线程递增计数器之前计算的，它覆盖了更新后的结果，导致计数从14回落到11。就这样，四个正确检测到的质数丢失了。

这种行为感觉很诡异。人们通常认为计算机非常擅长数学，快速、智能、复杂，但事实上，计算机很笨。它们无法识别这些情况，它们盲目地遵循指令，即使这些指令会导致问题。

如果这还不够诡异，看看这个：如果我多次运行同一个程序，几乎每次都会得到不同的结果。有时甚至能得到正确的结果。为什么？因为程序的行为取决于不可控事件的时间和顺序。无法预测这些事件何时会发生，这就是竞态条件如此危险的原因。

![](img/86c02e7233aa172866edb769ef167f66_34.png)

它被称为竞态条件，是因为多个操作竞相访问或修改共享数据，最终结果取决于哪一个先完成。

### 竞态条件的危险性总结

![](img/86c02e7233aa172866edb769ef167f66_36.png)

最后，我想强调一些可能看起来违反直觉的事情。关于竞态条件，最危险的事情是它们**非常罕见**。在我的服务器示例中，8500万次请求中只发生了14起事件，这是一个极小的概率。而这正是它们如此难以复现和调试的原因。

所以，下次你的项目使用并发时，请记住，如果你不小心，你生成的每个线程都可能是在启动一个定时炸弹的倒计时。在最好的情况下，你的程序会崩溃；在最坏的情况下，无法预测它最终会做什么。

幸运的是，计算机提供了工具来帮助我们防止竞态条件，但这是未来一集的主题。
