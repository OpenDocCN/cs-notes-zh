# 计算机安全：001：课程概述与安全基础 🛡️

在本节课中，我们将学习计算机安全课程的整体框架，并探讨安全的基本定义、核心原则以及威胁模型。我们将从宏观视角理解什么是安全，以及如何系统地思考安全问题。

## 什么是安全？🔐

安全的目标是保护有价值的资产免受威胁。在计算机系统中，这些资产可以是数据、计算资源或服务。安全并非一个绝对状态，而是一个持续对抗风险的过程。

上一节我们介绍了课程的整体目标，本节中我们来看看安全的具体定义和核心属性。

安全通常涉及三个核心属性：
*   **机密性**：防止未经授权的信息泄露。
*   **完整性**：防止未经授权的信息篡改。
*   **可用性**：确保授权用户能够访问信息和资源。

一个安全的系统需要在特定威胁模型下，合理地平衡这些属性。

## 了解你的威胁模型 🎯

设计安全措施前，必须明确你要防范谁以及防范什么。这就是威胁模型。威胁模型定义了攻击者的能力、目标和可能利用的漏洞。

以下是构建威胁模型时需要考虑的关键要素：
*   **攻击者**：是谁？（例如，脚本小子、犯罪组织、国家行为体）
*   **资产**：保护什么？（例如，用户数据、源代码、服务器）
*   **攻击面**：可能被攻击的地方有哪些？（例如，网络接口、用户输入、物理访问）
*   **假设**：你的系统依赖哪些安全假设？（例如，操作系统是安全的、密钥未被泄露）

清晰的威胁模型有助于集中资源，实施最有效的防御。

## 安全是经济学 💰

安全决策本质上是经济决策。防御措施需要成本（时间、金钱、性能），而攻击也会带来损失。安全的目标不是消除所有风险（这通常不可能且不经济），而是将风险降低到可接受的水平。

这意味着需要在预防成本、潜在损失和攻击概率之间进行权衡。有时，检测和响应攻击比预防所有攻击更具成本效益。

## 纵深防御 🏰

由于没有单一的防御措施是完美的，应采用多层防御策略，即“纵深防御”。即使一层防御被突破，其他层仍能提供保护。

这个概念可以类比为城堡的防御：外墙、护城河、内墙、卫兵共同构成防御体系。在计算机系统中，这可能意味着同时使用防火墙、入侵检测系统、加密和访问控制。

## 最小权限原则 🔑

最小权限原则要求每个程序或用户只应拥有完成其任务所必需的最小权限。这限制了错误或恶意代码可能造成的损害。

例如，一个负责生成报表的应用程序进程，不应该拥有直接修改数据库架构的权限。用代码描述，这意味着对资源的访问应进行明确的授权检查。
```c
// 不好的做法：进程默认拥有高权限
// 好的做法：在执行敏感操作前检查权限
if (user_has_permission(current_user, DB_WRITE)) {
    execute_query(dangerous_query);
} else {
    log_unauthorized_attempt(current_user);
}
```

## 职责分离 👥

为了减少欺诈或错误的风险，关键任务应拆分给多个个人或系统。这样，完成一项恶意操作需要多人合谋，增加了攻击难度。

例如，在财务系统中，申请付款的人和授权付款的人应该是不同的角色。这确保了没有任何单一个人可以独立完成一笔未经授权的交易。

## 完全仲裁与TOCTOU问题 ⏱️

完全仲裁原则要求每一次对资源的访问都必须经过检查，且不能绕过这个检查。一个常见的相关漏洞是“检查时间到使用时间”攻击。

TOCTOU攻击发生在系统检查某个条件（如文件权限）之后，但在使用该资源之前，条件被改变。攻击者可以利用这个时间差进行攻击。
```c
// 伪代码示例：存在TOCTOU漏洞
if (access(“file”, W_OK) != -1) { // 检查：文件是否可写？
    // 在这里，攻击者可以用一个符号链接将“file”指向/etc/passwd
    fd = open(“file”, O_WRONLY); // 使用：打开文件
    write(fd, user_data); // 实际写入的是被篡改指向的文件
}
```
防御方法是使检查和操作成为一个不可分割的原子操作。

## 不要依赖隐蔽性安全 🙈

“隐蔽性安全”指依靠系统设计细节的保密性（如算法、端口号）来保证安全。这被广泛认为是一种糟糕的安全实践，因为秘密很难长期保持。

根据**柯克霍夫原则**，一个密码系统应该是安全的，即使攻击者知道系统的所有细节，只要密钥是保密的。安全不应依赖于攻击者的无知。

## 默认安全失效 🔒

系统应该默认处于安全状态。即，默认配置应该是限制性的，任何权限的扩大都需要用户明确的授权。

例如，新创建的用户账户默认不应具有管理员权限；防火墙规则默认应该拒绝所有传入连接，然后根据需要显式地允许特定连接。

## 从开始就设计安全 🏗️

安全不应是事后才考虑的问题。在软件或系统开发的初始设计阶段，就必须将安全作为核心需求纳入。试图在现有系统中“嵌入”安全措施通常成本高昂且效果不佳。

这要求开发人员具备安全思维，并在架构设计、代码编写和测试的每个环节都考虑安全影响。

## 总结 📚

本节课我们一起学习了计算机安全的基础。我们首先定义了安全的含义及其核心属性（机密性、完整性、可用性）。然后，我们探讨了构建威胁模型的重要性，它是所有安全决策的起点。我们理解了安全本质上是一种经济权衡。最后，我们学习了一系列关键的安全设计原则，包括纵深防御、最小权限、职责分离、防范TOCTOU、避免隐蔽性安全、默认安全失效以及安全需要从设计之初就开始考虑。这些原则为构建更安全的系统提供了坚实的理论基础。
