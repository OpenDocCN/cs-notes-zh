![](img/169ab3fe9b7e64c8e30405fb98613439_1.png)

## 概述

在本节课中，我们将深入学习多版本并发控制（MVCC）系统中的垃圾收集机制。我们将探讨为什么需要垃圾收集、如何识别可回收的物理版本、不同的垃圾收集设计决策，以及如何在实际系统中进行性能分析以定位瓶颈。

## 为什么需要垃圾收集

在多版本并发控制系统中，每次更新操作都会创建数据元组的新版本。如果不及时清理旧版本，存储空间将无限增长，最终导致系统耗尽资源。因此，垃圾收集的核心任务是识别并回收那些不再对任何活跃事务可见的物理版本。

一个物理版本被认为是**可回收的**，当且仅当系统中没有活跃事务能够看到该版本。这意味着在快照隔离级别下，该版本对任何事务都不可见，或者创建该版本的事务已中止。

## 删除操作的处理

在之前的讨论中，我们介绍了插入和更新操作，现在我们来探讨删除操作的处理。删除操作需要记录一个逻辑元组在某个时间点已被删除，同时确保后续不会在该版本链后添加新版本。

处理删除主要有两种方法：

*   **设置删除标志位**：在元组头或单独的位图字段中设置一个标志，表示该逻辑元组已被删除。事务在读取时需要首先检查此标志。
*   **墓碑元组**：在版本链的末尾（或开头，取决于版本链方向）插入一个特殊的物理版本（墓碑元组），用以表示该元组已被删除。墓碑元组包含时间戳信息，任何早于该时间戳的快照仍能看到旧的版本。

## 垃圾收集的设计决策

以下是设计垃圾收集系统时需要考虑的几个关键维度。

### 版本追踪机制

首先，我们需要决定如何追踪事务创建的版本。

*   **后台清理**：由独立的垃圾收集线程定期扫描表空间，识别可回收的版本。
*   **协作式清理**：事务或查询在执行过程中，如果遇到可回收的旧版本，则立即进行清理。Hekaton 数据库采用了这种方式。
*   **事务级追踪**：每个事务记录自身创建或使其失效的所有版本。提交时，将这些信息传递给垃圾收集器。Peloton 数据库曾使用此方法。
*   **基于纪元的追踪**：将事务分组到不同的时间纪元中。当一个纪元内所有事务都完成后，该纪元内失效的所有版本都可以被回收。

### 垃圾收集的触发频率

其次，我们需要决定垃圾收集的触发频率，这需要在内存回收速度和运行时开销之间取得平衡。

*   **周期性触发**：以固定的时间间隔或当不可回收版本达到一定阈值时触发。
*   **持续/协作式触发**：将垃圾收集逻辑集成到正常的事务处理或查询执行步骤中。例如，Hyper 数据库在事务提交时进行部分清理。这种方式具有自调节特性。

### 回收判断的粒度

最后，我们需要确定判断版本是否可回收的元数据组织粒度。

*   **单版本追踪**：为每个物理版本单独维护其时间戳信息。在协作式清理中，这几乎是免费的，因为信息已存储在元组头中。
*   **分组版本追踪**：将同一事务在同一时间戳失效的一组版本进行分组追踪。这减少了元数据开销，但可能延迟某些版本的回收。
*   **表级回收**：在特殊情况下，如果能够确定没有活跃事务会访问某个表（例如，所有查询都是预定义的存储过程），则可以一次性回收整个表的所有旧版本。SAP HANA 的“时间旅行”存储使用了此方法。

## 判断回收的算法：时间戳 vs. 区间

判断一个版本是否可回收，主要有两种算法。

*   **传统时间戳比较**：计算所有活跃事务的最小开始时间戳（低水位线）。任何结束时间戳小于此低水位线的版本都可以被回收。这种方法简单，但可能过于保守。
*   **区间判断法**：分析活跃事务的时间戳区间与版本生命周期的交集。如果一个版本的生命周期区间（开始时间戳，结束时间戳）与所有活跃事务的可见区间（开始时间戳，无穷大）都没有交集，则该版本可以被回收。这种方法可以更精确、更早地回收版本，是 Hyper 论文中的一个重点。

对于仅追加存储，区间回收只需更新版本链指针。对于 Delta 存储，则可能需要进行版本合并（Consolidation），将多个 Delta 记录合并成一个包含所有最新属性值的记录。

## 内存碎片与压缩

回收内存后，我们面临如何使用这些空闲空间的问题。简单地重用空闲槽可能导致数据在物理存储上失去时间局部性。

*   **时间局部性的重要性**：在 OLTP 负载中，数据被访问或更新的概率与其创建时间相关。新数据更可能被更新，旧数据则趋于稳定。将大致同时创建的数据存储在相邻位置，有利于后续对旧数据块进行压缩（因为它们是只读的），而无需担心更新带来的解压/重压缩开销。
*   ** compaction**：为了减少碎片并恢复时间局部性，系统需要进行**compaction**操作。即，将多个半满的数据块合并，腾出完整的空闲块。compaction 可以通过数据最后更新时间、最后访问时间或数据间的语义关系（如外键）来指导。
*   **截断操作**：`TRUNCATE TABLE` 是一种特殊的删除操作。在支持事务性目录的系统（如 Peloton）中，可以高效地通过删除并重建表来实现截断，从而避免逐行删除的开销。

## 性能分析教程

在优化系统时，我们需要使用性能分析工具来定位瓶颈。我们将介绍如何使用 `perf` 工具。

### Amdahl 定律

在优化前，我们可以使用 Amdahl 定律估算优化效果。如果一个函数占用了总执行时间的 `P` 比例，我们将其加速了 `S` 倍，那么整体加速比 `Speedup` 为：
```
Speedup = 1 / ((1 - P) + (P / S))
```
这有助于我们确定优化工作的优先级。

### 使用 `perf` 进行性能分析

`perf` 是 Linux 下的性能分析工具，可以收集 CPU 性能计数器数据。

![](img/169ab3fe9b7e64c8e30405fb98613439_3.png)

1.  **编译带调试信息的发布版本**：确保二进制文件包含符号表，以便 `perf` 能将采样点映射到源代码行。
    ```bash
    cmake -DCMAKE_BUILD_TYPE=RelWithDebInfo ..
    ```
2.  **运行 `perf record`**：以 root 权限运行，指定采样事件和频率。
    ```bash
    sudo perf record -e cycles -c 2000 --call-graph dwarf -o perf.data ./benchmark
    ```
3.  **分析结果**：使用 `perf report` 或图形化工具（如 `hotspot`）查看结果。
    ```bash
    sudo perf report -i perf.data
    ```
    报告会显示在哪些函数中消耗的 CPU 周期最多，从而帮助定位性能瓶颈。
