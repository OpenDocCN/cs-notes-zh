## 概述

到目前为止，我们已经介绍了创建命令行工具所需的几个要素。之前我们学习了如何使用 `click` 库。现在，我们将开始通过模块来扩展工具的功能。这就是为什么拥有一个包含模块的目录结构非常有用。

## 重构主模块

上一节我们介绍了命令行工具的基本结构，本节中我们来看看如何通过模块化来优化代码结构。

我们当前有一个 `main.py` 文件，其中包含一些可以放到其他地方的代码。之前提到过，我们可以使用 Linting 和其他扩展功能。我们安装了 Python 扩展，如果将鼠标悬停在某些波浪下划线上，会看到一些建议。例如，这里提示“缺少函数或方法字符串”。因此，最好进行一些调整。

如果我在导入部分执行某些操作，它会提示我缺少模块 `dostream`。这些提示你可以选择添加或忽略，如果你确信实际上并不需要它们。

接下来我们要做的是：让这个 `main.py` 文件变得更小，将部分代码移出去。我们将进入包目录，添加一个新文件。如果你有多种不同类型的实用工具和想要添加的内容，你完全可以把它们放在其他地方。但这里我们将放在此处。

我将创建一个名为 `utils.py` 的文件，目前它是空的。现在，我们将对这个文件进行一些“手术”，看看如何让它变得非常小。我将选择所有这些内容，把它们移走。你会看到 Linter 开始报错，但不用担心。

以下是需要移动的步骤：

1.  从 `main.py` 中剪切选定的代码块。
2.  将其粘贴到新的 `utils.py` 文件中。
3.  由于 `json`、`subprocess` 等模块未导入，会出现问题，这很正常。
4.  回到 `utils.py` 文件顶部，粘贴必要的导入语句。
5.  保存文件后，红色的波浪下划线消失了。

现在，这个 `utils.py` 模块已经准备就绪。

## 创建 `__init__.py` 文件

如果我们回到 `main.py`，会看到 `run_lsblk` 函数带有红色下划线，提示未定义。我们需要在目录中添加一个特殊的文件。在 `blkpi` 目录下，我们需要添加一个新文件，名为 `__init__.py`。

这个文件的作用是允许我们将其作为一个标准的 Python 包来使用。我们不需要在里面放任何东西，但它能让我们进行常规的导入操作。我强烈建议你这样做。如果你创建的是另一种称为“命名空间包”的包，可能不一定需要这个文件。但对于命令行工具和常规的 Python 开发，我们将专注于这个用例。

所以，我们保留这个文件。下一步是回到 `main.py`，然后输入：

```python
from blkpi.utils import run_lsblk
```

自动补全功能之所以有效，是因为 VSCode 在后台理解到这是一个标准的 Python 模块，你可以使用导入、模块和函数。现在它运行正常，看起来没问题。

我们在这里看到一个关于“缺少函数或方法字符串”的提示，这对于我们正在尝试做的事情并不重要。如果我们添加一个小文档字符串，比如“CLI 的主入口点”，保存后，Linter 就会满意。现在我认为我们已经准备好了。

## 测试模块化后的工具

现在我们已经实现了模块化，让我们看看 `utils.py`。我们有两个函数，这看起来很不错。现在需要测试一下。

我将打开一个终端，然后看看我们的工具是否还在。输入 `blkpi`，发现它不存在。这是因为我们没有激活虚拟环境。这是你需要始终注意并保持一致的事情。你应该经常检查你期望的东西是否存在。

当我输入 `which blkpi` 时，可以看到它现在显示出来了，很好。清屏后，输入 `blkpi --help`，显示正确。

现在让我们试试是否能够运行 `blkpi lsblk`。我们没有遇到任何问题，之前的更改仍然有效，可以看到它运行得相当好。输入 `blkpi lsblk /dev/sda1` 也能工作。所以现在一切正常。

## 模块化的优势

当我关闭这些窗口后，情况变得非常好。现在代码更加模块化，我们有了一个 `utils.py` 文件，甚至可以进一步提取更多内容。例如，所有这些父设备、子设备的逻辑，我们可以创建一个单独的函数，并继续添加更多辅助函数。

这有助于提高代码的可读性和可测试性。这些函数将对测试更加友好。我们还没有涉及测试，稍后会进行讲解，但这是一个非常扎实的起点。

我们添加了几个东西：`__init__.py` 文件，创建了 `utils.py` 模块，并修改了 `main.py`。现在看看 `main.py`，它看起来非常好，只有大约 15 行长。这非常棒，因为我们只专注于特定功能，这已经非常非常好了。

## 总结

本节课中，我们一起学习了如何通过创建模块和 `__init__.py` 文件来扩展和优化命令行工具的结构。这使得 `main.py` 非常简洁，仅为约15行代码，是一个很好的起点，便于你添加更多模块，并为后续的测试工作奠定了良好的基础。这种结构在测试时也会很有帮助，因为我们可以更好地隔离功能。想象一下，如果所有代码都在主函数里，那么只想测试其中一小部分（比如 `run_lsblk` 函数）就会非常困难。
