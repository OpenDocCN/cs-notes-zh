# Rust编程4-5：4.5.2.6：在Rust中处理错误 🛠️

![](img/6f9cb96e59e62c60eee2bde24d2b82e4_0.png)

## 概述
在本节课中，我们将学习如何在Rust程序中优雅地处理错误。我们将看到如何避免使用`unwrap()`导致的程序崩溃，转而使用模式匹配和日志记录来更稳健地管理错误情况。

我们的工具目前已经具备了登录、命令行标志解析等多种功能，运行良好。然而，尽管我们已经设置了将日志输出到文件的能力，但我们尚未利用日志系统来处理程序运行中出现的错误。这是我们现在需要探索和改进的地方。

## 问题：`unwrap()`导致的程序崩溃

在`lib.rs`文件中，我们目前使用了一些`unwrap()`调用。如果不加注意地使用它们，可能会引发问题。为了演示这一点，我们将故意触发一个错误。

具体来说，我们将通过修改系统配置，使`lsblk`命令返回无效的JSON数据，从而在解析时引发错误。

让我们在终端中运行程序来观察会发生什么：

```bash
cargo run --info vd1
```

程序运行后，虽然日志功能正常，信息也已输出，但程序因为`unwrap()`而进入了恐慌状态。我们看到了类似“`expected ident or `,` at line 1 column 2`”的错误信息。

这是一个典型的JSON解析错误报告，但信息对用户来说并不友好。如果用户看到这个，很难理解问题所在。错误信息指向了`lib.rs`文件的第28行，这正是我们之前提到的可能出问题的`unwrap()`调用。

我们知道错误根源是`lsblk`输出了无效的JSON。问题的具体细节并不重要，关键在于我们遇到了错误，但当前代码没有妥善处理它。

## 解决方案：使用模式匹配和日志处理错误

既然我们已经在`main.rs`中设置好了日志构建器，整个应用程序的各个模块都可以使用日志功能，那么我们就可以利用它来处理这里的错误。

我们需要停止使用`unwrap()`，并转而处理可能发生的错误。以下是修改步骤：

首先，我们将注释掉原来的`unwrap()`调用，然后使用`match`表达式来处理`serde_json::from_str`的`Result`类型。

```rust
// 原来的代码
// let devices: Vec<Device> = serde_json::from_str(&output).unwrap();

// 修改后的代码
let devices: Vec<Device> = match serde_json::from_str(&output) {
    Ok(devs) => devs,
    Err(e) => {
        // 处理错误
    }
};
```

在`Err`分支中，我们有两件事要做：
1.  使用`log` crate提供的`error!`宏记录错误详情。
2.  返回一个值以停止当前函数的继续执行。

以下是完整的错误处理逻辑：

```rust
use log::error; // 需要导入 error 宏

let devices: Vec<Device> = match serde_json::from_str(&output) {
    Ok(devs) => devs,
    Err(e) => {
        error!("解析并序列化 lsblk 输出时出错: {}. 完整输出: {}", e, output);
        return serde_json::json!({}); // 返回一个空的 JSON 对象
    }
};
```

在上面的代码中：
*   `error!`宏用于记录错误日志，它会将信息输出到我们配置的日志目标（如文件）。
*   我们选择返回一个空的JSON对象（`serde_json::json!({})`）作为错误发生时的默认返回值。当然，你也可以根据业务逻辑返回其他值，例如`None`或一个默认的`Vec`。

## 验证改进

现在让我们再次运行程序，看看错误处理是否生效：

```bash
cargo run --info vd1
```

这次，程序没有崩溃。相反，我们在日志中看到了清晰的错误信息：“`error parsing and serializing lsblk output: expected ident or `,` at line 1 column 2. full output: ...`”。

我们甚至将导致问题的原始输出（`output`变量）也记录了下来，这为调试提供了极大的便利。例如，输出中可能包含一个无效的字符“`F`”，这正是导致JSON解析失败的根源。

通过这种方式，我们优雅地处理了错误：记录了详细的上下文信息供开发者排查，并且让程序得以继续运行或安全地退出当前操作，而不是突然崩溃。

## 总结

本节课我们一起学习了如何在Rust中处理错误。

1.  **识别问题**：我们首先指出了不加甄别地使用`unwrap()`方法会导致程序在遇到错误时直接崩溃，用户体验差且不利于调试。
2.  **引入解决方案**：我们利用Rust强大的`match`表达式和`Result`枚举类型，对可能失败的操作（如JSON解析）进行模式匹配。
3.  **优雅处理**：在`Err`分支中，我们使用`log` crate的`error!`宏记录下错误的详细信息（包括错误对象本身和原始数据），这比简单的恐慌信息有用得多。
4.  **控制流程**：我们通过`return`语句在发生错误时提前退出函数，并返回一个合理的默认值（如空JSON对象），保持了程序的健壮性。

![](img/6f9cb96e59e62c60eee2bde24d2b82e4_5.png)

这种错误处理模式是Rust编写可靠软件的核心实践之一。它要求开发者显式地处理所有可能出现的错误情况，从而使程序更稳定、更易于维护。记住，使用`match`处理`Result`，并用好日志，是构建高质量Rust应用的关键步骤。
