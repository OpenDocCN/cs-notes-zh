## 第1节：基础角色与房间渲染 🧍

上一节我们介绍了课程目标，本节中我们来看看如何实现游戏的基础渲染。

首先，我们需要在俯视角下渲染玩家角色和游戏房间。与之前的侧视角游戏（如马里奥）不同，俯视角游戏允许角色在X轴和Y轴上自由移动，而不受重力跳跃的限制。

### 核心概念：实体与状态机

我们使用“实体”（Entity）类来管理游戏中的所有动态对象（如玩家、敌人）。实体包含位置、速度、动画等属性，并可以通过状态机管理其行为（如闲置、行走、挥剑）。

**代码示例：实体状态定义**
```lua
-- 在常量文件中定义玩家动画数据
ENTITY_DEFS = {
    ['player'] = {
        animations = {
            ['walk-left'] = {frames = {1, 2, 3, 4}, interval = 0.15},
            ['walk-right'] = {frames = {5, 6, 7, 8}, interval = 0.15},
            ['walk-down'] = {frames = {9, 10, 11, 12}, interval = 0.15},
            ['walk-up'] = {frames = {13, 14, 15, 16}, interval = 0.15}
        },
        walkSpeed = 100
    }
}
```

### 实现步骤

以下是实现基础渲染的关键步骤：

![](img/8fc31d00de36a5a615a661d1a3cd4b84_30.png)

1.  **初始化玩家实体**：从定义文件中加载动画和属性。
2.  **创建游戏状态**：在“游戏进行状态”（PlayState）中管理当前房间和玩家。
3.  **渲染分离**：使用 `love.graphics.push` 和 `love.graphics.pop` 分离UI（如心形生命值）和游戏世界的渲染坐标系，确保UI始终固定在屏幕左上角。

通过以上步骤，我们得到了一个可以在空房间中移动的玩家角色。

## 第2节：地牢生成与房间管理 🏰

上一节我们实现了基础渲染，本节中我们来看看如何生成和管理一个由多个房间组成的地牢。

![](img/8fc31d00de36a5a615a661d1a3cd4b84_32.png)

![](img/8fc31d00de36a5a615a661d1a3cd4b84_34.png)

地牢是一个二维网格结构，每个网格单元代表一个房间。我们的目标是随机生成一个连通的地牢布局。

### 核心概念：队列与随机生长算法

我们使用队列（Queue）来实现地牢的生成算法。算法从一个起始房间（通常在地图底部中心）开始，然后随机地向其上下左右四个方向的邻居房间“生长”。

**公式：生长概率**
```
生长概率 = 60%
跳过方向概率 = 40%
```
对于当前房间的每个方向，有60%的几率在该方向创建一个新房间（如果该位置为空且在地图边界内），有40%的几率跳过该方向。这创造了有机的、非对称的地牢形状。

### 实现步骤

![](img/8fc31d00de36a5a615a661d1a3cd4b84_36.png)

以下是地牢生成与管理的关键步骤：

![](img/8fc31d00de36a5a615a661d1a3cd4b84_38.png)

1.  **初始化数据结构**：创建一个二维表格 `rooms` 来存储所有房间对象。
2.  **使用队列生成**：
    *   将起始房间坐标加入队列。
    *   当队列不为空且生成的房间数未达到上限时，从队列头部取出一个坐标。
    *   检查该坐标的四个正交方向，根据概率决定是否创建新房间。
    *   如果创建新房间，则将其坐标加入队列尾部，并在 `rooms` 表中实例化该房间。
3.  **生成门廊**：在所有房间生成后，遍历 `rooms` 表，为相邻的房间之间创建开启状态的门廊对象。
4.  **管理当前与下一个房间**：地牢对象需要维护 `currentRoom`（玩家所在房间）和 `nextRoom`（在屏幕过渡时即将进入的房间）的引用。

通过这个算法，我们能够生成一个每次游戏都不同的随机地牢。

## 第3节：攻击框与受击框 ⚔️

![](img/8fc31d00de36a5a615a661d1a3cd4b84_42.png)

![](img/8fc31d00de36a5a615a661d1a3cd4b84_44.png)

上一节我们构建了地牢，本节中我们来看看如何实现战斗系统，特别是区分攻击框和受击框。

![](img/8fc31d00de36a5a615a661d1a3cd4b84_46.png)

在许多游戏中，角色的攻击范围和其身体的可受击范围是不同的。在《塞尔达传说》中，当玩家挥剑时，剑身有一个矩形区域（攻击框）可以伤害敌人，而玩家身体有一个更小的矩形区域（受击框）用于判断是否被敌人击中。

### 核心概念：AABB碰撞检测

![](img/8fc31d00de36a5a615a661d1a3cd4b84_48.png)

![](img/8fc31d00de36a5a615a661d1a3cd4b84_50.png)

我们仍然使用轴对齐边界框（AABB）算法进行碰撞检测。关键是为实体定义多个具有不同语义的矩形。

![](img/8fc31d00de36a5a615a661d1a3cd4b84_52.png)

![](img/8fc31d00de36a5a615a661d1a3cd4b84_54.png)

**公式：AABB碰撞检测**
```
如果 (rectA.x < rectB.x + rectB.width) 且
   (rectA.x + rectA.width > rectB.x) 且
   (rectA.y < rectB.y + rectB.height) 且
   (rectA.y + rectA.height > rectB.y) 则
   发生碰撞
```

### 实现步骤

以下是实现攻击与受击系统的关键步骤：

1.  **添加“挥剑”状态**：为玩家实体创建一个新的状态 `SwingSwordState`。当按下空格键时，切换到该状态。
2.  **定义攻击框**：在 `SwingSwordState` 中，根据玩家面对的方向（上、下、左、右），计算并创建一个矩形作为攻击框。这个矩形的位置和大小需要精心调整以匹配剑的精灵图。
3.  **碰撞检测**：在 `SwingSwordState` 的更新函数中，遍历当前房间内的所有敌人实体。检查玩家的攻击框是否与敌人的受击框（通常就是其本身的边界框）发生碰撞。
4.  **造成伤害**：如果发生碰撞，则减少敌人的生命值。如果敌人生命值降至0或以下，将其从房间中移除。
5.  **玩家受击**：在房间的更新逻辑中，检查玩家是否与任何敌人碰撞。如果碰撞且玩家不处于无敌状态，则减少玩家生命值，并触发一段时间的无敌状态（期间玩家闪烁且不会受到伤害）。

![](img/8fc31d00de36a5a615a661d1a3cd4b84_56.png)

通过区分攻击框和受击框，我们实现了精确且公平的战斗交互。

## 第4节：触发器与游戏对象 🎯

![](img/8fc31d00de36a5a615a661d1a3cd4b84_58.png)

![](img/8fc31d00de36a5a615a661d1a3cd4b84_60.png)

上一节我们实现了战斗，本节中我们来看看如何通过触发器与游戏对象来增加游戏互动性，例如打开门廊的开关。

![](img/8fc31d00de36a5a615a661d1a3cd4b84_62.png)

![](img/8fc31d00de36a5a615a661d1a3cd4b84_64.png)

![](img/8fc31d00de36a5a615a661d1a3cd4b84_66.png)

触发器是一种特殊的游戏对象，当玩家与之互动（如走过）时，会触发特定的事件或改变游戏状态。

![](img/8fc31d00de36a5a615a661d1a3cd4b84_68.png)

### 核心概念：游戏对象与回调函数

我们创建一个通用的 `GameObject` 类。每个对象可以定义一个 `onCollide` 回调函数，该函数在玩家与该对象碰撞时被调用。

### 实现步骤

以下是实现触发器的关键步骤：

![](img/8fc31d00de36a5a615a661d1a3cd4b84_70.png)

1.  **创建开关对象**：在房间的 `generateObjects` 函数中，实例化一个类型为“switch”的 `GameObject`。为其设置一个随机的、位于房间内的位置。
2.  **定义碰撞行为**：在创建开关对象时，覆盖其 `onCollide` 函数。当函数被调用时，执行以下操作：
    *   将开关的状态设置为“已按下”。
    *   遍历房间内的所有门廊对象，将其 `open` 属性设置为 `true`。
    *   播放一个音效。
3.  **碰撞检测**：在房间的更新逻辑中，检查玩家是否与任何游戏对象碰撞。如果碰撞，则调用该对象的 `onCollide` 函数。

![](img/8fc31d00de36a5a615a661d1a3cd4b84_72.png)

通过这种设计，我们可以轻松地创建各种交互式对象，如宝箱、可拾取物品等。

## 第5节：屏幕滚动与房间过渡 🔄

![](img/8fc31d00de36a5a615a661d1a3cd4b84_74.png)

![](img/8fc31d00de36a5a615a661d1a3cd4b84_76.png)

![](img/8fc31d00de36a5a615a661d1a3cd4b84_78.png)

上一节我们增加了互动元素，本节中我们来看看如何实现《塞尔达传说》标志性的屏幕滚动过渡效果。

当玩家走向一个打开的门廊并试图离开房间时，游戏会自动将玩家移向相邻房间，同时摄像机（视角）平滑地滚动到下一个房间的位置。

### 核心概念：事件系统与补间动画

![](img/8fc31d00de36a5a615a661d1a3cd4b84_80.png)

我们使用事件系统来解耦碰撞检测和过渡逻辑。当玩家碰撞到打开的门廊时，触发一个“移动”事件。地牢对象监听这个事件，并启动一个补间动画来移动摄像机和玩家。

![](img/8fc31d00de36a5a615a661d1a3cd4b84_82.png)

![](img/8fc31d00de36a5a615a661d1a3cd4b84_84.png)

### 实现步骤

以下是实现屏幕过渡的关键步骤：

1.  **检测门廊碰撞**：在玩家的行走状态中，检测玩家是否碰撞到墙壁。如果是，则进一步检查碰撞点是否与一个打开的门廊重合。
2.  **派发事件**：如果玩家正在进入一个打开的门廊，则派发一个事件（如 `‘shift-left’`, `‘shift-right’`），并调整玩家的位置使其与门廊中心对齐。
3.  **事件监听与处理**：地牢对象预先监听了所有方向的“移动”事件。当事件触发时，调用 `beginShifting` 函数。
    *   根据移动方向，确定下一个房间的坐标，并加载该房间作为 `nextRoom`。
    *   为 `nextRoom` 设置一个偏移量（如向左移动则X偏移为 `-VIRTUAL_WIDTH`），使其初始位置在屏幕外。
    *   计算玩家在下一个房间中的目标位置（通常是刚进入门廊的对面位置）。
4.  **执行补间动画**：使用 `Timer.tween` 同时动画化以下属性：
    *   `cameraX`, `cameraY`：通过 `love.graphics.translate` 影响整个场景渲染，实现摄像机移动。
    *   玩家的 `x`, `y`：让玩家自动走向目标位置。
5.  **完成过渡**：补间动画结束后，在回调函数中执行“重置”：
    *   将 `currentRoom` 设置为 `nextRoom`。
    *   将 `nextRoom` 设为 `nil`。
    *   将摄像机偏移和玩家位置重置为相对于新房间原点 `(0, 0)` 的坐标。
    *   关闭新房间的所有门廊（等待下次触发开关）。

![](img/8fc31d00de36a5a615a661d1a3cd4b84_86.png)

这个过程创造了一个无缝的房间过渡幻觉，而实际上游戏坐标在每次过渡后都被归一化了。

![](img/8fc31d00de36a5a615a661d1a3cd4b84_88.png)

## 第6节：模板缓冲区与图形遮罩 🎭

上一节我们实现了房间过渡，本节中我们来看看如何使用模板缓冲区来解决玩家在穿过门廊时“ clipping through the wall ”（穿墙）的视觉问题。

![](img/8fc31d00de36a5a615a661d1a3cd4b84_90.png)

模板缓冲区是GPU中的一个特殊缓冲区，用于在渲染过程中控制哪些像素可以被绘制到屏幕上。我们可以用它来创建遮罩。

### 核心概念：模板测试

![](img/8fc31d00de36a5a615a661d1a3cd4b84_92.png)

在渲染管线中，片段着色器计算出像素颜色后，在写入帧缓冲区之前，会进行模板测试。我们可以设置测试条件（如“模板值等于1”），只有通过测试的像素才会被绘制。

### 实现步骤（Love2D 12+）

![](img/8fc31d00de36a5a615a661d1a3cd4b84_94.png)

![](img/8fc31d00de36a5a615a661d1a3cd4b84_95.png)

以下是使用模板缓冲区创建门廊遮罩的关键步骤：

![](img/8fc31d00de36a5a615a661d1a3cd4b84_97.png)

1.  **禁用颜色写入**：在绘制遮罩前，调用 `love.graphics.setColorMask(false, false, false, false)`。这确保接下来的绘制操作只影响模板缓冲区，而不会在屏幕上留下颜色。
2.  **设置模板状态**：调用 `love.graphics.setStencilState`。
    *   `action`: `‘replace’` – 当绘制发生时，如何修改模板缓冲区。
    *   `mode`: `‘always’` – 总是通过模板测试。
    *   `value`: `1` – 将通过测试的像素对应的模板缓冲区位置设置为值 `1`。
3.  **绘制遮罩形状**：在门廊的入口处绘制矩形（这些矩形在调试模式下可以显示为红色半透明框）。由于颜色掩码被禁用，这些矩形不会显示在屏幕上，但会将对应模板缓冲区的值设为 `1`。
4.  **恢复颜色写入**：调用 `love.graphics.setColorMask(true, true, true, true)`。
5.  **设置模板测试**：再次调用 `love.graphics.setStencilState`，将 `mode` 改为 `‘less’`，`value` 保持为 `1`。这意味着只有模板缓冲区值小于 `1` 的像素才允许被绘制。
6.  **渲染玩家**：在此状态下渲染玩家。当玩家精灵的像素尝试绘制到模板值为 `1` 的区域（即门廊入口）时，测试失败，这些像素将被丢弃，从而产生玩家“走在门廊下方”的视觉效果。
7.  **重置模板状态**：玩家渲染完成后，调用 `love.graphics.setStencilState()` 重置模板状态，避免影响后续渲染。

![](img/8fc31d00de36a5a615a661d1a3cd4b84_99.png)

通过模板缓冲区，我们实现了复杂的像素级遮罩效果，极大地提升了游戏的视觉表现力。
