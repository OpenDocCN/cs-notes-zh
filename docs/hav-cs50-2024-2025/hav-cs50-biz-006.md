## 历史背景与协议基础

上一节我们介绍了课程概述，本节中我们来看看互联网通信的演变。在互联网出现之前，我们已有多种电子通信形式，如电报、电话和无线电。这些通信都需要人类在某一端进行实时解读。

![](img/f964ce1313121fe47b7266d0b3f2158c_3.png)

![](img/f964ce1313121fe47b7266d0b3f2158c_5.png)

互联网带来了全新的范式：计算机之间可以直接通信，无需人类实时干预。这种机器对机器的通信依赖于**协议**。协议是一系列约束或约定，用于规范和结构化某种交互。

以下是日常生活中的协议示例：
*   **问候**：不同文化有不同的问候方式，如握手、拥抱、碰拳。
*   **用餐**：正式场合对餐具摆放有特定规则。
*   **寄信**：不同国家对收件人地址、邮政编码等信息的书写位置有特定标准。

![](img/f964ce1313121fe47b7266d0b3f2158c_7.png)

计算机协议则更为精确，因为计算机不擅长处理模糊性。我们将探讨一系列机器对机器的通信协议，它们共同构建了互联网。

## 互联网协议（IP）📦

为了实现机器间的通信，我们首先需要一种方法来唯一地标识和定位网络上的每台计算机。这就是**互联网协议**的核心作用。

IP协议主要解决三个问题：
1.  **寻址**：为每台计算机分配一个唯一的地址。在IPv4中，这个地址由32位二进制数表示，理论上可以标识约43亿台设备。地址格式通常写作四组十进制数，如 `192.168.1.1`。
2.  **路由**：确定数据从源地址到目的地址的传输路径。
3.  **分包**：将大的消息分解成更小的数据包进行传输。这样，如果某个小包传输失败，只需重传该包，而无需重传整个消息。

每个IP数据包都包含一个**包头**，其中包含了版本、源地址、目的地址等信息。由于协议预先定义了这些字段的顺序和长度，接收方计算机可以自动解析。

## 传输控制协议（TCP）🔗

上一节我们介绍了IP协议如何发送数据包，本节中我们来看看TCP协议如何解决IP协议带来的一些问题。单纯使用IP协议可能导致数据包**乱序到达**或**完全丢失**。

TCP协议在IP协议之上增加了一个控制层，主要解决以下问题：
*   **数据包排序**：为每个数据包编号，接收方可以按正确顺序重组消息。
*   **可靠交付**：接收方对收到的每个数据包发送**确认**。如果发送方未收到确认，则会重传该数据包。
*   **错误校验**：使用校验和等机制确保接收到的数据与发送的数据一致。
*   **多路复用**：引入**端口号**的概念。不同的网络服务使用不同的端口（如HTTP用80/443端口，电子邮件用25/587端口），这样一台计算机可以同时处理多个独立的网络会话。

TCP也有自己的包头，其中包含了源端口、目的端口、序列号、确认号和校验和等字段。IP和TCP包头会包含在您发送的每一个数据流中。

![](img/f964ce1313121fe47b7266d0b3f2158c_18.png)

IP和TCP这两个协议共同构成了最初的ARPANET，并最终发展成为我们今天所知的互联网。

![](img/f964ce1313121fe47b7266d0b3f2158c_20.png)

## 路由与域名系统（DNS）🗺️

随着互联网规模扩大，新的问题需要解决。首先是如何将数据包送达远距离的目的地，即**路由**问题。

计算机本身并不需要知道完整的路径。这项工作由**路由器**完成。路由器检查数据包的目的地址，判断是否属于自己负责的网络。如果不是，则将其转发给**网关**（另一个路由器）。这个过程不断重复，数据包经过一系列路由器“接力”，最终到达负责目的网络的路由器，并送达目标设备。这个过程类似于邮政系统处理信件。

当我们可以与全球计算机通信时，另一个问题出现了：我们如何记住那些复杂的IP地址？这就需要**域名系统**。

DNS就像一个全球电话簿，它将人类可读的域名（如 `harvard.edu`）翻译成机器可读的IP地址。一个名为ICANN的国际组织负责管理顶级域名（如 `.com`, `.edu`）。当您的计算机需要访问一个网站时，它会向DNS服务器查询该域名对应的IP地址。如果本地DNS服务器不知道，它会向上级或权威DNS服务器查询，并将结果缓存一段时间以提高效率。

## 动态主机配置协议（DHCP）与用户数据报协议（UDP）⚡

随着网络设备增多，手动为每台设备配置IP地址变得非常繁琐。**动态主机配置协议**解决了这个问题。

当设备加入网络时，它会向DHCP服务器“请求”一个可用的IP地址。服务器从地址池中分配一个地址给该设备，并约定一个“租期”。租期结束后，设备可以续租或释放该地址，地址则回到池中供其他设备使用。这实现了IP地址的自动管理和高效复用。

并非所有通信都需要TCP那样的高可靠性。对于视频会议、在线流媒体等实时应用，偶尔丢失一两个数据包是可以接受的，但延迟和卡顿则很糟糕。

**用户数据报协议**就是为这种场景设计的。与TCP不同，UDP不保证数据包顺序、不进行确认重传，因此开销更小、速度更快。它适合那些“宁愿丢包，也不要延迟”的应用。

## IPv4地址耗尽与IPv6 🔄

![](img/f964ce1313121fe47b7266d0b3f2158c_24.png)

IPv4的43亿个地址已无法满足全球爆炸式增长的联网设备需求。我们采用了两种主要方案应对。

短期方案是**网络地址转换**。它划定了一些私有IP地址范围（如 `192.168.x.x`），这些地址不能在公网直接路由。家庭或企业内部的设备使用私有IP，当它们访问外网时，路由器会将其私有IP转换为一个公网IP。这样，多个设备可以共享一个公网IP，缓解了地址压力。

![](img/f964ce1313121fe47b7266d0b3f2158c_26.png)

长期根本解决方案是 **IPv6**。它将地址空间从32位扩展到128位，其地址数量之多，几乎不可能耗尽。IPv6地址通常以十六进制表示，如 `2001:0db8:85a3::8a2e:0370:7334`。尽管推广缓慢，但IPv6是未来的必然趋势。

![](img/f964ce1313121fe47b7266d0b3f2158c_28.png)

## 超文本传输协议（HTTP）🌍

我们讨论了底层通信协议，现在来看看我们日常使用最多的应用层协议——**HTTP**，它是万维网的基础。

当您在浏览器中输入 `https://nytimes.com` 并按下回车时，背后发生了一系列事情：
1.  **浏览器起草请求**：浏览器创建一个HTTP请求报文。一个简单的GET请求可能如下所示：
    ```http
    GET / HTTP/2
    Host: nytimes.com
    ...（其他头部信息）
    ```
2.  **使用底层协议发送**：这个请求会通过之前介绍的所有协议（DNS查询IP、TCP分包、路由寻址）发送到《纽约时报》的服务器。
3.  **服务器处理并响应**：服务器上的Web服务器软件接收到请求，找到对应的网页文件，然后生成一个HTTP响应报文发回。响应报文包含状态码、内容类型和实际数据：
    ```http
    HTTP/2 200 OK
    Date: Wed, 21 Oct 2026 07:28:00 GMT
    Content-Type: text/html
    ...（其他头部信息）

    <!DOCTYPE html><html>...（网页HTML内容）...</html>
    ```
4.  **浏览器渲染**：浏览器收到响应后，解析HTML，并根据其中的链接再次发起请求来加载图片、样式表、脚本等资源，最终渲染出完整的页面。

![](img/f964ce1313121fe47b7266d0b3f2158c_30.png)

![](img/f964ce1313121fe47b7266d0b3f2158c_31.png)

HTTP状态码表示请求结果：
*   `200 OK`：成功。
*   `301 Moved Permanently`：永久重定向。
*   `404 Not Found`：未找到资源。
*   `500 Internal Server Error`：服务器内部错误。

![](img/f964ce1313121fe47b7266d0b3f2158c_33.png)

为了提升效率，HTTP协议本身也在演进。HTTP/1.1需要为每个资源建立独立连接，而HTTP/2引入了**多路复用**等技术，允许在单个连接上同时传输多个资源，大大加快了网页加载速度。

一个URL的组成部分清晰地指明了资源的位置：
`https://www.example.com/products/hammer.html`
*   **方案**：`https`（协议）。
*   **子域名**：`www`。
*   **域名**：`example.com`。
*   **顶级域名**：`.com`。
*   **路径**：`/products/hammer.html`（服务器上的文件路径）。

![](img/f964ce1313121fe47b7266d0b3f2158c_35.png)

## 扩展性与云计算 ☁️

![](img/f964ce1313121fe47b7266d0b3f2158c_37.png)

有了可靠的通信基础，如何支撑不断增长的业务规模？这涉及到**扩展性**问题。

![](img/f964ce1313121fe47b7266d0b3f2158c_39.png)

假设一家电力公司平时其停电地图网站日均访问量只有100次，但暴风雨后可能有数百万人同时访问。如何应对这种突发流量？
1.  **垂直扩展**：升级单台服务器的硬件（更快的CPU、更大的内存）。但成本高昂且存在物理极限。
2.  **水平扩展**：增加更多服务器。但管理、维护和机房空间成本随之增加。
3.  **托管**：租用数据中心的空间和硬件，将运维工作外包。

更优的解决方案是**云计算**。它利用**虚拟化**和**容器化**技术，在物理服务器上模拟出多台虚拟计算机或独立的应用环境。

*   **虚拟化**：通过**虚拟机管理程序**在物理机上运行多个完整的“客户”操作系统实例。
*   **容器化**：在主机操作系统上运行容器引擎，为应用程序创建隔离的“沙箱”环境，共享主机系统内核，更加轻量。

![](img/f964ce1313121fe47b7266d0b3f2158c_41.png)

云计算的核心优势是**弹性**。在流量低谷时，可以关闭不必要的虚拟资源以节省成本；在流量高峰时，可以迅速启动大量虚拟实例来分担负载，并通过**负载均衡器**将请求分发到这些实例上。这实现了**高可用性**，通过消除单点故障和动态调整资源，确保服务尽可能持续可用。

![](img/f964ce1313121fe47b7266d0b3f2158c_43.png)

![](img/f964ce1313121fe47b7266d0b3f2158c_44.png)

![](img/f964ce1313121fe47b7266d0b3f2158c_45.png)

从几所大学实验室间的数据包传输，到如今支撑全球业务的弹性云网络，互联网的实现是一系列协议和技术不断迭代、解决问题的成果。
