上一节我们介绍了Google早期分布式系统的概念，本节中我们来看看从Google和Amazon的具体实践中，我们能总结出哪些关键的设计原则。

## 从Google视频中得出的结论

观看完三个关于Google的视频后，我们可以得出以下几点核心结论。

以下是主要收获：

1.  **感知延迟与分布式计算**：对于Google搜索这类服务，用户对0.1秒和0.001秒的延迟感知差异不大。关键在于系统能在可接受的延迟内（如0.25秒）返回看似智能的结果。这体现了分布式计算中“分散-收集”模式的价值：虽然这种方式成本较高，但如果是解决问题的唯一方法，并且问题能容忍一定的延迟，那么它就是有效的。

2.  **资源利用与并行处理**：即使一个查询整体耗时0.25秒，也并不意味着所有资源在这段时间内都被完全占用。系统可以将查询分散到数千个服务器上，每个服务器只执行极其微小的操作。这意味着可以同时有数千个事务在“分散-收集”的流程中并行处理，极大地提高了资源利用率。

3.  **可扩展的架构**：系统的美妙之处在于其可扩展性。随着网络规模或搜索量的增长，只需增加服务器数量即可（例如从200台增加到300台或600台），而整体响应时间可以保持稳定。在这种架构中，很可能存在许多小型、独立的关系型数据库，用于处理爬虫、索引或搜索等特定环节。

4.  **硬件设计哲学：简单与耐用**：数据中心的硬件设计追求极简，避免不必要的包装和装饰（如彩色塑料或炫酷的灯光），核心是节能和耐用。每个计算节点都是独立的、坚固的“小盒子”，甚至自带微型电池以应对短时断电。硬件被视为可消耗品，并且会考虑其生命周期，将旧计算机用于不同的任务。

5.  **通过复制实现数据可靠性**：以Gmail为例，Google通过数据复制而非传统备份来保障可靠性。你的邮件数据会被复制到7到10个不同的地方，分布在同一个机架、不同容器、不同数据中心乃至全球不同地理位置。即使整个数据中心失效，用户也不会丢失邮件。这种复制策略是最终一致性的核心体现。

![](img/1eff4f382c08f7a783b2a9acb35c591e_3.png)

6.  **磁盘的智能使用**：为了平衡性能与成本，Google会智能地使用磁盘。例如，将磁盘前10%的高速区域用于需要高性能的关键操作，而剩余部分则用于存储备份副本。这种物尽其用的方式非常高效。

7.  **最终一致性的本质**：最终一致性是整个系统的基石。当你更新或删除一封邮件时，所有副本不会立即同步，但会在几秒钟后达成一致。这种设计在保证可用性的同时，接受了短暂的数据不一致状态。

8.  **MapReduce的优雅**：MapReduce框架展示了如何将复杂计算（如为数十亿网页计算PageRank）分解为可分布式执行的任务。它被设计成一种分布式算法，能够在夜间等空闲时间利用所有CPU资源重新计算排名，并优雅地处理新数据加入带来的扰动。

总而言之，Google的实践代表了一种从依赖单一大型关系数据库，转向使用大量小型、分布式组件的技术过渡。

## 亚马逊的贡献与AWS的诞生

与此同时，推动NoSQL、分布式系统和最终一致性发展的另一个关键因素是亚马逊。

以下是亚马逊带来的关键变革：

1.  **从书店到技术巨头**：亚马逊最初是一家图书公司，但为了支持其业务，它不得不构建自己的服务器集群。他们开发了如DynamoDB这样的分布式数据库，以确保订单系统能够快速、大规模地运行。

2.  **改变世界的亚马逊云服务（AWS）**：亚马逊最大的贡献是推出了AWS。当时的背景是，大家都想使用廉价的硬件，而亚马逊希望在廉价硬件之上增加虚拟化层。问题在于，如何经济地提供虚拟化服务？如果购买昂贵的高端服务器进行虚拟化，资本成本会很高。

3.  **早期云硬件的成本结构**：在云服务早期，硬件成本结构决定了应用设计：
    *   **CPU和磁盘容量很便宜**。
    *   **磁盘I/O速度（读写速度）和内存非常昂贵**。
    *   **数据传输（网络带宽）也很昂贵**。
    因此，云服务商能提供的是：大量慢速磁盘、不多的内存，以及看似“免费”的CPU和磁盘容量。

4.  **应用架构的重新思考**：这对应用开发者意味着，他们要么继续购买昂贵且快速过时的高端服务器，要么以极低的成本（如每月10美元）租用云服务器。选择后者，就必须彻底重新设计应用程序，使其适应“内存少、磁盘慢但数量多”的环境。如果能做到这一点，成本优势将非常巨大。

## “地毯集群”与新的应用范式

![](img/1eff4f382c08f7a783b2a9acb35c591e_7.png)

这种由大量廉价、标准化的计算节点通过网络连接而成的系统，可以形象地称为“地毯集群”。

以下是利用“地毯集群”的关键方法：

*   **分散数据**：将数据分布到多个节点上（分片）。
*   **复制数据**：为了提高可靠性和读取性能，将数据复制多份。
*   **广播查询**：将查询请求发送到多个节点并行执行。
*   **网络聚合结果**：通过网络将各个节点的结果收集并汇总。

这个过程可能需要一些时间，但由于数据移动和计算可以高度并行化，因此可以同时处理大量请求。像Teradata这样的数据仓库，其本质可能就是将一个大型查询分解，发送到许多内置了SQLite等轻量级数据库的小型集群中执行，然后在外部协调层处理多读多写的复杂性，使其看起来像一个统一的数据库。

本节课中我们一起学习了第一代云计算应用的核心设计理念。我们看到了Google如何通过“分散-收集”、最终一致性和智能硬件管理来构建超大规模服务；也了解了亚马逊AWS如何基于特定的硬件成本结构，催生了需要全新架构设计的云原生应用模式。这些实践共同定义了利用廉价硬件和分布式系统构建可靠、可扩展应用的基础。
