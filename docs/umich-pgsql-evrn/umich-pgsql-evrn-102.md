# 102：SQL与NoSQL技术讲座 🗄️

在本节课中，我们将要学习数据库架构的核心概念，特别是SQL（关系型）与NoSQL（非关系型）数据库之间的根本区别。我们将通过**ACID**和**BASE**这两个关键模型来理解它们的设计哲学、适用场景以及权衡取舍。

## 概述

数据库设计中的一个基本原则是避免数据冗余。然而，随着应用规模和复杂度的增长，有时我们不得不为了性能或可扩展性而做出妥协。本节将探讨这种演变背后的技术动因，并解释为何会出现NoSQL运动。

## 从“不冗余”到“有时冗余”

上一节我们介绍了避免数据冗余的基本原则。本节中我们来看看为何这一原则在实践中存在例外。

在关系型数据库（如MySQL、SQLite或PostgreSQL）教学中，我们常强调“不要复制数据”。这是一个简化但有效的入门规则，因为数据规范化本身是一个复杂的概念。在绝大多数情况下（例如95%），随意复制数据是错误的，通常源于对数据规范化的理解不足或设计上的惰性。

然而，当应用规模变得极其庞大时，有时你**被迫**为了扩展性而进行数据复制。这引出了数据库架构的演进，并部分体现在NoSQL运动中。

![](img/9139342e6dd2a9d42ddcc660ed94c289_1.png)

## SQL vs. NoSQL：问题的本质

关于SQL与NoSQL孰优孰劣的争论，其核心可能问错了问题。SQL本身只是一种用于交互的语法。一个更好的问题是：数据模型应该是**关系型**的还是**非关系型**的？

*   **关系型模型**：数据以行和列的形式组织在表中，核心在于行与列之间定义的**关系**。这不仅仅是电子表格，而是通过外键等机制建立的联系。
*   **非关系型模型**：数据通常以文档形式存储，文档内包含一系列**键值对**。它不是一个严格的行列结构，而更像一个包含数据的“块”。

如今，这种划分方式也已显得过于简单。更健康、更本质的区分方式是考察数据库提供的保证：即**ACID**属性与**BASE**理念。

## 核心模型：ACID 与 BASE

上一节我们提到了更本质的区分方式。本节中我们来看看ACID和BASE这两个核心模型的具体含义。

### ACID：强一致性模型

ACID是一个历史更久远的概念，它代表**原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）、持久性（Durability）**。

其核心思想可以概括为：数据库竭尽全力确保，在任何一个给定的时间点，所有观察者对同一数据的看法都是完全一致的。例如，如果某行某列的值是42，那么所有同时读取它的人看到的都是42。当它被改为20时，这个变化对所有观察者是**瞬间且一致**生效的，不会出现有人看到20，有人却还看到42的混乱情况。整个过程是隔离且持久的。

**公式/代码描述**：
可以想象一个事务操作：
```sql
BEGIN TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE user_id = 1; -- 原子操作的一部分
UPDATE accounts SET balance = balance + 100 WHERE user_id = 2; -- 原子操作的另一部分
COMMIT; -- 确保一致性、隔离性和持久性
```
如果`COMMIT`成功，两个更新要么全部生效（原子性），要么全部不生效，并且所有用户之后查询都会看到更新后的、一致的结果。

### BASE：最终一致性模型

BASE是一个相对较新、更“取巧”的缩写，其核心操作概念是**最终一致性（Eventual Consistency）**。

BASE意味着系统**基本可用（Basically Available）**，状态**软（Soft state）**，并保持**最终一致（Eventually consistent）**。最终一致性是指：当某个数据值被更新后，系统不保证所有观察者能立即看到新值。不同的观察者可能在一段时间内看到旧值或新值，甚至同一观察者连续查询可能得到不同结果。但系统保证，只要经过一段足够长的时间（例如几秒或几分钟），且没有新的更新，所有观察者最终都会看到相同的、最新的值。

以下是两种模型在并发读写下的对比示意图：

![](img/9139342e6dd2a9d42ddcc660ed94c289_1.png)

**ACID工作流程**：
1.  初始状态：数据X=42。
2.  两个写入请求（设为10和20）同时到达。
3.  数据库选择一个请求（例如设为20）进入“内部圈”，并设置屏障。
4.  在更改期间，**阻塞**其他写入，并确保所有读取者看到的值从42**一致地**变为20，不会来回跳动。
5.  完成此事务后，再处理下一个写入请求（设为10），并再次确保所有读取者看到的值从20**一致地**变为10。

**BASE工作流程**：
1.  初始状态：数据X=42，在多个服务器上有副本（例如3个）。
2.  写入请求`X=10`到达服务器A，将其副本更新为10（时间戳t1）。写入请求`X=20`到达服务器C，将其副本更新为20（时间戳t2）。
3.  **不一致窗口期**：此时，不同读取者查询不同服务器，可能看到42、10或20。同一读取者多次查询甚至可能看到值在来回变化。
4.  **协调与传播**：服务器之间开始通信，同步数据。根据时间戳（t2 > t1），值20被认为是更新的。
5.  **最终一致**：经过一段时间传播，所有服务器副本都更新为X=20。此后，所有读取者都将看到20。

## 两种模型的权衡与应用

上一节我们通过示意图理解了两种模型的工作流程。本节中我们来看看它们的优缺点和典型应用场景。

最终一致性（BASE）系统的优势在于**极高的可扩展性**。它们通过将数据复制到大量服务器上来实现这一点。这与ACID的“不复制”原则背道而驰，但使得读写操作可以分散到无数节点上，轻松应对海量流量。

那么，最终一致性是“坏”的吗？这取决于应用场景：
*   **不适合的场景**：银行系统、交易、成绩记录等。这些场景要求数据的强一致性，否则会导致严重错误。
*   **适合的场景**：社交网络动态、新闻提要、评论系统等。在这些场景中，短暂的数据不一致（如几秒内看到旧状态）通常是可以接受的。

实际上，许多大型应用会采用混合架构。例如，即使用户系统主体是NoSQL，**新账户创建**这个需要强唯一性校验的环节，也常常用一个小的关系型数据库（ACID）来处理。

## 技术实现细节对比

了解了宏观权衡后，我们进一步看看在具体技术实现上，ACID和BASE数据库有何不同。

以下是ACID与BASE风格数据库在一些关键特性上的对比：

![](img/9139342e6dd2a9d42ddcc660ed94c289_3.png)

*   **主键生成**：
    *   ACID：通常使用**序列整数（Serial Integer）**作为主键，简单高效。
    *   BASE：使用**全局唯一标识符（UUID/GUID）**。这是一个结合了随机数和时间戳的长字符串，确保在分布式环境中全局唯一。
*   **事务与数据新鲜度**：
    *   ACID：**事务**确保你读取的数据不是过时的。
    *   BASE：应用程序可能需要包含**重试循环**，以处理可能读取到陈旧数据的情况。
*   **唯一性约束**：
    *   ACID：可以轻松定义唯一约束。例如在PostgreSQL中可以使用`INSERT ... ON CONFLICT DO NOTHING`。
    *   BASE：难以实现即时全局唯一约束。你只能先插入，然后再通过应用逻辑检测和修复重复数据。
*   **查询模式**：
    *   ACID：通过精心设计的**连接（JOIN）** 和索引，编写高效查询，精确获取所需数据。
    *   BASE：查询模式可能更倾向于“检索并丢弃”。由于数据分布在成千上万个节点上，查询可能先拉取大量数据到应用层，再进行过滤和处理。

## 总结

本节课中我们一起学习了数据库架构中SQL与NoSQL背后的核心思想。

我们从“避免数据冗余”这一基本原则出发，探讨了在超大规模应用下进行数据复制的必要性。我们了解到，SQL与NoSQL之争的本质在于数据模型（关系型 vs. 非关系型）和一致性保证（ACID vs. BASE）的不同。

*   **ACID模型**（如PostgreSQL, MySQL）提供**强一致性**，保证数据的即时准确，适用于金融、交易等关键系统。
*   **BASE模型**（如MongoDB, Cassandra）提供**最终一致性**，通过数据复制实现极高的可扩展性，适用于可容忍短暂不一致的社交、内容类应用。

![](img/9139342e6dd2a9d42ddcc660ed94c289_3.png)

两者并非简单的替代关系，而是针对不同场景的权衡选择。现代复杂系统通常会根据不同模块的需求，采用混合的数据库架构。理解这些根本区别，将帮助你在设计系统时做出更合适的技术选型。
