在本节课中，我们将要学习数据库设计中两个至关重要的概念：键（Keys）和索引（Indexes）。我们将了解它们如何帮助数据库高效地连接表、快速查找数据，以及它们背后的基本工作原理。理解这些概念是优化数据库性能的关键。

## 键：数据的唯一标识符 🗝️

上一节我们介绍了数据库的基本操作，本节中我们来看看如何为数据行建立唯一的“把手”，以便高效地引用它们。这就是键的作用。

键用于在表之间建立连接，这正是“关系型数据库”中“关系”一词的体现。我们需要在表中设置一些列，作为引用数据行的“句柄”。

### 主键：自动递增的序列

最常见的方法是给每一行分配一个自动递增的数字。数据库会确保每个新插入的行获得一个唯一的、顺序递增的ID，无论数据插入的速度有多快。这个过程由数据库内部管理，通过类似“加锁、加1、插入”的机制来保证唯一性。

在PostgreSQL中，实现这一点非常简单。我们使用 `SERIAL` 关键字来定义一个自动递增的整数列，并将其指定为主键。

```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100)
);
```

在上面的代码中：
*   `id SERIAL` 创建了一个自动递增的整数列。
*   `PRIMARY KEY` 声明该列为主键。这告诉数据库为此列建立一个索引，并且这是我们最常使用的索引。整数索引经过大量优化，速度极快。

当我们执行插入操作时，无需指定 `id` 的值，数据库会自动生成。

```sql
INSERT INTO users (name, email) VALUES ('张三', 'zhangsan@example.com');
```

### 逻辑键与唯一约束

除了主键，我们还可以定义逻辑键。逻辑键通常是业务上具有唯一性的字段，例如用户的电子邮件地址。我们通过 `UNIQUE` 约束来实现。

```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100) UNIQUE
);
```

`UNIQUE` 约束告诉数据库：不允许在该列插入重复的值。数据库会为这个列创建一个索引来高效地检查唯一性。当你尝试插入重复的电子邮件时，操作会失败。

一个额外的好处是，数据库知道这个唯一索引也可以用来加速基于该列的查询。例如，当你在一百万条记录中根据电子邮件查找用户时，这个索引能让查询速度提升数十倍。

## 索引：数据的快捷方式 🚀

前面我们一直在谈论如何让数据库工作得更快、更高效，并提到了索引的概念。现在，让我们深入探讨索引。

索引的本质是数据的“快捷方式”。想象一下，Twitter需要在5亿用户中快速找到你的登录信息。如果必须顺序扫描数TB的数据，即使使用高速磁盘，也需要数分钟。索引的目的就是避免全表扫描，让你能“精确地”知道该去哪里找数据。

以下是两种最常见的索引类型：

### B-树索引

B-树（Balanced Tree，平衡树）是最常用的索引结构。数据最终存储在磁盘上，而索引本身就是额外的磁盘数据，它存储了关于“数据位置”的信息。

B-树的工作原理是分层和排序。它像一棵树，从根节点开始，每一层都根据键值将数据划分为不同的范围。通过逐层比较，数据库可以快速定位到包含目标数据的磁盘块。

**工作原理简化描述：**
1.  读取一个小的索引块（根节点）。
2.  根据目标值，确定下一个需要读取的索引块（分支节点）。
3.  重复此过程，直到到达指向实际数据行的叶子节点。
4.  读取最终的数据块。

这种结构的优势在于，查找所需访问的磁盘块数量是数据总量的对数级。例如，在一百万条记录中查找，可能只需要读取6个块（根节点、几个中间节点、叶子节点、数据块），而不是一百万次磁盘读取。

B-树索引适用于多种查询：
*   **精确匹配查找**：例如，通过电子邮件登录。
*   **范围查找**：例如，查找某个日期范围内的订单。
*   **排序操作**：索引本身是有序的，可以避免额外的排序开销。
*   **前缀查找**：例如，查找所有以“A”开头的姓名，这可以转化为一个小的范围扫描。

### 哈希索引

哈希索引采用不同的机制。它使用一个哈希函数（Hash Function）将键值（如一个字符串或数字）计算成一个固定长度的数字（哈希值）。这个哈希值直接对应一个存储位置的“偏移量”。

**哈希计算示例（概念性）：**
可以将字符串的每个字母转换成数字（A=1, B=2...），求和，然后除以一个大的数（如100万），取余数作为哈希值。

哈希索引的查找速度通常比B-树更快，几乎是常数时间复杂度。然而，它有很大的局限性：**哈希索引仅适用于精确匹配查找**。

它不适用于：
*   **范围查找**
*   **前缀查找**
*   **排序操作**

因为哈希函数打乱了数据的原始顺序。因此，哈希索引非常适合主键查找或类似GUID（全局唯一标识符）这种只需要精确匹配的场景。

在实际使用中，我们通常不需要手动选择索引类型。数据库优化器会根据列的数据类型和用途（如声明为 `PRIMARY KEY` 或 `UNIQUE`）自动选择最合适的索引类型（例如，整数主键可能使用哈希，字符串唯一约束可能使用B-树）。我们将这些复杂的决策交给数据库背后的专家和算法。

## 总结 📚

本节课中我们一起学习了数据库键与索引的核心知识。

我们首先了解了**键**，包括作为行唯一标识符的**主键**（通常使用 `SERIAL PRIMARY KEY` 实现），以及保证业务数据唯一性的**逻辑键**（使用 `UNIQUE` 约束）。它们是在表间建立高效关系的基础。

接着，我们深入探讨了**索引**作为数据“快捷方式”的原理。我们介绍了两种主要类型：
1.  **B-树索引**：适用于精确匹配、范围查询、排序和前缀查找，是通用性最强的索引。
2.  **哈希索引**：仅适用于精确匹配查找，速度极快但应用场景有限。

SQL的强大之处在于，我们可以通过简单的声明（如 `PRIMARY KEY` 和 `UNIQUE`）来描述数据的结构和约束，而将实现这些功能的复杂细节（如选择何种索引算法）交给数据库系统本身。我们描述了数据的模式（Schema），然后使用创建、读取、更新、删除（CRUD）这些简单的原语来操作数据，数据库则会确保一切高效、正确地运行。
