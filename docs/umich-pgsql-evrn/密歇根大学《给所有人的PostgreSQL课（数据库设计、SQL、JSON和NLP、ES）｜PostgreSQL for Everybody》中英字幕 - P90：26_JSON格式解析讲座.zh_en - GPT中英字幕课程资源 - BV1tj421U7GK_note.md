## 数据序列化简介

上一节我们介绍了课程主题，本节中我们来看看数据序列化的核心概念。数据序列化是指在不同程序、语言或网络节点之间传输数据结构时，将数据转换为通用格式的过程。最常见的两种数据结构是线性结构和键值对结构。

在Python中，线性结构对应**列表**，键值对结构对应**字典**。其他语言如JavaScript、PHP或Java也拥有类似的结构。本质上，大多数复杂数据结构都可以建模为键值对或线性列表。另一种重要的数据结构是**信息树**，我们稍后会讨论。

序列化之所以必要，是因为当Python中的字典需要发送给JavaScript使用时，双方必须约定一种交换格式。我们称之为“序列化”，因为在早期网络时代，数据是以连续的字节流形式通过线路传输的。这个过程也称为**编组**与**解组**。

## XML：早期的序列化格式

在JSON流行之前，XML是主流的序列化格式。XML即可扩展标记语言，它使用类似HTML的标签树结构来表示数据。

```xml
<array>
  <entry>
    <key>value</key>
  </entry>
</array>
```

XML格式是自描述的，因为你可以自定义标签名称。它被广泛用于系统间的数据编组与解组，或用于将数据存入数据库和文件。许多库、框架和策略都围绕XML构建。

然而，随着Web应用交互性增强，出现了AJAX模式。服务器需要与浏览器中的JavaScript通信，动态更新页面局部内容而无需刷新整个页面。最初，人们使用XML作为这种数据交换的格式，服务器将数据结构转换为XML，浏览器再将其解析回JavaScript对象。

但XML存在一个问题：它是一种层次化的树结构，虽然擅长表示树形数据，但将其转换回简单的数组或字典对象过程复杂且繁琐。

## JSON的诞生与优势

鉴于XML的复杂性，道格拉斯·克罗克福德提出了一个更简单的方案。他注意到JavaScript中定义对象和数组常量的语法本身就可以作为一种优秀的序列化格式。这就是**JavaScript对象表示法**的由来。

JSON本质上是JavaScript对象和数组常量语法的子集。它比完整的JavaScript语法限制更多，以确保安全性和易于解析。例如，在Python中，JSON看起来与字典和列表非常相似。

```python
# Python字典，与JSON格式高度相似
data = {
    "name": "Alice",
    "age": 30,
    "courses": ["Math", "Science"]
}
```

道格拉斯·克罗克福德创建了[json.org](https://www.json.org)网站，精确地定义了JSON格式。这为各种语言（如PHP、Java、Python）构建统一、可互操作的解析库提供了标准，极大地促进了JSON的普及。

JSON的优势在于其简洁性和与JavaScript的天生亲和力。随着Node.js的出现，JavaScript可用于服务器端编程，而像MongoDB这样的数据库直接使用JSON格式存储数据。这使得开发者能够在前端、后端和数据库层使用同一种语言和数据格式，形成统一的技术栈。

## NoSQL运动与PostgreSQL的响应

JSON的流行催生了所谓的“NoSQL”数据库运动，其中许多数据库（如MongoDB）本质上是JSON文档数据库。它们提供了灵活的模式，适合处理半结构化数据。

面对NoSQL的竞争，成熟的关系型数据库如PostgreSQL、MySQL等做出了回应。它们通过添加原生的**JSON**或**JSONB**数据类型来支持JSON文档的存储和查询。这样，用户可以在一个成熟、稳定、功能强大的关系型数据库中，同时享受到关系模型的严谨性和JSON格式的灵活性，从而在某种程度上获得了两全其美的效果。我们将在后续课程中详细探讨PostgreSQL中的JSON功能。
