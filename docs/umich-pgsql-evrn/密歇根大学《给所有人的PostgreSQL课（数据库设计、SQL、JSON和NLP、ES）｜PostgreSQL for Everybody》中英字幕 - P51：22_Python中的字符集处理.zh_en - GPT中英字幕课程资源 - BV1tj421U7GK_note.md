## 背景：Python 2到Python 3的演变

上一节我们介绍了字符集的基本概念，本节中我们来看看Python语言是如何处理字符集的。从Python 2到Python 3的一个重要转变是内部字符串格式采用了Unicode。

在Python 2中，字符串的内部格式是ASCII或8位字节。这主要是因为Python 2诞生于约20年前，其设计深受北欧和美洲地区的影响，当时ASCII字符集是主流。如今，Python 3已被广泛接受，但当初关于是否真的需要从Python 2升级到Python 3的争论持续了十多年。事实证明，Python 3的升级是必要的，而采用Unicode是正确的选择。

基于这个背景，Python 3做出了一个明智的决定：内存中的所有字符串都使用简单的Unicode格式。

## Python 3的字符串与字节类型

在Python 3中，处理字符串变得相对简单。Unicode字符虽然每个占用32位，但在内存中处理速度非常快。例如，遍历一千个字符时，由于每个字符固定为4字节，程序可以快速定位到第40个字符（只需跳转到第160个字节处），无需逐个检查字符边界。

然而，当需要将数据存储到磁盘、通过网络发送或保存到文件时，我们通常需要将其转换为UTF-8格式。这样做既是为了与其他编程语言（如PHP）的互操作性，也是因为外部系统通常期望接收UTF-8格式的数据，而非原始的Unicode格式。

因此，虽然Python内部使用Unicode，但它必须能够读写UTF-8格式的数据。数据库表、网络连接和文件通常都使用UTF-8编码。

Python 3还有另一种数据类型叫做`bytes`类型，用于处理8位字符。在处理压缩、哈希等特定操作时，`bytes`类型有其用武之地。

## 编码与解码过程

由于Python内部字符串是Unicode，每当Python与外部已知系统通信时，都需要经过一个编解码过程。

你可以将UTF-8想象成一种压缩格式。从文件、UTF-8网络流或数据库读取数据时，Python需要对其进行**解码**（类似于解压缩），将其转换为内部的Unicode格式进行处理。当需要将Python内部的字符串数据输出时，则需要对其进行**编码**（类似于压缩），转换为UTF-8格式。

以下是编解码的核心概念：
*   **解码 (Decode)**： `bytes` -> `str` (Unicode)。将外部字节数据转换为Python内部字符串。
*   **编码 (Encode)**： `str` (Unicode) -> `bytes`。将Python内部字符串转换为外部字节数据。

## 文件、网络与数据库中的处理

在实际操作中，编解码过程通常是自动或半自动处理的。

![](img/3c420271aa013d2ac086aa30ad1b044e_3.png)

**1. 文件处理**
打开文件时，可以指定`encoding`参数。默认值通常是`None`，但在绝大多数情况下（可能超过90%），系统会默认使用UTF-8编码。UTF-8的优势在于它能完美兼容旧的ASCII文件。只有当处理像Windows-1252或Latin-1这类包含非UTF-8高位字符（值大于127）的旧编码文件时，才需要显式指定编码。

当你以文本模式打开文件时，读取操作会自动完成解码。如果以二进制模式打开，则会直接得到`bytes`对象。

**2. 网络数据处理**
对于网络数据，部分高级库（如`urllib`）会自动完成解码。但如果直接使用套接字进行底层网络通信，你可能需要手动调用`decode()`方法，将接收到的原始字节数据转换为Python字符串。

**3. 数据库处理**
通过数据库连接器与数据库交互时，处理最为自动化。连接器能够识别数据库使用的字符集（理想情况下应为UTF-8）。即使是一个使用旧字符集（如某种遗留的日文字符集）的数据库，连接器也会在将数据传递给Python时，自动将其转换为Unicode；并在将Python数据写回数据库时，自动转换回数据库所需的字符集。这一切都得益于连接器知晓数据库的创建格式。

## 潜在问题与排查

虽然大部分编解码过程是自动的，但偶尔也会出现问题。如果你发现某些字符显示异常或数据看起来很奇怪，编解码过程可能是问题的根源。

以下是排查步骤：
1.  确认数据源的原始编码。
2.  检查Python中读取或接收数据时是否使用了正确的解码方式（或是否已自动解码）。
3.  检查输出数据时是否使用了正确的编码方式。
