# PostgreSQL for Everybody：2：关系型数据库扩展

![](img/c8e72cdfa5902198c96c1a2c28bcaab8_1.png)

在本节课中，我们将探讨为什么不能在所有场景下都使用ACID（原子性、一致性、隔离性、持久性）数据库。虽然ACID特性非常完美且一致，但在扩展性方面存在挑战。我们将介绍几种扩展数据库的方法，包括垂直扩展、只读副本、多主架构、多存储类型以及多租户架构，并分析它们各自的优缺点。

## 垂直扩展：投入更多硬件

上一节我们介绍了ACID数据库的优势，本节中我们来看看当数据量增长时，首先会遇到的扩展方案：垂直扩展。

垂直扩展的核心思想是为数据库服务器增加更多硬件资源，以提升其处理能力。这通常包括以下几个方向：

*   **增加磁盘与I/O能力**：添加更多硬盘、使用RAID阵列（如RAID 0或RAID 1）来提升并行读写带宽。数据库经常需要扫描大量数据以返回一个小的结果集，更高的I/O吞吐量至关重要。
*   **增加CPU与内存**：添加更多处理器核心，并扩充内存容量。ACID数据库能高效利用内存作为缓存。随着数据增长，可以从8GB内存升级到16GB、32GB，甚至更高，直到遇到新的性能瓶颈。
*   **使用固态硬盘**：将机械硬盘（HDD）替换为固态硬盘（SSD）曾是一项重大改进。SSD的随机访问速度固定，没有机械硬盘的旋转延迟。更先进的技术允许SSD同时读取多个数据块，进一步提升了性能。

然而，垂直扩展存在极限。硬件成本高昂，升级到一定程度后，性价比会急剧下降。特别是在2010-2011年虚拟化技术兴起时，虚拟化大型服务器（如配备8个CPU和32GB内存）非常困难且昂贵，通常需要长期租赁或购买专用硬件。

## 只读副本：分担读取压力

当垂直扩展达到极限时，我们需要寻找其他调优方法。一种经典的关系型数据库调优技术是添加只读副本。

其工作原理是：设置一个主数据库（通常经过垂直扩展优化），负责处理所有写入操作（如`INSERT`、`UPDATE`、`BEGIN TRANSACTION`）。主数据库会将所有数据变更记录到事务日志中。然后，可以配置多个从服务器，它们持续监听主数据库的事务日志，并将这些变更应用到自己的数据库副本上，从而保持数据同步。

以下是关键点：
*   **最终一致性**：副本的数据更新会有微小延迟（可能不到四分之一秒），这实际上引入了一种类似BASE（基本可用、软状态、最终一致性）模型的最终一致性。
*   **分担读取负载**：所有不修改数据的查询操作（如`SELECT`、`JOIN`、`COUNT`）都可以被发送到这些只读副本上执行。这样可以为主数据库分担大量读取压力。
*   **架构混合**：这种模式展示了现代混合架构的雏形，它融合了ACID（主库）和BASE（副本）的特性。副本在极短时间内可能存在不一致性。

需要注意的是，副本数量不能无限增加，因为所有副本都需要从同一个事务日志流中读取数据，这本身可能成为瓶颈。

## 多主架构与多存储类型

除了只读副本，还有其他扩展思路。

**多主架构**是指设置两个或多个主数据库，它们都能处理写入。但这需要主数据库之间进行大量协调通信，例如在事务进行时锁定相关资源并同步状态。这种协调开销很大，通常不如直接扩展一个更强大的主数据库来得高效。

**多存储类型**是另一种常见策略。并非所有数据都适合存储在关系数据库中。例如，博客系统中的个人资料图片或上传的大文件（如PDF、视频）。虽然数据库可以存储二进制大对象（BLOB），但这会给数据库备份和恢复带来负担。

更常见的做法是采用混合存储：
1.  在数据库表中，只存储文件的元数据（如文件名、哈希值、存储路径）。
2.  将实际的文件内容存储在专用的文件系统或对象存储服务中。
3.  应用程序逻辑负责根据数据库中的记录，从文件系统中读取并发送文件内容。

这种方式的优势在于备份策略可以分离：文件系统的备份可能很简单（尤其是当文件以哈希值命名时），而关系数据库的备份则可以独立进行。

## 多租户与“伪云”应用

最后，我们来看一种在云计算早期常见的架构：多租户“伪云”应用。

在这种架构中，服务提供商为每个客户（租户）单独部署一个独立的、小型的数据库实例。所有客户共享同一套应用程序代码。从整体看，系统可能服务于成千上万个客户，拥有成千上万个数据库，但每个数据库本身的数据量并不大（例如，每个客户数据库只处理数万条记录）。

这就像为许多独立的教育机构分别部署邮件系统，每个系统服务几十万用户。关系数据库非常擅长处理这种规模（例如25万条记录以下）的数据。因此，在2002年左右，数据库技术就是围绕处理大量此类“小型”独立数据库实例而演进的。

![](img/c8e72cdfa5902198c96c1a2c28bcaab8_3.png)

这种架构虽然看起来像是云应用（服务众多客户），但实际上并未面临单一数据库需要处理海量数据（如数千万或上亿条记录）的真正云规模挑战，因此被称为“伪云”。

## 总结

本节课我们一起学习了关系型数据库的几种扩展策略：
1.  **垂直扩展**：通过升级硬件提升单机性能，但有物理和成本上限。
2.  **只读副本**：通过异步复制数据到多个从节点来分担读取压力，引入了轻微的最终一致性。
3.  **多主架构**：允许多个节点写入，但协调开销巨大，实践中较少采用。
4.  **多存储类型**：根据数据特性（如大文件）混合使用数据库和专用存储系统，优化备份与管理。
5.  **多租户“伪云”**：通过为每个客户部署独立数据库实例来服务大量客户，但每个实例本身并未面临超大规模数据挑战。

这些方法在云计算兴起之前，是应对数据增长和性能压力的主要手段。它们为理解后续真正云原生数据库（如NoSQL）所解决的超大规模扩展问题奠定了基础。下一节，我们将探讨颠覆这一格局的力量：第一代真正的云应用。
