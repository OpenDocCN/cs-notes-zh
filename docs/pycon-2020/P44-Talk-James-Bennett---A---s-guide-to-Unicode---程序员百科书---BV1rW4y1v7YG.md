这与像 Python 这样的代码单元的工作方式类似，Python 在窄构建上使用其 2 字节 16 位编码，或者甚至可能是更复杂的方式，比如仅暴露 UTF-8 字节序列。这将为某些表情符号提供更大的答案，但这是一个快速了解语言所做的事情及其暴露的抽象的好方法。

这表明它有 Unicode 支持，因为这是一个现在需要知道的重要事情。至于 Python，我们现在有的字符串是代码点序列，这意味着我们可以找到关于它们的信息。如果我们从字符串中获取某些内容，那么长度为 1 的字符串就是一个代码点。如果我们迭代一个字符串，我们就是在迭代代码点。

当我们切片一个字符串或对字符串进行索引时，我们获取的是代码点，我们实际上可以找出代码点的数值，并可以将其转换为十六进制，作为表示 Unicode 代码点的传统。标准库中还有一个非常有用的模块叫 Unicode 数据，这为我们提供了大量的访问权限。

Unicode 数据库以及 Unicode 提供的关于其代码点和字符的信息，因此我们可以提出问题，比如这个代码点的名称是什么，或它被分配到哪个类别，双向或组合渲染行为是什么，所有这些都是找出有用信息的方式。我们还可以从那个模块访问 Unicode 标准化。

我们可以使用任何 Unicode 标准化形式，因此在这里，例如，我们可以将带有变音符号的预组合字符分解为两个代码点序列，或者我们可以将预组合的 1/2 分解为使用兼容等价的序列。此外，Python 中还可以进行大小写折叠。

这很有用，给我们提供了不区分大小写比较的访问权限，任何时候你需要在 Python 中进行不区分大小写的字符串比较，这就是你应该去使用的。我们中的许多人可能养成了从早期不使用 Unicode 时的习惯，使用大写或小写，而我们中的许多人在处理数据库时可能仍然会这样做。

因为我们在 SQL 库或数据库中可能没有大小写折叠的抽象，但在 Python 中我们有这个功能，这就是我们进行不区分大小写比较的方式。

![](img/2a3245dcd9af276985281a539c729f78_23.png)

这是我会绝对坚持的规则，我称之为任何编程语言（不仅仅是 Python）处理中与文本有关的黄金法则，即要意识到你程序的边界，并在那里进行编码和解码，仅在那儿；我所说的边界是指，如果你的程序读取和写入文件，那么这就是一个边界。

当它打开并读取文件的内容或将内容写回文件系统时，如果你的程序通过网络通信，那么这是一个边界；当它通过该连接发送信息或接收信息时，这些都是你应该进行编码和解码的点，这些是你应该处理字节的地方。

对象，但一旦你将它们编码或解码，你应该完全使用字符串，内部不应传递字节对象或处理编码的字节序列，因为大多数较旧的方法以及许多较旧的 Python 代码在从 2 跃升到 3 时遇到麻烦正是因为这个原因。

字节字符串和 Unicode 字符串的混合，或者根本不使用 Unicode 字符串的情况，可能会出现，并且没有考虑编码和解码以及它们需要发生的地方。因此，这是你的黄金法则：如果你只记住这一点，请寻找你程序的边界，识别它们，在那里进行编码和解码，仅在那儿进行。

否则就应处理字符串，真正使用 Unicode。现在，仍然存在一些困难，尤其是在处理文件时，尤其是在某些类型的 Unix 操作系统上，这些问题可分为几个不同的类别。有些系统没有可靠的方法来询问系统使用什么编码。

对于它的文件系统，文件名是字节序列，但你可能没有任何方法来确定如何将其解码为 Unicode 代码点的序列；也有一些文件系统在技术上没有要求能够解码，路径可以是你想要的任意字节序列。

并且在任何已知编码中都无法有效解码，Python 在 Python 3 版本系列中取得了进展，变得更好，有一些技巧和工具。现在，Python 通常会尽力让你将文件系统视为 UTF-8，处理可能无效或完全无法解码的路径时也有一些技巧。

Python 现在所做的类似于 UTF-16 的做法，当它遇到一个无法解码为代码点序列的字节时，它将其原样保留，通过将其转换为代理对范围内的代码点来实现，这样当需要在文件系统上写入内容或进行其他操作时，就可以再次转换回原始字节。

编码有时可以通过告诉 Python 你的文件系统使用的编码来规避，有时你只能希望它能正常工作，因为有些系统的配置是绝望的，但这确实有所改善，现在即使在那些配置不良的系统上也基本可靠，这是一个很大的进步。
