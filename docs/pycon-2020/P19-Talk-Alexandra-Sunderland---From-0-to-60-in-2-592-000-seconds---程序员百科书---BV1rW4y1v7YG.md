大家好，谢谢大家参加我今天关于你们团队中入职工程师的演讲，遗憾的是我们不能见面，但我真的很高兴能够与更多观众分享，特别是因为世界已经发生了许多变化，从我提交这份报告到现在。我认为无法忽视的现实是，有些人将远程开始新的工作，这给已经相当困难的事情增添了一层复杂性，因此我想借此机会谈谈不同的远程入驻方法，解决一些细微差别，帮助每个人提供最佳体验。在过去的十年里，我一直在做工程师，并在过去一年半中担任研究员，目前在加拿大渥太华的工程团队工作。我花了很多时间整理入职流程，帮助工程师建立新团队。我在费罗创建了入职流程，此前我在 SurveyMonkey 工作，在那里我教授一些新聘请的工程师所经历的课程，帮助团队为他们的学生做准备，确保许多人能顺利加入团队，这是我非常有经验的领域。

多年来，我一直在收集反馈，认为这个话题非常重要。我认为作为工程师，尤其是作为经理，最重要的技能之一是同情心，无论是对使用你产品的人，与你的团队互动的人，还是需要经历你在团队中设置的流程的人。这种同理心的一部分是对那些可能紧张入职的人保持关注，他们可能不知道你用的是什么工具，你的流程是什么，你的技术栈是什么，对你来说可能显而易见，甚至可能不是新员工想问的事情，这使我们更难以跟上预期的速度。可能让人觉得他们做得不好，或者不够聪明。我将告诉你如何避免这种情况，这样你的新队友就能迅速跟上步伐，你可以在团队中创造出很好的文化。我之所以对那些开始新工作的人感兴趣，是因为我在开始我的第一份编程工作时，还是个孩子，在当地的初创公司工作之前，我很幸运地知道如何编程，但基本上只在视觉基础上，我从未听说过网络或与编程相关的事情。

比如 Python、HTML 或 JavaScript，我甚至不知道网络应用可以作为一项业务，因为这在你下载大多数软件时，想要的只是简短介绍。我花了几周时间在谷歌上搜索什么是 API，什么是网络钩子，什么是请求，什么是服务器，以及如何登录服务器，越来越详细，因为我不知道发生了什么，也不知道该问谁。我觉得大家都认为我应该知道所有答案，因为没有入职流程，我感到有点失落。嗯，我想那个夏天我学到的最多，但这种体验是不可扩展的，当你试图让五十名或五十个工程师进入你的公司时。我帮助过很多人，我在这个过程中也搞砸了很多。我第一次入职时，我本该把项目交接给他，我为他做的，我基本上只是让他坐一天，逐行遍历整个代码库，告诉他我所知道的一切，以为这样就能结束，如果没有围绕业务需求的上下文，这类信息毫无用处。

理解产品的原因以及知道使用它的人很重要，因为没有人会记得成千上万行代码的细节，没有人会关心超载的信息，这在接下来的几个月里，我遇到了很多本可以避免的问题，如果我专注于一个策略，让他们能找到答案。在我上一份工作中，工程组织有这个入职过程，新员工要在头两周完成一系列课程，以跟上我们使用的不同技术。我教过金贾的入门课程，它是一个与 Django 一起使用的模板系统。然而，我们公司有很多不同的产品和团队，只有少数新员工会使用生姜，所以很多人会跳过会议或者找不到相关内容。这些课程教授给初级开发人员和高级开发人员，几乎每个人，许多事情都在文档中轻松查找的内容上，大家都感觉在这类会议中浪费时间，我最终感到沮丧，觉得自己没有教授任何有用的东西，于是辞去了教官的职务。

尽管你可能知道我想设置课程或一些大的结构来教每个人他们需要知道的一切，同样重要的是，不要跳过任何对你来说显而易见的事情，假设新人什么都知道，尤其在技术上。也许有人是一个出色的 Python 程序员，但从未尝试过，所以他们需要一个简单的介绍，或者至少一个提醒，告知他们应该调查一下。这也与组织和人际关系有关，保持对人们背景的警觉是很重要的。也许你的团队中有人在一家科技公司工作过，知道什么是混合方法，带着敏捷、瀑布和所有处理过的内容，但新员工可能来自没有这样的结构的机构，他们可能会迷失，如果你不花时间向他们解释团队的工作流程。几个月前，我上了一个人的车，部分基于高级代码的概述，他们阻止了我，因为他们以前甚至没有看过产品的现场演示，不知道公司的整体结构，还误以为我是他们的经理。

我假设人们对正在发生的事情有基本的理解，但我错了。很多时候，人们会感到尴尬，因为他们会问一些看起来很基本的问题，尤其是当事情仍然是新的时，因为你的自信让他们觉得应该已经知道答案。他们可能不敢提起这些问题，入职过程非常重要，假设这个人具备表演的技能和能力，他们被雇来做的工作，但尚未具备任何背景，加上重复一些他们可能已经知道的事情，这在这个阶段是无伤大雅的，因为重复和强化是关键。当有这么多新知识需要学习时，最终在与更多人一起犯更多错误后，我缩小了我们现在使用的入职框架，可能有一半的公司，无论你想出什么方法，最重要的步骤之一是征求反馈。这样你就能了解哪些部分需要改进，哪些部分没有帮助，哪些是有价值的，值得保留。我给每位新员工发送相同的反馈调查，在他们工作的第一周结束时，这样我能很好地了解需要改变的地方，并获得许多宝贵的见解。

这个框架让我们的进程变得更好，今天我想和大家分享这个框架，以便你们可以创建自己的入职流程，帮助工程师加入你的团队。在我们开始之前，将你做的每一件事框起来是非常重要的，你需要通过新员工的视角来设置每一个过程。你需要丰富的同理心，回想你刚开始时遇到的所有问题和困难。记住当他们开始时，可能不认识任何人或任何流程或传统，试图跟上速度时会感到格格不入。在这个时候，尤其重要的是要有更深的同理心，考虑他们是否远程工作，如果目前情况不正常，他们可能在家里没有一个合适的办公环境，或者没有快速的互联网。他们可能在照顾孩子，或者总的来说，家中没有理想的工作条件，无论他们的情况如何，耐心和理解都很重要，常常让他们放心你在他们身边，你们都明白这不是一个完全正常的工作环境。

入职最重要的部分是尽早建立正确的关系，通过早期建立这些联系，这个人打开与他人沟通的渠道会更容易，问任何他们需要答案的问题。从某个地方进来可能会很尴尬，不知道谁负责什么或向谁要求某些事情，或者不知道人们是否意识到你的存在。你可以开始建立的一些最重要的互动是你的一对一，确保你尽早与你的新直接报告进行第一次一对一的交流，并保持规律的周期。这样他们就知道他们有时间和空间与你谈论他们的职业目标，他们的想法与期望。这是一个很好的机会，让你了解他们的状态，谈论一些他们可能不想在团队会议上提出的事情。把他们介绍给团队成员很重要，有时会被遗忘。提前让你的团队知道，有人加入，并让他们知道要欢迎新成员。一定要把他们介绍给其他团队成员，他们将与之合作，作为一个工程师，你可能与设计师或产品经理、销售人员合作。

或者你有什么能让这些沟通渠道尽早打开的方法，以便未来的合作更顺畅？在项目开始时，如果能够帮助他们找到一个好朋友，他们就会更容易提问，尤其是当他们知道有一个专门的人可以求助，而这个人在初期阶段可能会被问题压得喘不过气来。当你远程工作时，这一步尤其重要，必须经过深思熟虑。从走廊的谈话开始，在咖啡机旁偶遇某人的机会，如果你能自我介绍，一个好的做法是用一条简讯或电子邮件向新员工介绍自己以及他们的团队或公司。我觉得轻松的氛围比较好，因为这样每个人都可以互动，用一些小表情符号让你感到宾至如归，马上就会很开心，而这种互动是电子邮件无法提供的。为他们安排一个朋友是非常重要的，因为在看不见的情况下，如果有人对你的问题感到恼火，你就不会去问他们。这是一个时间点，在那里问尽可能多的问题是至关重要的，如果他们设置了一个专门的问题解答渠道，他们就不会感到内疚。

而不是仅仅发送电子邮件介绍他们与之合作的人，设置个人视频通话，这样他们就能感受到这个人是如何交流的，获取到语气和语言的细微信息，这些都是重要的。避免很多尴尬的混淆。不要安排他们背靠背，虽然视频通话疲劳是真实存在的，但如果你的新员工是个内向的人，这将是一个已经很有压力的过程中令人筋疲力尽的一步。你的团队掌握了许多知识，而这些往往是外人不经过询问就无法自己弄清楚的。你的团队绝对拥有这种知识，它制造了一个瓶颈。弄清楚是什么，把它写下来。如果团队中有人遇到任何问题，有人可能会说，“哦耶，上周我遇到的事情，我做了任何事情来解决它。”并把它写下来。如果在团队会议上宣布一种新的做事方式，或者安装一个新的衬垫，或者其他什么，写下来。这不一定要看起来像您所期望的文档类型，从第三方库或 API，但它应该涵盖你团队所特有的内容。

不仅仅是帮助你的新员工，最终也会给团队中的人带来回报。作为我自己回忆，发展文件时，很多时候需要记住如何运行某些测试或设置一个环境。积极地鼓励团队中的每个人为这个文档做出贡献。这应该是一个合作的努力，代表每个人的想法。您可以在任何在线协作工具中设置此知识库，我们主办了一个内部网站，建立在阅读文件无花果主题，因为它对我们来说又好又熟悉。Python 开发者更倾向于基于代码的文档，你应该建立一个团队和公司的 FAQ，团队中也有很多非代码的内容是众所周知的，比如如何正确创建一个拉取请求，如何申请休假，应该遵循什么 Jira 过程，以及如何将票交给 QA，甚至什么是开发会议的礼仪，以及是否需要准备。虽然随着时间的推移可以逐渐弄清楚，但提前回答这些问题可以减轻压力，这是团队中每个人都可以做出贡献的事情。

它应该是新员工提出问题的渠道，当他们发现希望了解某些事情时，这个常见问题和代码基础文档对远程工作的人尤其有帮助，因为他们缺乏他人的背景信息，或者周围没有人。他们可以迅速提问，这能帮助人们在遇到情况时更自信，或者指引他们到其他问题。这是加入新团队的一部分，其中不确定性往往是最大的压力源，不知道该完成什么。当看到人们落水时，他们努力在最短的时间内做尽可能多的事情，或在头几个月加班，假装他们不只是为了让别人觉得，他们拥有一切，感觉自己应该和现有团队成员一样好。加入时，他们并没有掌握你所拥有的所有机构知识，可能需要战斗几年。为他们要实现的不同目标设定一个现实的时间表，例如，将开发设置过程分解成小块，获取第三方整合，处理任务，或者任何可能跨越一天的事情，然后为完成一张特定的票设定一个目标，创建一个拉取任务。

或者任何你能在一周内完成的小单位的工作，这些任务应该是容易实现的，这样新员工在检查每一个时都会觉得自己在正轨上。然后为他们设定目标，在接下来的三个月里跟踪他们的进展，他们应该清楚地了解他们应该拥有什么知识。在这一点上，他们应该以什么速度工作，通过准确地定义你要找的内容，他们不会把自己累坏，会知道如何计划工作，以达到目标，这给了他们一些努力的动力。如果他们超出预期，他们会有机会为自己感到满意。得到这一点会促进他们的生产力，作为回报，庆祝他们在团队中实现的目标和里程碑，他们不容易就此满足，开心地向他们的桌旁宣布，他们创造了第一个拉取请求。因此，注意那些时刻，并及时把它们表扬出来，这样团队就可以一起庆祝。即使当我在办公室的时候，我也感到团队非常欢迎我，当我的经理对我第一个请求的评价说我做得很好，恭喜我。这似乎很明显，但有时会被遗忘，确保你有一台电脑和任何需要的硬件。

在他们预期的开始日期之前，作为一名程序员加入一个团队可能有点困难，尤其是当你没有笔记本电脑或键盘时，或者你甚至还没有被分配到一张桌子。人们争先恐后地清理桌子作为你的储藏室，当人们远程工作时，你得更进一步，弄清楚如何把硬件送到那个人的家里。一定要考虑到，在给他们笔记本电脑之前，可能要做的任何安装。不过，如果你在一个足够大的公司，你可能不必担心给他们买笔记本电脑的过程。现在在家工作的人可能没想到会这样，他们的家可能没有合适的办公空间。试着弄清楚你的公司是否可以分配预算来装备他们的家庭办公室，或者有什么设备可以借给他们。一张桌子、椅子和监视器比坐在沙发上看笔记本电脑好几周要好得多，这将使他们更有效率。我在大学里做了几个月时，我不得不做理疗来修复我的背部。确保他们有快速无限的互联网也是很好的，否则，做一些事情，比如视频通话或安装软件，可能是他们担心的一个原因。

征求反馈不是自然而然的事情，尤其是关于你或你所做的事情。你可能很骄傲，但这是整个过程中至关重要的一步，因为如果你不这么做，你可能会错过关于哪些地方不起作用的关键见解。我做了一个反馈调查。在每个新工程师加入团队的第一周后发送给他们，这是我在费罗的反馈功能中发送的。我问他们一些星级问题，以便我们可以随着时间的推移跟踪指标，但也有很多悬而未决的问题，关于哪些工作得很好，哪些工作得不好。我询问了他们的想法，如何让他们的第一周变得更轻松，如果他们能再检查一遍。这在我们塑造这个过程中发挥了重要作用。我们得到的每一条反馈，都让下一个人更容易入队。我们还使人们很容易将他们的反馈直接贡献到过程中，常见问题部分和团队文档可由任何人编辑，他们被鼓励纠正任何不再有效的内容，并提出拉取请求，以确保我们所有的信息都是最新的。

这些小小的贡献帮助他们感觉自己也是团队的一部分，所以有很多技巧和步骤，但很难想象这是一个实际的过程，这意味着你可以管理你的新员工，所以我要向你们展示一下我们是如何把这个组合在一起的。这是一个说明的示例模板，我们用于加入团队的人，这将被创建并附加到每个人在他们的日历中的全天事件中，标志着新雇员的第一天，它被分成不同的部分，这涵盖了我在这张纸条上概述的每一个步骤。你会发现一个有用的资源列表，你队友的名字，还有一大堆可以消化的小目标，因此我们在底部有两个工程部分，这些是分解安装过程和吉拉过程，以小的可检查的格式，这样你就能感觉到你在完成一些事情。管理人员可查阅本说明，这样他们就可以很容易地看到新员工在第一天的进展，或几天后，不用不停地问他们，他们怎么样了，以及他们是否，因为从这里可以很容易地看到，在这张纸条之外。

我们还设置了很多不同的东西，就像一对一、经理、团队午餐、销售、演示标签和不同的会议，因为人际关系确实是入职最重要的部分，以这六个步骤为基准：关系、知识、常见问题解答、目标、空间和反馈。你会很好地在你的路上，为你的团队创建一个可重复的入职过程，为了创造公平的竞争环境，这些都是你所需的最基本的东西，并确保每个加入的人在最初几周内都得到平等的地位，这是一个你永远不想听天由命的过程。虽然其中一些步骤似乎是以后要解决的问题，但当你有下一个雇员的时候，你可以从今天开始做一些事情，为那一天做准备，无论你是经理、团队领导还是工程师，都可以从为代码库创建一组文档开始。您还可以为非代码的事情创建一个团队常见问题，比如哪些会议实际上是会议，哪些地方你不必去。例如，我们的日程表上有一个每周星期五的会议，用作笔记空间，写下我们的内容，在一周内发货，实际上并没有见面。

这个困惑的新员工很长一段时间，因为我们没有写下来，所以当我们在办公室时，大家会了解，因为人们会发现我们并不是突然站起来去开会，但是当我们远程工作时，这种背景不存在。最后，你可以向你的团队寻求反馈。你是否有某种官方程序？你团队中的每个人在某个时候开始，他们可能有一些想法，什么可以让这个过程更容易，问他们或发出调查，并以此为基点来评估你应该如何设置这个，这个过程从零开始需要一些时间。你想从基础开始，然后再做任何太复杂的事情，最终你可以开始尝试一些东西，比如课程、正式会见和问候会议，但请记住，有一种东西是太多的过程，如果你的公司有十个人，你可能不需要一个星期的上课过程。教练和大量的结构，从小做起，需要积累，你仍然希望给人们一些灵活性，他们是如何开始新工作的。我希望这次谈话能引发一些想法，了解如何建立或改进自己团队的入职流程，或者如果你在我今天说的事情上有任何成功。
