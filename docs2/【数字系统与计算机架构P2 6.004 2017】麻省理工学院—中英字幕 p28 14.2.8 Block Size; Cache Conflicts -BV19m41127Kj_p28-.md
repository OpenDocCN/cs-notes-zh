We can tweak the design of the direct map cache a little to take advantage of locality and save some of the overhead of tag fields and valid bits。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_2.png)

We can increase the size of the data field in a cache from one word to two words or four words， etc。

The number of data words in each cache line is called the block size and is always a power of two。

Using a larger block size makes sense。If there is a high probability of accessing nearby words。

 why not fetch a larger block of words on a cache miss？



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_4.png)

Trading the increased cost of the miss against the increased probability of future hits。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_6.png)

Compare the 16 word direct map cache shown here with a block size of4 with a different 16 word direct map cache with a block size of1。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_8.png)

In this cache， for every 128 bits of data， there are 27 bits of tags and valid。

 so 17% of the SRAM bits are overhead。

![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_10.png)

In the sense that they are not being used to store data。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_12.png)

In the cache with block size1， for every 32 Bs of data， there are 27 bits of tag and valid bit。

 so 46% of the SRAM bits are overhead。So a larger block size means we'll be using the SRAM more efficiently。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_14.png)

Since there are 16 bytes of data in each cache line， there are now four offset bits。

 the cache uses the higher order two bits of the offset to select which of the four wordss to return to the CPU on a cash hit。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_16.png)

![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_17.png)

There are four cache lines so we'll need two cash line index bits from the incoming address。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_19.png)

And finally， the remaining 26 address bits are used as the tag field。

Note that there's only a single valid for each cache line。

 so either the entire forward block is present in the cache or it's not。

Would it be worth the extra complication to support caching partial blocks？Probably not。

Looccalality tells us that we'll probably want those other words in the near future。

So having them in the cache will likely improve the hit ratio。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_21.png)

What's the trade off between block size and performance？

 We've argued that increasing the block size from one was a good idea。

Is there a limit to how large blocks should be， Let's look at the costs and benefits of an increased block size。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_23.png)

With a larger block size， we have to fetch more words on a cache miss。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_25.png)

And the miss penaltyelity grows linearly， with increasing block size。

Note that since the access time for the first word from the DRAM is quite high。

 the increased miss penalty isn't as painful as it might be。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_27.png)

Increasing the block size past1 reduces the mis ratio since we're bringing words into the cache that will then be cash hits on subsequent accesses。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_29.png)

Assuming we don't increase the overall cash capacity。

 increasing the block size means we'll make a corresponding reduction in the number of cash lines。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_31.png)

Reducing the number of lines impacts the number of separate address blocks that can be accommodated in the cache。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_33.png)

As we saw in the discussion on the size of the working set of a running program。

 there are a certain number of separate regions we need to accommodate to achieve a high hit ratio。

 program， stack data， etc ceter。

![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_35.png)

So we need to ensure there are a sufficient number of blocks to hold the different addresses in the working set。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_37.png)

The bottom line is that there is an optimum block size that minimizes the mis ratio and increasing the block size past that point will be counterproductive。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_39.png)

![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_40.png)

Combining the information in these two graphs， we can use the formula for average memory access time to choose the block size that gives us the best possible average memory access time。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_42.png)

![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_43.png)

In modern processors， a common block size is 64 bytes or 16 words。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_45.png)

![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_46.png)

Direct map caches do have an Achilles heel。Consider running the three instruction Lo A code with the instructions located starting at Word address 1024。

 and the data starting at Word Ad 37， where the program is making alternating accesses to instruction and data。

 for example， a loop of load instructions。

![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_49.png)

Assuming a 1024 line direct map cache with a block size of one。

 the steady state hit ratio will be 100% Once all six locations have been loaded into the cache since each location is mapped to a different cash line。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_51.png)

Now consider the execution of the same program， but this time the data has been relocated to start at Word address 2048。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_55.png)

![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_56.png)

Now the instructions and data are competing for use of the same cache lines。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_58.png)

For example， the first instruction at Ad 1024 and the first data word at Ad 2048 both map to Ca line 0。

 so only one of them can be in the cache at a time。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_60.png)

![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_61.png)

So fetching the first instruction fills Caline zero with the contents of location 1024。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_63.png)

But then the first data access misses and then refills cache line Ze with the contents of location 2048。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_65.png)

The data address is said to conflict with the instruction address。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_67.png)

The next time through the loop， the first instruction will no longer be in the cache and its fish will cause a cash miss called a conflict Miss。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_69.png)

![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_70.png)

So in the steady state， the cache will never contain the word requested by the CPU。



![](img/90a3ba7a435be1f01b39f4d7b85b3a1e_72.png)

This is very unfortunate。 We were hoping to design a memory system that offered the simple abstraction of a flat uniform address space。
