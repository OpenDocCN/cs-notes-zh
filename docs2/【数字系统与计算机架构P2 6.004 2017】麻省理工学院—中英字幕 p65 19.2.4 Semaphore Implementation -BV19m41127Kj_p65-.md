Here are the OS handlers for the weight and signal supervisor calls。

Since supervisor calls are run in kernel mode， they can't be interrupted。

 so the handle code is naturally executed as a critical section。

Both handlers expect the address of the Semapho location to be passed as an argument in the users R0。

The weight handler checks the sepho's value， and if is non0， the value is decremented。

 and the handler resumes execution of the user's program at the instruction following the weight supervisor call。

 If the Sepho is 0， the code arranges to reex the weight supervisor call when the user program resumes execution。

And then call sleep to mark the processes inactive until the corresponding wake up call is made。

The signal handler is simpler， it increments the semaphore value and calls wakeake up to mark as active any processes that were waiting for this particular semaphore。

Eventually， the round Robin Schr will select a process that was waiting and it will be able to decrement the semaphore and proceed。

Note that the code makes no provision for fairness。 In other words。

 there's no guarantee that a waiting process will eventually succeed in finding the semaphore non0。

The Schr has a specific order in which it runs processes。

 so the next in sequence waiting process will always get the sepho even if there are later in sequence processes that have been waiting longer。

If fairness is desired， Wade could maintain a queue of wedding processes and use the queue to determine which process is next in line。
