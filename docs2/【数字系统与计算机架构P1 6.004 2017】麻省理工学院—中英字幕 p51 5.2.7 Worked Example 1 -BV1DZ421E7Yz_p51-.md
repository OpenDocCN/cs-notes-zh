The goal of this problem is to design a reliable latch。

 A latch can be designed using a multipleer where the G input is used to determine whether or not a new data value should be loaded into the latch。



![](img/0676796b47d8ce8f114a5a2bf1e07838_1.png)

The way the latch works is that when g equals0， the old value of Q is fed back through the multipleer。

 so it retains its old value。

![](img/0676796b47d8ce8f114a5a2bf1e07838_3.png)

When g equals 1， the multipleer selects d as the input， so Q gets updated to be the value of d。



![](img/0676796b47d8ce8f114a5a2bf1e07838_5.png)

The logic function that describes the operation of the latch is Q equals not G and Q or G and D。



![](img/0676796b47d8ce8f114a5a2bf1e07838_7.png)

We want to build the stch using only and or and inverter gates。

We are given three proposed designs for the latch for each proposed design。

 We want to specify whether the design is bad， good or obese。

 bad means that the latch does not work reliably。 Good means that the latch works reliably。

 and obese means that the latch works， but uses more gates than necessary。

Latch Repoal A is shown here。Taking a closer look at this circuit。

 we see that the logic function that it implements is Q equals G and Q or D。

 This is not the correct logic equation for a latch。 So this design is bad。



![](img/0676796b47d8ce8f114a5a2bf1e07838_9.png)

Next， we take a look at proposals B and C。

![](img/0676796b47d8ce8f114a5a2bf1e07838_11.png)

The logic equation for proposal B is Q equals not G and Q or G and D。

 This is exactly the same logic equation that we specified for our latch， however。

 this implementation is not lenient because it does not guarantee that you will not see glitches on your output signal when G changes value。



![](img/0676796b47d8ce8f114a5a2bf1e07838_13.png)

![](img/0676796b47d8ce8f114a5a2bf1e07838_14.png)

Proposal C， however， includes the same logic function， Q equals not GNq or GN D， plus one more term。

 which is the or of Q and D。

![](img/0676796b47d8ce8f114a5a2bf1e07838_16.png)

If you create a car no map for both of these logic functions。

 you see that the QD term is redundant because it doesn't add any additional ones to the carno map。

 This means that logically the two functions are equivalent。

 but the effect of the extra term is that it makes the implementation lenient。

 So proposal C is good and proposal B is bad。
